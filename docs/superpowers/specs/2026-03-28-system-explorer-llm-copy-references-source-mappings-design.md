# System Explorer: LLM Copy, References & Source Mappings

**Date:** 2026-03-28
**Status:** Draft
**Scope:** Three new features for the system-explorer generated static HTML courses

## Summary

Add three capabilities to system-explorer's generated HTML courses:

1. **Copy as Markdown** — A per-page button that copies a specially formatted, LLM-friendly markdown version of the page content
2. **References & Resources** — Inline contextual links within content and a consolidated references section at the bottom of each page
3. **Source Code Mappings** — Embedded code snippets labeled with file paths and linked to GitHub source

All three features follow Approach A: enrich the analysis.md format with new structured fields, and have system-to-course parse and render them into HTML.

## Feature 1: Copy as Markdown Button

### User Experience

A "Copy as Markdown" button appears in the top-right area of each content page (not index.html). Clicking it copies a specially formatted markdown version of the page to the clipboard, optimized for pasting into LLM conversations.

The button shows "Copied!" feedback for 2 seconds, consistent with the existing code block copy button pattern.

### Formatted Output Structure

```markdown
# [System Name]: [Page Title]
> Source: [Page URL or file path] | Level: [Beginner/Intermediate/Advanced]
> This is the [Core Concepts] section of an interactive course about [DuckDB],
> an embedded analytical database for OLAP workloads.

[Page content converted to clean markdown]

---
*Generated by System Explorer*
```

### Content Conversion Rules

- Content is converted from the rendered HTML DOM, reflecting any enrichments added during generation
- Heading hierarchy is preserved (h2 -> ##, h3 -> ###, etc.)
- Code blocks preserve language annotations
- Glossary terms are expanded inline: `vectorized execution (a technique that processes data in batches of ~2048 values)` instead of just the bare term
- Interactive elements (architecture diagrams, decision trees, concept cards) are flattened to descriptive text with their expanded content included
- Callout boxes are converted to blockquotes with type prefix: `> **Insight:** ...`
- Performance gauges are converted to text: `Analytical Query Speed: Excellent (92%)`

### Implementation

**Metadata source:** A `<!-- page-context: ... -->` HTML comment injected into each generated page by system-to-course, populated from the analysis.md Metadata section. Format:

```html
<!-- page-context:
  system: "DuckDB"
  description: "An embedded analytical database for OLAP workloads"
  page: "Core Concepts"
  level: "Beginner"
-->
```

The JavaScript reads these fields to construct the context header in the copied markdown.

**JavaScript:** DOM traversal function that walks the content area and converts elements to markdown. Uses `navigator.clipboard.writeText()` for clipboard access.

**Placement:** Fixed-position button in the page header area, right-aligned, styled consistently with the existing design system. Uses a document icon and "Copy as Markdown" label. On mobile, collapses to icon-only.

## Feature 2: References & Resources with Links

### Analysis.md Format Extension

Each section gains an optional references block at the end, using HTML comment syntax:

```markdown
## Architecture
<!-- level: intermediate -->
<!-- references:
- [Architecture Deep Dive](https://duckdb.org/docs/internals/overview) | official-docs
- [Push-Based Execution Paper](https://arxiv.org/abs/...) | paper
- [Storage Format Blog](https://duckdb.org/2024/...) | blog
-->
```

**Reference types** (used for icon selection and grouping):
- `official-docs` — Official documentation
- `paper` — Academic/research papers
- `blog` — Blog posts and articles
- `github` — GitHub repositories or specific files
- `video` — Video content (talks, tutorials)
- `tutorial` — Step-by-step tutorials
- `community` — Community resources (forums, discussions)

### Inline Contextual Links

Standard markdown links within the analysis.md content body. The system-analyzer is already free to include these — the change is making this **expected and encouraged** in the analysis template guidance.

**Guidance for system-analyzer:**
- Link on first mention of a concept that has an authoritative external source
- Link claims that benefit from a primary source
- Prefer official documentation over third-party summaries

**HTML rendering:** External links (non-relative URLs) receive `target="_blank"`, `rel="noopener noreferrer"`, and a small external-link icon (↗) appended via CSS `::after` pseudo-element.

### Per-Page Reference Footer

Rendered at the bottom of each content page, above the prev/next navigation, if that page's source section has a `<!-- references: ... -->` block.

**Layout:** Each reference entry shows:
- Type icon (mapped from reference type: `official-docs` -> "📄", `paper` -> "📜", `blog` -> "📰", `github` -> "💻", `video` -> "🎥", `tutorial` -> "📖", `community` -> "💬")
- Linked title (opens in new tab)
- Domain hint (extracted from URL, e.g., "duckdb.org")

**Styling:** Consistent with existing callout boxes — subtle warm background, left border accent, contained within content max-width. Section heading: "References & Resources" with a book icon.

## Feature 3: Implementation Details with Codebase Source Mappings

### Analysis.md Format Extension — Metadata

New fields in the Metadata section:

```markdown
## Metadata
- **Name:** DuckDB
- **GitHub:** duckdb/duckdb
- **Tag:** v1.5.1
...existing fields...
```

`GitHub` provides the default `org/repo` for source links. `Tag` provides the default git ref. Both are used as fallbacks when individual code blocks don't specify their own.

### Analysis.md Format Extension — Code Blocks

Code blocks in Implementation Details and How It Works sections gain source annotations as comment lines at the top of the block:

````markdown
```cpp
// source: src/parser/parser.cpp:142-168
// github: duckdb/duckdb
// tag: v1.5.1
unique_ptr<SQLStatement> Parser::ParseStatement() {
    auto token = Peek();
    switch (token.type) {
        case TokenType::SELECT:
            return ParseSelectStatement();
    }
}
```
````

**Annotation fields** (only apply to code blocks that represent actual source code — regular example/tutorial code blocks remain unannotated):
- `// source: <file-path>:<start-line>-<end-line>` — Required for source-annotated blocks. File path relative to repo root, with optional line range. A code block without `// source:` is treated as a regular code block.
- `// github: <org/repo>` — Optional. Overrides the Metadata-level GitHub field
- `// tag: <ref>` — Optional. Overrides the Metadata-level Tag field

### Generated HTML — Source-Annotated Code Block

**File path header bar:** Displayed above the code block, showing the file path prominently. Right-aligned link icon (↗) opens the file on GitHub.

**GitHub URL construction:**
```
https://github.com/{github}/blob/{tag || "main"}/{source_path}#L{start}-L{end}
```

Resolution order for `github` and `tag`: code block annotation > Metadata section > fallback (`main` for tag, omit link if no github).

**Code body:** Same dark theme as existing code blocks. The `// source:`, `// github:`, and `// tag:` metadata lines are stripped from display. Copy button copies clean code only (no metadata lines).

### Source Map Summary

At the top of the Implementation Details page (and How It Works, if it has source-annotated blocks), a collapsible "Source Map" section lists all files referenced on that page.

**Each entry shows:**
- File path (linked to GitHub)
- Line range
- Anchor link to the corresponding code block on the page

**Footer line:** `Repository: {org/repo} @ {tag}` with link to the repository.

**Behavior:** Collapsed by default. Click to expand. Uses the same accordion pattern as existing expandable elements.

## Pipeline Changes

### system-analyzer

**analysis-template.md:**
1. Add `GitHub` and `Tag` fields to the Metadata section template
2. Add per-section `<!-- references: ... -->` block documentation with type list
3. Add code block source annotation documentation (`// source:`, `// github:`, `// tag:`)
4. Add guidance encouraging inline contextual links in content

**SKILL.md:**
- Instruct the analyzer to research and include reference URLs for each section
- Instruct the analyzer to look up actual source file paths and line numbers when analyzing implementation details
- Instruct the analyzer to include inline contextual links on first mention of concepts with authoritative sources

### system-to-course

**page-templates.md:**
1. Add "Copy as Markdown" button to content page header template
2. Add references footer section template (above prev/next nav)
3. Add source-annotated code block template with file path header bar
4. Add source map collapsible section template

**interactive-elements.md:**
1. Copy-as-markdown button: DOM-to-markdown conversion logic, context header injection, clipboard API usage, "Copied!" feedback pattern
2. References footer: `<!-- references: ... -->` parsing, type-to-icon mapping, external link rendering with domain hints
3. Source-annotated code block: metadata line parsing and stripping, GitHub URL construction with fallback logic, file path header bar
4. Source map: aggregation of all source annotations on a page, collapsible accordion rendering

**design-system.md:**
1. Copy-as-markdown button styles (positioning, hover state, mobile icon-only variant)
2. Reference footer styles (icon grid layout, domain hint typography, warm background)
3. Source-annotated code block styles (file path header bar with monospace font, link icon)
4. External link icon CSS (`a[target="_blank"]::after` pseudo-element with ↗ character)

**SKILL.md:**
- Parse `<!-- references: ... -->` blocks from analysis.md sections and render footer sections
- Parse code block source annotations (`// source:`, `// github:`, `// tag:`) and generate GitHub links with fallback to Metadata-level values
- Inject `<!-- page-context: ... -->` metadata into each generated page for the copy button
- Generate the copy-as-markdown JavaScript in each content page
- Strip metadata comment lines from displayed code blocks

### Unchanged

- **system-finder** — Operates before analysis, no changes needed
- **system-explorer orchestrator** — Pipeline flow unchanged, phases remain the same
