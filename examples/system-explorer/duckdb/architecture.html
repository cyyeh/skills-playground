Storage Layout

DuckDB organizes persistent data into row groups of approximately 122,000 rows. Within each row group, data is stored column-by-column with three key optimizations:

  • Compression -- Each column segment is independently compressed using the best algorithm for its data distribution (constant, dictionary, RLE, bitpacking, FSST for strings, ALP for floats).
  • Zone maps -- Each column segment stores its min and max values. When a query filters on WHERE date > '2025-01-01', entire row groups where max(date) < '2025-01-01' are skipped without reading any data.
  • Partial decompression -- DuckDB can operate on compressed data during execution (e.g., dictionary vectors), deferring decompression until necessary.
🔭
Architecture insight: Zone maps are why sorting your data by commonly filtered columns (like date or region) makes such a big performance difference. Sorted data creates tight min/max ranges, allowing DuckDB to skip more row groups entirely.

Parallelism Model

DuckDB automatically parallelizes queries across all available CPU cores using morsel-driven parallelism:

  1. Morsel assignment -- Row groups are divided into morsels assigned to worker threads. Each thread processes its morsel independently through the pipeline.
  2. No user configuration -- Parallelism is automatic. No SET parallel_workers or manual partitioning required.
  3. Parallel-aware operators -- Hash joins, aggregations, and sorts all support parallel execution with shared hash tables and merge steps.
  4. Parallel I/O -- Even CSV and Parquet file reading is parallelized across threads scanning different portions of a file.

Concurrency & Memory Management

DuckDB uses a single-writer, multiple-reader model. Multiple connections can read simultaneously, but only one connection can write at a time. This is a deliberate design choice -- DuckDB is not designed for high-concurrency OLTP workloads.

Memory is managed through:

  • Buffer manager -- DuckDB manages its own buffer pool, paging data in and out of memory as needed.
  • Spill to disk -- When intermediate results exceed available memory, DuckDB spills to temporary files rather than crashing with an out-of-memory error.
  • Streaming execution -- Where possible, operators stream data through the pipeline without materializing entire intermediate results.
  • Configurable limits -- Users can set SET memory_limit = '4GB' to control DuckDB's memory usage.
ℹ️
Note: In v1.5.0, non-blocking checkpointing was added, allowing reads to continue during checkpoint operations. This yielded a 17% throughput improvement on the TPC-H benchmark.
