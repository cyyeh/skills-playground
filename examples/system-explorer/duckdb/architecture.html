Architecture

High-Level Design

DuckDB follows a classic layered database architecture, but with a key twist: everything runs in-process. There is no network layer, no wire protocol, and no separate server. The entire system -- from SQL parsing to disk I/O -- executes within the host application's address space.

Optimizer Passes

The optimizer applies 20+ passes to transform naive query plans into efficient ones. A query joining three tables might have 6 possible join orderings, and picking the right one can mean the difference between milliseconds and minutes.

PassPurpose
Filter PushdownMoves filter predicates closer to data sources to reduce intermediate result sizes early
Join Order (DPccp)Dynamic programming algorithm to find the optimal join ordering
Projection PullupRemoves columns that are never used downstream
CTE InliningInlines Common Table Expressions when materialization is wasteful
TopN OptimizationConverts ORDER BY + LIMIT N into a single operator that avoids full sorting
Statistics PropagationGathers and propagates column statistics to inform downstream decisions
Row Group PruningEliminates entire storage row groups based on min/max metadata
Late MaterializationDefers reading columns until they are actually needed
🔭
Design insight: The optimizer runs passes in a carefully ordered sequence. Filter pushdown runs before join ordering so the join optimizer sees reduced cardinalities. Some passes (like CTE inlining and column lifetime analysis) run twice -- once early and once late -- because earlier passes may create new optimization opportunities.

ℹ️
Example: SELECT region, SUM(sales) FROM orders GROUP BY region becomes: Parser produces a GroupByNode, Binder verifies orders exists, Planner creates LogicalAggregate -> LogicalGet, Optimizer pushes filters and prunes unused columns, and the Physical Planner creates PhysicalHashAggregate -> PhysicalTableScan for parallel morsel-based execution.
