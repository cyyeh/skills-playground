Core Concepts

The Seven Pillars of DuckDB

DuckDB is built on seven core ideas that together make analytical SQL as simple as pip install duckdb. Each concept solves a specific problem in data processing.

🏠

In-Process Architecture

Runs inside your application as a library, not as a separate server. Zero network overhead, zero-copy data transfer.

Think of a built-in oven (DuckDB) vs. a catering service (PostgreSQL). The oven is right there in your kitchen, instantly available, with no delivery lag. A Python script can hand a pandas DataFrame directly to DuckDB without serialization.

📊

Columnar Storage

Data stored column-by-column, organized into row groups. Only reads the columns your query needs.

Imagine a library where all Chapter 1s from every book are on one shelf. When you need Chapter 3 from 10,000 books, you visit a single shelf instead of pulling every book. This is the foundation of OLAP performance.

⚡

Vectorized Execution

Processes data in batches of 2,048 tuples that fit in CPU L1 cache, enabling SIMD auto-vectorization.

Like a conveyor belt: a tray of 2,048 items moves through each station together, keeping every station busy while fitting in the workspace (CPU cache). This eliminates per-row overhead.

🔄

Push-Based Parallelism

Morsel-driven execution distributes work across all CPU cores adaptively, with no user configuration needed.

Like a head chef distributing tickets: each cook gets a batch of orders, works through them, then gets more. If one cook finishes early, they pick up the next batch. No one waits idle.

🔒

MVCC Concurrency

Multi-Version Concurrency Control ensures readers never block writers and vice versa.

Like a shared Google Doc where each person sees a consistent snapshot at the moment they opened it. Edits are tracked separately, and conflicts are detected at commit time. Read-only transactions have zero overhead.

🧩

Extension System

Dynamically load new data types, functions, file formats, and storage backends. 150+ community extensions.

DuckDB's core is a Swiss Army knife with essential blades. Extensions are specialized attachments: a Parquet reader, S3 access, full-text search. You only carry what you need.

📦

Single-File Design

Zero dependencies, single amalgamation file. Databases are a single file with magic bytes DUCK.

Like a self-contained camping stove that needs no external gas line. You bring the single unit, and it works on Linux, macOS, Windows, ARM, x86, or even WebAssembly.

In-Process Architecture

DuckDB runs inside your application's process as a library. There is no network protocol, no daemon to manage, and no client-server communication overhead. This enables zero-copy data transfer between the host process and the database.

A Python script can hand a pandas DataFrame directly to DuckDB's query engine without serialization. This eliminates the largest bottleneck in traditional analytics pipelines: moving data between systems.

🔭
Architecture insight: DuckDB was created at CWI in the Netherlands, the same research institute that produced MonetDB and its vectorized execution research. The in-process design was a deliberate choice to fill the gap left by SQLite (which is optimized for OLTP, not analytics).

Columnar Storage

Data is stored column-by-column rather than row-by-row. All values of a single column are stored contiguously on disk and in memory, organized into horizontal partitions called row groups.

Analytical queries typically touch few columns but many rows (e.g., SELECT AVG(price) FROM orders). Columnar storage means DuckDB reads only the columns needed, skips the rest, and achieves far better compression because similar values are adjacent.

💡
Tip: DuckDB's columnar format is the foundation of its OLAP performance. When you see 10-100x speedups over row-oriented databases on aggregation queries, columnar storage is the primary reason.

Vectorized Execution

Rather than processing one row at a time (Volcano model) or one full column at a time, DuckDB processes data in fixed-size vectors of 2,048 tuples (STANDARD_VECTOR_SIZE). Each operator pulls or pushes a DataChunk through the pipeline.

Vector sizes are chosen to fit in the CPU's L1 cache (32-128 KB on modern processors). This eliminates interpretation overhead, enables SIMD auto-vectorization by modern compilers, and dramatically reduces function call overhead.

How They Fit Together

These concepts form a layered architecture where each layer builds on the previous one:

  1. In-process design eliminates network overhead and enables zero-copy data transfer
  2. Columnar storage organizes data for analytical access patterns
  3. Vectorized execution processes that columnar data in CPU-cache-friendly batches
  4. Push-based parallelism scales those vector operations across all cores
  5. MVCC keeps readers and writers out of each other's way
  6. Extension system adds capabilities without bloating the core
  7. Single-file design makes the entire system trivially deployable
ℹ️
The result: A database that can be embedded in a Python script, process hundreds of millions of rows on a laptop, and require zero operational overhead.
