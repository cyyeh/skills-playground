Core Concepts

The Seven Pillars of DuckDB

DuckDB is built on a set of core concepts that together enable blazing-fast analytical queries in a zero-setup package. Each concept is a design decision that optimizes for the analytical workload pattern: scanning many rows but few columns, with complex aggregations and joins.

📊

Columnar Storage

Data organized by column, not by row. Only reads the columns your query needs.

Think of a library where separate filing cabinets hold "birthdays," "addresses," and "names." To find all birthdays, you open just one cabinet. This is dramatically faster when you only need a few fields from millions of records.

⚡

Vectorized Execution

Processes data in batches of ~2048 values, maximizing CPU cache efficiency.

Like a factory loading a tray of 2048 widgets and running the entire tray through each station. The tray fits perfectly on the workbench (CPU L1 cache), so the worker never leaves their station to fetch parts.

🏠

In-Process

Runs as a library inside your application. No server, no network, no admin.

Like having a professional chef in your house instead of ordering from a restaurant. You speak directly to them -- no waiter, no wait time, no overhead. The tradeoff: only people in your house can eat.

📖

Rich SQL Dialect

PostgreSQL-compatible SQL with modern extensions like PIVOT, COLUMNS(), and FROM-first syntax.

🧩

Extensions

Loadable modules for Parquet, JSON, S3, spatial, and 100+ more capabilities.

🔀

Parallel Execution

Morsel-driven parallelism distributes work across all CPU cores automatically.

🗃

Catalog

Metadata registry tracking schemas, tables, views, functions, types, and extensions.

Columnar Storage

Columnar storage means data is organized by column rather than by row. When a query asks "what is the average salary?", DuckDB reads only the salary column and skips everything else.

Analytical queries typically scan many rows but only a few columns (e.g., SELECT AVG(price) FROM sales). Columnar layout means DuckDB reads orders of magnitude less data from disk and memory, and the contiguous column data compresses extremely well.

🔭
Architecture insight: Row-based storage (like PostgreSQL) stores all columns of a row together. This is great for SELECT * FROM users WHERE id = 42 (fetching one full row) but terrible for SELECT AVG(salary) FROM users (reading one column from millions of rows). DuckDB's columnar layout is the opposite tradeoff -- optimized for the analytical pattern.

Vectorized Execution

Vectorized execution processes data in batches (called "vectors") of up to 2048 values at a time. Each operator in the query plan works on an entire vector before passing results to the next operator.

The vector size of 2048 is deliberately chosen to fit within the CPU's L1 cache (typically 32-128 KB per core). This keeps data in the fastest level of the CPU's memory hierarchy.

💡
Tip: Vectorized processing reduces per-row function call overhead by ~2000x and enables the compiler to automatically generate SIMD instructions that process multiple values in a single CPU cycle.

In-Process Architecture

DuckDB runs as a library embedded directly in your application process. There is no separate database server, no network communication, and no client-server protocol. You link DuckDB into your Python, R, Java, or C++ program and call it like a function.

Eliminating the network layer removes serialization/deserialization overhead and enables zero-copy data sharing with host applications. A Python program can pass a Pandas DataFrame to DuckDB and get results back without any data copying.

Extensions

Extensions are like apps on a smartphone. DuckDB core comes with essential capabilities, but you install extensions to add new abilities -- spatial operations, cloud storage access, or reading from live PostgreSQL databases.

DuckDB ships with ~24 core extensions and has over 100 community extensions. Key extensions include parquet, httpfs, json, spatial, iceberg, and postgres_scanner.

ℹ️
Note: Extensions can be authored in C++, Rust, Python, or even Shell scripts, demonstrating the flexibility of DuckDB's extension API. Install with INSTALL extension_name; LOAD extension_name;.

Parallel Execution

DuckDB uses morsel-driven parallelism to distribute work across all available CPU cores. Data is divided into morsels (chunks), and multiple threads process different morsels through the same pipeline simultaneously.

DuckDB's parallelism model scales nearly linearly with core count for most analytical queries, turning a 10-second single-threaded query into a 1-second query on a 10-core machine.

🔭
Design insight: Traditional databases use exchange operators that partition data between parallel workers at fixed points. This can lead to skewed workloads. DuckDB's morsel-driven approach distributes work dynamically, like a work-stealing scheduler, achieving better load balancing.
