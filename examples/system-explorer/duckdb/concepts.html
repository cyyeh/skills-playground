Core Concepts

The Six Pillars of DuckDB

DuckDB is built around a few powerful ideas that work together to make analytical queries fast on a single machine. Each concept below addresses a specific problem in data analysis. Click "Learn more" on any card to dig deeper.

🏠
In-Process Database

DuckDB runs as a library inside your application -- like a calculator built into your spreadsheet. No separate server, no network overhead.

When you import duckdb in Python, the entire database engine loads into your process. This eliminates the client-server overhead that databases like PostgreSQL require. There is no TCP connection, no serialization/deserialization of results -- data moves at memory speed.

📊
Columnar Storage

Data is stored column-by-column, not row-by-row. To compute the average salary, you read one column instead of opening every record.

Columnar layout means better compression (similar values compress well), faster scans (read only the columns you need), and happier CPU caches (processing contiguous arrays of the same type fits modern CPU architectures perfectly).

⚡
Vectorized Execution

Processes data in batches of 2,048 values at once, using tight loops the CPU can optimize with SIMD instructions.

Think of grading exams: instead of grading one student's entire exam, you grade Question 1 for all students, then Question 2. Your brain stays in one mode, you get faster. DuckDB does the same with data operations, dramatically reducing per-row overhead.

🧩
Extensions

Like apps on a phone -- DuckDB does a lot out of the box, but extensions add file formats (Iceberg, Lance), remote access (S3), and more.

Core extensions auto-install when first used. The extension system adds new file formats (Parquet, Iceberg, Delta Lake), remote storage (S3, HTTP, Azure), domain-specific functionality (spatial/GIS, full-text search), and database connectors (PostgreSQL, MySQL, SQLite).

🔍
Data Virtualization

Query Parquet files, CSVs, pandas DataFrames, and S3 objects directly -- without importing or copying the data first.

DuckDB's "replacement scan" feature intercepts references to external objects (like a DataFrame variable name in SQL) and reads them in-place. It is like reading a library book on the shelf instead of photocopying it to take home.

📁
Single-File Persistence

A DuckDB database is one file (like SQLite or Excel). Email it, scp it, or check it into version control.

DuckDB databases use the .duckdb extension. They support ACID transactions via MVCC, allowing multiple concurrent readers with a single writer. No WAL directories, no tablespaces, no config files -- just one portable file.

Columnar vs. Row Storage

This is the most important concept for understanding why DuckDB is fast for analytics. Here is the difference at a glance:

🔭
Architecture insight: Imagine a filing cabinet. A row-based database stores each person's complete file in one folder. To calculate the average salary, you open every folder. A columnar database has one drawer labeled "Salaries" -- you open it and scan straight through.

Row Storage (e.g., PostgreSQL)

[Alice, 30, NYC, 95000]
[Bob, 28, LA, 82000]
[Carol, 35, SF, 110000]

To get AVG(salary): must read every field of every row

Columnar Storage (DuckDB)

Names: [Alice, Bob, Carol]
Ages: [30, 28, 35]
Cities: [NYC, LA, SF]
Salaries: [95000, 82000, 110000]

To get AVG(salary): read only the salary column

How Vectorized Execution Works

Each step processes a vector of 2,048 values -- tight loops the CPU can optimize with SIMD instructions.

💡
Tip: Vectorized execution is why DuckDB can match or beat distributed systems on single-machine datasets. The tight loops minimize cache misses and let the CPU's out-of-order execution work at full efficiency.

What Makes DuckDB Different?

DuckDB fills a gap that didn't have a good solution before:

  • SQLite and PostgreSQL are great for transactional data, but painfully slow when aggregating millions of rows.
  • Cloud data warehouses (BigQuery, Snowflake) are powerful but require infrastructure, network, and money per query.
  • pandas and Polars are flexible but lose SQL's expressiveness and hit memory limits on larger datasets.

DuckDB gives you a single-file, zero-dependency database that runs inside your process and handles analytical workloads at remarkable speed. You can query a 10 GB Parquet file on your laptop in seconds.

ℹ️
Note: DuckDB's sweet spot is datasets from 1 MB to 200 GB -- large enough that pandas struggles, small enough that you don't need a distributed cluster.
