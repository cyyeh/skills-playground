Core Concepts

The six fundamental ideas that define DuckDB. Click any card to expand the technical details.

🔭
The best mental model: "SQLite for analytics" Just as SQLite gives you a zero-configuration embedded database for transactional workloads, DuckDB gives you the same simplicity but optimized for analytical queries that scan, aggregate, and join large datasets. It runs in-process, stores data in a single file, and requires no server. If you remember one thing from this page, remember this analogy — it captures DuckDB's entire design philosophy.

Six Core Concepts

💻
In-Process Database
Like a calculator built into your spreadsheet app — no separate program, no network connection, just instantly available inside your application.

DuckDB runs as a library loaded into your application's process. When your Python script does import duckdb, the entire database engine becomes part of your script. This eliminates the client-server overhead that databases like PostgreSQL require.

There is no separate daemon to start, no TCP connections to manage, and no serialization of queries across a network boundary. The query engine reads and writes memory directly inside your process.

Columnar Storage
Instead of filing each person's complete record in one folder, imagine one drawer labeled "Salaries" containing just salary values for everyone. Need the average? Open one drawer.

DuckDB stores data column-by-column rather than row-by-row. This means:

  • Better compression — similar values in the same column compress efficiently (e.g., a column of country codes)
  • Faster scans — analytical queries that aggregate one or two columns don't need to read the other 50 columns
  • Happier CPU caches — processing contiguous arrays of the same type fits modern CPU architectures perfectly

OLAP workloads benefit enormously from columnar storage because they typically touch a small number of columns across many rows.

Vectorized Execution
Like a teacher grading Question 1 for all students, then Question 2 for all students — staying in one mental mode makes you faster and lets you spot patterns.

DuckDB processes data in vectors (batches) of 2,048 values. Each operation — filtering, summing, comparing — runs over the entire vector at once using tight loops that the CPU can optimize with SIMD instructions.

Vectorized execution dramatically reduces the per-row overhead of traditional row-at-a-time engines. Instead of calling a virtual function for every single row, DuckDB calls it once per batch of 2,048 — eliminating thousands of function-call overheads.

Extensions
Like apps on your phone — DuckDB does a lot out of the box, but you install extensions when you need spatial queries, cloud storage access, or specific file formats.

DuckDB's extension system allows adding capabilities at runtime:

  • File formats — Parquet, Iceberg, Lance, Delta Lake
  • Remote access — S3, HTTP, Azure Blob Storage
  • Domain functions — spatial/GIS, full-text search
  • Database connectors — PostgreSQL, MySQL, SQLite scanners

Core extensions auto-install when you first use them. For example, querying a Parquet file for the first time will automatically load the Parquet extension.

Data Virtualization
Instead of photocopying a library book to read at home, DuckDB lets you read the book right on the shelf — querying files directly without importing them.

Zero-copy querying means DuckDB can query Parquet files, CSV files, Pandas DataFrames, and even remote S3 objects directly without importing or copying the data first.

This is powered by DuckDB's "replacement scan" feature, which intercepts references to external objects (like a pandas DataFrame variable name in SQL) and transparently reads them in-place. No ETL step required.

Single-File Persistence
Just like a SQLite .db file or an Excel .xlsx file — one file you can email, scp to a server, or check into version control. No WAL directories, no configuration files.

DuckDB databases use the .duckdb extension by default. They support ACID transactions via a custom MVCC implementation and allow multiple concurrent readers with a single writer.

No WAL directories, no tablespaces, no configuration files — just one portable file that contains your entire database.

Putting it all together These six concepts work in concert. DuckDB runs in-process (no server overhead), stores data in columns (fast analytical scans), processes those columns in vectors (CPU-optimized batches), extends its capabilities via extensions (modular design), reads external data via virtualization (no ETL), and persists everything in a single file (zero configuration). The result: a database that is as easy to start as import duckdb yet powerful enough to crunch billions of rows on a laptop.
