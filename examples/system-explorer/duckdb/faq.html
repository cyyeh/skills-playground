Frequently Asked Questions

Answers to the most common questions about DuckDB's capabilities, limits, and design decisions. Click any question to expand the answer.

By default, DuckDB allocates up to 80% of physical RAM for its buffer pool. This is configurable with SET memory_limit = '4GB'. Memory is divided between the buffer pool (cached pages + temporary results) and overhead (internal structures, catalog metadata). If intermediate results exceed the limit, DuckDB spills to the temp_directory on disk automatically. Set temp_directory = '' to disable spilling (queries exceeding memory will fail instead).

No. DuckDB uses a single-writer model. Multiple processes can read the same .duckdb file simultaneously in read-only mode, but only one process can open it for writing. Within a single process, multiple threads can read and write concurrently using MVCC. Workarounds include using MotherDuck for multi-user access, designing a single-writer/multi-reader architecture, or using separate DuckDB files per writer.

In-memory mode (duckdb.connect()) is the fastest -- no disk I/O -- but data is lost when the process exits. Persistent mode (duckdb.connect('my_db.duckdb')) writes data to a single file with WAL-based crash recovery. The database file is typically 30-50% the size of raw data thanks to compression. You can attach both in-memory and persistent databases to the same connection with ATTACH.

Three mechanisms: (1) Streaming execution reads and processes data one chunk at a time. (2) Intermediate spilling writes excess hash tables and sort buffers to temporary files. (3) Buffer eviction frees cached pages using LRU-like eviction. Configure with SET memory_limit, SET temp_directory, and SET max_temp_directory_size. Warning: spilling is 10-100x slower than in-memory processing and targets occasional overflows, not routine larger-than-memory workloads.

Extensions run as native code in the same process with full system access -- there is no sandboxing. Core extensions are cryptographically signed by the DuckDB team. Community extensions require SET allow_unsigned_extensions = true. In production, pin extension versions, use only signed extensions, and audit community extensions before enabling them.

DuckDB is 10-100x faster for aggregations on GB+ datasets, uses columnar compressed storage (vs Pandas' eager row-oriented loading), and parallelizes automatically. Pandas is single-threaded with ~1-5 GB practical limit. They are complementary: use DuckDB for heavy lifting (joins, aggregations) and Pandas for final-mile operations (plotting, ML input). Zero-copy integration means no serialization overhead between them.

Yes, with caveats. Good patterns: read-only analytics endpoints, single-writer with read-only API servers, per-request in-memory databases. Anti-patterns: using DuckDB as the primary transactional database, multiple API servers writing to the same file, or expecting high-concurrency transactional performance. DuckDB is optimized for fewer, heavier analytical queries.

Maximum database size: limited by disk space. Maximum memory: configurable (default 80% RAM). Max columns: ~100,000 per table. Max row group: ~122,880 rows. Max string: ~4 GB per value. Concurrent writers: 1 per database file. Concurrent readers: unlimited. DuckDB performs best on datasets up to ~100-200 GB on modern machines.
