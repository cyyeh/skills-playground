Getting Started

The Basics

Essential questions for anyone evaluating DuckDB for the first time.

DuckDB works well with datasets up to roughly 2 TB on a single machine. The experience varies based on where your data sits relative to RAM:

  • Fits in RAM — full speed. Analytical queries over in-memory columnar data are extremely fast thanks to vectorized execution.
  • Larger than RAM — still works. DuckDB spills intermediate results to disk transparently, though queries will be slower due to I/O.
  • Beyond ~2 TB — time to look at distributed systems like ClickHouse, Snowflake, or Spark that can spread work across multiple nodes.

DuckDB is at its best with datasets between 1 MB and 200 GB. In this range you get near-instant query performance on commodity hardware with zero infrastructure overhead — no clusters, no cloud bills, no configuration files.

DuckDB is fully ACIDAtomicity, Consistency, Isolation, Durability — four guarantees that database transactions are processed reliably. Atomicity means all-or-nothing, Consistency means valid state, Isolation means concurrent transactions don't interfere, Durability means committed data survives crashes.-compliant and uses WALWrite-Ahead Log — a technique where changes are first written to a sequential log before being applied to the main data file. This ensures crash recovery: after a restart, the database replays the log to restore a consistent state.-based recovery to protect your data. Here is what that means in practice:

  • Uncommitted transactions are rolled back automatically on restart — partial writes never corrupt the database.
  • The database file remains consistent — DuckDB uses atomic file operations to ensure the on-disk state is always valid.
  • Committed data is never lost — once a transaction commits, the data is durable even if the process crashes immediately after.

Unlike systems that keep the WAL as separate files, DuckDB embeds the write-ahead log inside the single .duckdb database file. This keeps the "single file, no extra configuration" simplicity intact.

Schema & Performance

Techniques for managing evolving data models and tuning query performance.

Schema evolutionSchema Evolution — the process of modifying a database or file schema over time (adding columns, renaming fields, changing types) while maintaining backward compatibility with existing data. — changing your table or file structure over time — is handled at multiple levels in DuckDB:

Native tables (SQL DDL):

  • ALTER TABLE ... ADD COLUMN — add new columns with default values
  • ALTER TABLE ... DROP COLUMN — remove columns no longer needed
  • ALTER TABLE ... RENAME COLUMN — rename columns in place

Parquet files:

DuckDB has built-in schema evolution support for Parquet. When querying multiple Parquet files where columns have been added or removed over time, DuckDB automatically fills missing columns with NULL. This makes it straightforward to query a directory of Parquet files even if the schema changed between writes.

Iceberg tables:

DuckDB respects Apache Iceberg v2/v3 schema evolution semantics, including column adds, drops, renames, and type promotions. The Iceberg extension reads the table metadata to handle schema changes transparently.

If you are running the same or similar queries repeatedly, these four strategies can significantly improve performance:

  1. Use persistent tables in a .duckdb file — Instead of re-reading CSV or Parquet files every time, import data into a persistent DuckDB database. DuckDB's native columnar format with compression and statistics is optimized for repeated reads.
  2. Add ART indexesAdaptive Radix Tree — DuckDB's index structure, optimized for point lookups and range queries. For analytical (scan-heavy) workloads, indexes often hurt more than they help because they add write overhead without benefiting full-column scans. cautiously — ART indexes help with point lookups and exact-match filters, but for typical analytical workloads (aggregations, scans), they can actually slow things down by adding write-time overhead. Only add them when profiling confirms a bottleneck on specific lookups.
  3. Sort data for zone mapZone Map — metadata stored per column chunk that records the minimum and maximum values in that chunk. The query engine uses zone maps to skip entire chunks whose min/max range doesn't overlap with the query filter, dramatically reducing I/O. effectiveness — DuckDB stores min/max statistics per column chunk. If your data is sorted by a frequently filtered column (e.g., timestamp), DuckDB can skip entire chunks that fall outside the filter range, turning full scans into targeted reads.
  4. Use Parquet with row group statistics — When writing Parquet files, ensure row group statistics are enabled (they are by default). DuckDB reads these statistics to prune row groups before scanning, achieving the same zone-map-style skip optimization on external files.
