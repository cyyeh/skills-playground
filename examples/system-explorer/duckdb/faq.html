DuckDB works well with datasets up to roughly 2 TB on a single machine. The actual limit depends on available RAM and disk:

  • Datasets that fit in RAM: Full speed, all operations in memory.
  • Datasets larger than RAM: DuckDB spills intermediate results to disk. Queries still work but slow down for operations requiring large intermediates (like big hash joins).
  • Beyond ~2 TB: Consider ClickHouse, Snowflake, or Spark.

The sweet spot is 1 MB to 200 GB -- large enough that pandas struggles, small enough that you don't need a cluster.

Yes, but understand its concurrency model. DuckDB supports multiple concurrent readers but only a single writer. This makes it well-suited for:

  • Read-heavy analytical dashboards
  • Batch processing pipelines
  • Embedded analytics in applications
  • Scheduled reporting jobs

It is not suited for workloads with many concurrent write transactions -- use PostgreSQL or MySQL for that.

Many data professionals use both: DuckDB for heavy lifting (aggregations, joins, filtering), then convert results to pandas for visualization or ML.

DuckDB is ACID-compliant with WAL-based recovery. If a crash occurs during a write:

  • Uncommitted transactions are rolled back
  • The database file remains consistent
  • Committed data is never lost

The WAL is embedded in the single database file -- there is no separate WAL directory that could be lost or cause corruption.

For many workloads, yes. Companies have replaced cloud data warehouses for:

  • Cost savings -- DuckDB is free; cloud warehouses charge per query or per TB scanned
  • Latency -- No network round-trip; queries on local data are faster
  • Simplicity -- No infrastructure to manage

You would still need a cloud warehouse when you require multiple teams querying concurrently, petabyte-scale data, built-in governance and access control, or real-time dashboard serving with high concurrency.

DuckDB supports standard SQL DDL: ALTER TABLE ADD COLUMN, DROP COLUMN, RENAME COLUMN. For file-based workflows (querying Parquet directly), schema evolution is handled by Parquet's built-in support -- DuckDB reads available columns and fills missing ones with NULL. For Iceberg tables, DuckDB respects Iceberg's schema evolution semantics (v2 and v3).

Several approaches:

  • Persistent tables -- Load data into a .duckdb file instead of querying raw files each time
  • Indexes -- Create ART indexes for frequently filtered columns (use cautiously -- analytical workloads often do not benefit)
  • Sorted data -- Store data sorted by commonly filtered columns so zone maps can skip more row groups
  • Parquet with row group statistics -- Write sorted Parquet files so DuckDB's predicate pushdown is maximally effective
