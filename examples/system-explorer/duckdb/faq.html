No. DuckDB enforces a single-writer-per-database-file constraint at the process level. Within a single process, multiple threads can write concurrently (appends never conflict; same-row updates fail for the second writer). Workarounds include: a single writer process with a request queue, writing to Parquet files and ingesting them, delegating writes to Postgres/MySQL via extensions, or using MotherDuck for multi-user cloud access.

DuckDB transparently spills intermediate results to disk when memory exceeds memory_limit. It uses the temp_directory setting for overflow. It can also stream from Parquet/CSV without loading them fully. However, performance degrades when spilling, especially for large hash joins. The key insight: DuckDB requires enough RAM for the largest intermediate result, not the entire dataset.

DuckDB has processed hundreds of gigabytes on single machines. The practical limit depends on: available RAM (performance), disk speed (spilling), and patience. For scan-and-aggregate queries, DuckDB handles datasets much larger than RAM. For complex multi-way joins on very large tables, diminishing returns appear at tens-of-GB to low-TB. Multi-TB workloads are better served by distributed systems.

Each major version may increment the storage format version. DuckDB reads older versions but writes in the current one. Since v1.2.0, you can pin STORAGE_VERSION. For migrations, use EXPORT DATABASE (Parquet/CSV) and IMPORT DATABASE on the new version. The LTS release (v1.4.x, supported until September 2026) provides stability for production.

With caveats. DuckDB can serve analytical queries if a single application process manages the database. It is NOT a multi-user database server. For read-heavy dashboards, DuckDB behind a single app server with connection pooling works well. For multi-user, multi-writer apps, use MotherDuck or a traditional OLTP database for writes.

DuckDB is typically faster and more memory-efficient. Pandas materializes entire DataFrames in memory; DuckDB uses lazy evaluation, columnar compression, and spill-to-disk. DuckDB queries pandas DataFrames directly (zero-copy via Arrow) using a fraction of the memory. The trade-off: SQL vs. the pandas API. For workloads exceeding a few GB, DuckDB is almost always faster.

Full ACID transactions with a write-ahead log (WAL). Changes go to the WAL first, then checkpoint to the main file (controlled by checkpoint_threshold). On crash, the WAL is replayed on next open. Since v1.4.0, AES-256 encryption is available. Since v1.5.0, checkpointing is non-blocking.

All official extensions are cryptographically signed. DuckDB verifies signatures before loading. In untrusted environments, set enable_external_access = false to prevent file I/O and extension loading entirely. Community extensions are also signed through the build process, but audit them like any third-party dependency.
