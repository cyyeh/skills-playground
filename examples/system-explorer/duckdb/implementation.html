Getting Started

Bash
# Install via pip
pip install duckdb
# Or via conda
conda install python-duckdb -c conda-forge
# CLI tool (new in 1.5.0)
pip install duckdb-cli
Python
import duckdb

# Minimal in-memory query
duckdb.sql("SELECT 42 AS answer").show()

# Query files directly -- no CREATE TABLE needed
duckdb.sql("SELECT * FROM 'sales_2025.parquet' WHERE region = 'APAC' LIMIT 10").show()

# Persistent database with context manager
with duckdb.connect("analytics.db") as con:
    con.sql("CREATE TABLE events (ts TIMESTAMP, user_id INT, action VARCHAR)")
    con.sql("SELECT action, COUNT(*) FROM events GROUP BY action").show()

Configuration Essentials

SQL
-- Memory management
SET memory_limit = '8GB';
SET temp_directory = '/tmp/duckdb';
SET max_temp_directory_size = '50GB';

-- Parallelism
SET threads TO 4;

-- Profiling and debugging
SET enable_profiling = 'json';
SET enable_progress_bar = true;
💡
Memory sizing: Set memory_limit to 60-80% of available RAM. DuckDB will spill to disk via temp_directory when exceeded, but spilling degrades performance significantly for large joins and sorts.

Code Patterns

ETL Pipeline with Parquet Partitioning

Python
import duckdb

con = duckdb.connect("warehouse.db")

# Ingest raw CSV data
con.sql("""
    CREATE TABLE raw_events AS
    SELECT * FROM read_csv('events_*.csv', auto_detect=true, header=true)
""")

# Transform and export as partitioned Parquet
con.sql("""
    COPY (
        SELECT date_trunc('month', event_time) AS month,
               event_type, COUNT(*) AS event_count
        FROM raw_events GROUP BY ALL
    ) TO 'output' (FORMAT PARQUET, PARTITION_BY (month), COMPRESSION ZSTD)
""")

Multi-Source Analytics

SQL
-- Attach a Postgres database
INSTALL postgres;
LOAD postgres;
ATTACH 'dbname=production host=db.example.com' AS pg (TYPE POSTGRES, READ_ONLY);

-- Cross-database analytical query
SELECT pg.customers.name,
       COUNT(legacy.orders.id) AS order_count
FROM pg.customers
JOIN legacy.orders ON pg.customers.id = legacy.orders.customer_id
GROUP BY pg.customers.name;

Deployment Considerations

  • Thread tuning: DuckDB defaults to all cores. In shared environments, set threads explicitly to avoid contention.
  • Concurrency: Single-writer-per-file constraint. Use a single process with connection pooling for web apps.
  • Storage versioning: Major versions may increment the storage format. Use EXPORT DATABASE / IMPORT DATABASE to migrate.
  • Security: Set enable_external_access = false in untrusted environments. AES-256 encryption available since v1.4.0 LTS.
⚠️
Production tip: Pin to LTS releases (currently v1.4.x, supported until September 2026) and plan for migration effort during major version upgrades.
