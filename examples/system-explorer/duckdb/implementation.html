Python Integration Patterns

DuckDB's Python API offers deep integration with the data science ecosystem. Here are the most common patterns:

Zero-Copy DataFrame Queries

Python
import duckdb
import pandas as pd

# Zero-copy query on a pandas DataFrame
df = pd.read_csv('large_file.csv')
result = duckdb.sql("SELECT category, AVG(price) FROM df GROUP BY category")

Relation API Chaining

Python
# Chain operations fluently
(duckdb
    .read_parquet('s3://bucket/data/*.parquet')
    .filter("year = 2025")
    .aggregate("region, SUM(revenue) AS total")
    .order("total DESC")
    .limit(10)
    .show())

Persistent Analytics Store

Python
con = duckdb.connect('analytics.duckdb')
con.sql("""
    CREATE OR REPLACE TABLE daily_metrics AS
    SELECT date_trunc('day', timestamp) AS day,
           COUNT(*) AS events,
           COUNT(DISTINCT user_id) AS users
    FROM read_parquet('events_*.parquet')
    GROUP BY 1
""")
💡
Tip: Use .fetchdf() to get results as a pandas DataFrame, .fetchnumpy() for NumPy arrays, or .arrow() for zero-copy Apache Arrow tables. Each avoids unnecessary data copies.

Advanced SQL Features

DuckDB extends standard SQL with powerful features that simplify analytical queries. These are among the most useful:

SQL
-- Apply operations to multiple columns at once
SELECT MIN(COLUMNS('amount_*')), MAX(COLUMNS('amount_*'))
FROM transactions;

-- Select all columns except specific ones
SELECT * EXCLUDE (internal_id, debug_flag)
FROM users;

COLUMNS expression is DuckDB-specific and eliminates tedious column listing in wide tables.

SQL
-- Time-series aligned join: match each trade to the
-- most recent price at or before the trade timestamp
SELECT t.*, p.price
FROM trades t ASOF JOIN prices p
ON t.ticker = p.ticker AND t.timestamp >= p.timestamp;

ASOF joins solve a common time-series problem without self-joins or window functions.

SQL
-- Pivot rows to columns
PIVOT sales ON region USING SUM(amount);

-- List comprehensions and lambdas
SELECT list_transform([1, 2, 3, 4], lambda x: x * x);
SQL
-- DuckDB-specific: write FROM before SELECT
FROM sales
SELECT region, SUM(amount)
WHERE year = 2025
GROUP BY region;

FROM-first syntax lets you specify the data source first, which feels more natural for exploratory queries and enables better IDE autocomplete.

Configuration & Tuning

SQL
-- Memory limit (default: 80% of system RAM)
SET memory_limit = '8GB';

-- Thread count (default: all cores)
SET threads = 4;

-- Temp directory for spill-to-disk
SET temp_directory = '/fast-ssd/duckdb-tmp';

-- Enable progress bar for long queries
SET enable_progress_bar = true;

-- Disable insertion order for faster bulk loads
SET preserve_insertion_order = false;
⚠️
Warning: Setting preserve_insertion_order = false can significantly speed up bulk loads and queries, but results may come back in a different order than inserted. Only disable this when order does not matter.

Deployment Patterns

DuckDB fits into several deployment patterns depending on your use case:

Python
# One-off analysis -- no server, no setup
import duckdb
duckdb.sql("""
    SELECT country, COUNT(*) as users
    FROM 's3://analytics/users_*.parquet'
    GROUP BY country ORDER BY users DESC
""").show()
Python
con = duckdb.connect()
con.sql("""
    COPY (
        SELECT a.user_id, a.event, b.plan_type
        FROM read_parquet('events/*.parquet') a
        JOIN read_csv('users.csv') b ON a.user_id = b.id
        WHERE a.timestamp > '2025-01-01'
    ) TO 'output/enriched.parquet'
      (FORMAT PARQUET, COMPRESSION ZSTD)
""")
Python
from fastapi import FastAPI
import duckdb

app = FastAPI()
db = duckdb.connect('analytics.duckdb', read_only=True)

@app.get("/metrics/{metric}")
def get_metric(metric: str, days: int = 30):
    result = db.execute("""
        SELECT date, value FROM metrics
        WHERE metric_name = ? AND date > current_date - ?::INTEGER
    """, [metric, days]).fetchdf()
    return result.to_dict(orient='records')
SQL
INSTALL iceberg;
LOAD iceberg;

-- Query an Iceberg table on S3
SELECT * FROM iceberg_scan('s3://lakehouse/db/sales');

-- Time travel to a previous version
SELECT * FROM iceberg_scan('s3://lakehouse/db/sales', version := '2');
ℹ️
Note: For web applications, always open the database in read_only=True mode when only serving read queries. This allows multiple concurrent connections without write locking.
