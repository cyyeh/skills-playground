Should You Use DuckDB?

Use this interactive decision tree to determine if DuckDB is the right fit for your use case.

Is DuckDB Right for Your Project?

Is your primary workload analytical queries (aggregations, scans, joins over many rows)?

Do you need multiple processes writing to the same database concurrently?

Is your dataset larger than 1 TB?

Consider SQLite or PostgreSQL. DuckDB is optimized for analytical (OLAP) workloads. For high-frequency small transactions (OLTP), a row-oriented database will perform better.
Consider ClickHouse or a traditional data warehouse. DuckDB's single-writer constraint means only one process can write at a time. For multi-writer distributed systems, you need a server-based solution.
Consider a distributed system. For multi-TB workloads, ClickHouse, BigQuery, or Snowflake can distribute queries across a cluster. DuckDB handles larger-than-memory datasets by spilling to disk, but multi-TB joins will be slow.
DuckDB is an excellent fit! For analytical queries on datasets up to hundreds of GB with a single writer process, DuckDB delivers outstanding performance with zero infrastructure overhead. Just pip install duckdb and go.

When to Use DuckDB

  • Local data analysis: Query CSV, Parquet, or JSON files with SQL without a server
  • Embedded analytics: Add analytical capabilities to desktop, mobile, or web apps (DuckDB-Wasm)
  • ETL/ELT pipelines: Process and transform data locally before pushing to a warehouse
  • CI/CD data testing: Validate data quality with near-zero startup time
  • Replacing pandas: Process large datasets with SQL using a fraction of the memory
⚠️
When NOT to use DuckDB: High-concurrency OLTP workloads, multi-writer distributed systems, real-time streaming ingestion, multi-user server deployments, or multi-TB datasets needing distributed processing.
