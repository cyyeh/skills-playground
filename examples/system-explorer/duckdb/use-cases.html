Use Cases & Case Studies

When to Use DuckDB

🔍

Ad-Hoc Data Exploration

Query CSV, Parquet, or JSON files directly with zero setup. Auto-detects schemas. Data journalists use DuckDB to explore public datasets instantly.

🧪

Data Science & ML

Zero-copy integration with Pandas, Polars, Arrow. Hex achieved 5-10x speedups for notebook data transformations.

🔄

ETL Pipelines

FinQore replaced Spark and reduced ETL from 8 hours to 8 minutes. No cluster needed for medium datasets.

📊

Embedded Analytics

South Australia's climate dashboard runs entirely in the browser via DuckDB-WASM. No server-side query processing.

✅

CI/CD Testing

dbt-duckdb runs test suites without a cloud warehouse. Near-zero startup, full SQL compatibility.

📡

Edge & IoT

~25 MB binary with no dependencies. Runs analytics on devices with as little as 256 MB RAM.

Should You Use DuckDB?

Decision Guide

Is your primary workload analytical (aggregations, scans, joins over many rows)?

Does your dataset fit on a single machine (under ~200 GB working set)?

Do you need concurrent writes from multiple processes?

Consider PostgreSQL, MySQL, or SQLite. DuckDB is optimized for analytical workloads, not transactional ones with frequent small INSERT/UPDATE/DELETE operations.
Consider ClickHouse, BigQuery, or Snowflake. For datasets exceeding a single machine, you need a distributed system. DuckDB is single-node only.
Consider MotherDuck or ClickHouse. DuckDB supports only a single writer per database file. MotherDuck provides cloud-based multi-user DuckDB access.
DuckDB is an excellent fit! Single-machine analytical workloads with single-writer access are DuckDB's sweet spot. You get data warehouse-grade performance with zero infrastructure.
