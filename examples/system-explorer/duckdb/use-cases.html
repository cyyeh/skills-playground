Use Cases

When to Use DuckDB

DuckDB shines for single-machine analytical workloads. Here are the use cases where it fits best:

🔍
Ad-hoc Data Analysis

SQL on local files with zero setup. Faster than pandas for aggregations and joins on large datasets.

🔄
ETL Pipelines

Read from multiple formats (CSV, Parquet, JSON), transform with SQL, and write output -- all without a server.

🧭
Data Exploration

Query CSVs, Parquet, and JSON files directly without an import step. Auto-detects schemas and types.

📊
Embedded Analytics

Add analytical query capabilities to your application without deploying a separate database server.

✅
CI/CD Data Testing

Fast startup, no infrastructure -- ideal for validating data pipelines in automated testing environments.

🔒
Privacy-Sensitive Analysis

Data never leaves the local machine. No cloud uploads, no network traffic -- just local computation.

Is DuckDB Right for Your Use Case?

Is your workload primarily analytical (scans, aggregations, joins) rather than transactional (single-row inserts/updates)?

Does your dataset fit on a single machine (under ~2 TB)?

Do you need many concurrent users writing to the same database simultaneously?

DuckDB is an excellent fit. Your workload is analytical, fits on one machine, and does not require many concurrent writers. You will benefit from zero-configuration deployment, fast vectorized execution, and direct file querying.
Consider PostgreSQL or MySQL instead. DuckDB is optimized for analytical scans and aggregations, not transactional workloads with frequent single-row inserts and updates.
Consider ClickHouse, Snowflake, or Spark. For datasets beyond ~2 TB, you need a distributed system. DuckDB runs on a single machine and has no built-in sharding or replication.
Consider a cloud data warehouse or MotherDuck. DuckDB supports only a single writer. For team-wide concurrent access, use Snowflake, BigQuery, or MotherDuck (cloud-hosted DuckDB with shared databases).

Real-World Usage

Companies across industries have adopted DuckDB for production workloads:

🌍
Watershed

Uses DuckDB for data pipeline operations, converting activity data into carbon footprint calculations by querying Parquet files directly.

☁️
MotherDuck

Cloud service built on DuckDB. Users ATTACH 'md:' to offload heavy queries to cloud compute while keeping the local DuckDB experience.

📈
Rill Data

Built their BI dashboard engine on DuckDB, powering interactive exploration of large datasets with sub-second query times.

🌐
Evidence

Built Universal SQL with DuckDB-Wasm, enabling browser-based interactive dashboards that query data locally in WebAssembly.

💡
Tip: A common production pattern is replacing expensive cloud analytics services (BigQuery, Athena) with DuckDB running in scheduled cloud functions, processing log files and Parquet from object storage at a fraction of the cost.
