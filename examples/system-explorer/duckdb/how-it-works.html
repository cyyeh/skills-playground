The Vector Format

DuckDB's internal data representation uses several vector types for efficiency:

Flat Vector

Standard contiguous array. A column of integers read from storage. The most common type.

Constant Vector

Single value repeated for all rows. The literal 2025 in WHERE year = 2025 -- stored once, not 2,048 times.

Dictionary Vector

Index into a smaller value set. A dictionary-compressed string column stores codes, not full strings.

Sequence Vector

Offset + increment pattern. Row IDs (0, 1, 2, 3, ...) need only two numbers to represent the entire vector.

Operators output one of these types, and the next operator handles any type through the Unified Vector Format -- a common abstraction that avoids the combinatorial explosion of type-pair handling.

💡
Tip: String handling is particularly clever: strings of 12 bytes or fewer are inlined directly into the vector (no pointer indirection). Larger strings store a pointer plus a 4-byte prefix -- the prefix allows many comparisons (like WHERE name > 'M') to short-circuit without following the pointer.

Compression Algorithms

DuckDB automatically selects the best compression for each column segment. The compression analyzer runs during data loading and picks the algorithm yielding the best ratio independently per segment.

Direct File Querying

One of DuckDB's most powerful features is querying files directly without loading them first. DuckDB's multi-hypothesis CSV parser automatically infers column types, delimiters, and headers. For Parquet files, it leverages metadata (row group statistics, page indexes) to push down filters and skip irrelevant data before reading it.

SQL
-- Query a local Parquet file
SELECT * FROM 'sales_2025.parquet' WHERE region = 'APAC';

-- Query a CSV with auto-detection
SELECT * FROM read_csv('data.csv');

-- Query Parquet files on S3
SELECT * FROM 's3://my-bucket/data/*.parquet';

-- Query a remote CSV over HTTP
SELECT * FROM 'https://example.com/data.csv';
⚠️
Warning: When querying remote files over S3 or HTTP, the initial query may be slower due to network latency. For repeated queries on the same remote data, consider loading it into a persistent DuckDB table first with CREATE TABLE ... AS SELECT.
