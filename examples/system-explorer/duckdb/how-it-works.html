How It Works

Vectorized Query Execution

DuckDB's execution engine processes data in vectors of STANDARD_VECTOR_SIZE (2,048) tuples. Each vector represents a single column's values within a batch. Multiple vectors are bundled into a DataChunk that flows through the operator pipeline.

Vector Types

Vector TypeStorageUse Case
FlatContiguous array, 1:1 mappingDefault format after computation
ConstantSingle value for all rowsLiteral expressions (WHERE x > 42)
DictionaryChild vector + selection indexCompressed data from storage
SequenceOffset + incrementRow identifiers, sequential data

The Unified Vector Format provides a generic interface so operators don't need to handle every combination. Operators call ToUnifiedFormat() on any vector to get a normalized representation.

💡
String optimization: Strings of 12 bytes or fewer are inlined directly in the string_t struct, avoiding heap allocation. Longer strings store a pointer, length, and a 4-byte prefix used for fast comparison short-circuiting.

Columnar Storage & Compression

Persistent data is organized into row groups (commonly 122,880 rows). Each row group contains column segments, compressed independently using the best-fit algorithm.

AlgorithmData TypeTechnique
ConstantAnySingle value for entire segment
RLEAnyRun-length encoding for repeated values
Bit PackingIntegerMinimal bits per value
Frame of ReferenceIntegerBase value + offsets
DictionaryStringUnique value table + indices
FSSTStringFast Static Symbol Table
ALPFloat/DoubleAdaptive Lossless floating-Point
Chimp/PatasTimestampXOR-based compression
ZstdAnyGeneral-purpose block compression

Zone maps (min/max indexes) are maintained per row group per column. If the filter predicate falls outside the min/max range, the entire row group is skipped.

Morsel-Driven Parallel Execution

DuckDB's parallelism follows the morsel-driven paradigm. Work is divided into small chunks distributed across CPU cores adaptively.

Why morsel-driven? This approach avoids the overhead of exchange operators and enables adaptive load balancing: if one thread finishes its morsel early, it simply picks up the next one from the queue.

Transaction & Concurrency

DuckDB's MVCC implementation is inspired by Thomas Neumann's paper on fast serializable MVCC:

  • Writers update data in-place but save previous versions in undo buffers
  • Readers check for version info per row; if none exists (common case), they read with zero overhead
  • Optimistic concurrency means no locks acquired upfront; conflicts detected at commit time
  • Appends never conflict, even from multiple threads
  • Same-row conflicts produce an error: the second writer must retry
⚠️
Single-writer constraint: Only one process can write to a database file at a time. Multiple processes can read concurrently in READ_ONLY mode. Within a single process, multiple threads can write concurrently.

ℹ️
Benchmark context: On TPC-H benchmarks, DuckDB on a single machine completed the suite in ~76 seconds, while Apache Spark on 32 machines required ~8 minutes. Against ClickHouse, DuckDB often wins on small-to-medium datasets (no network overhead) while ClickHouse wins on larger distributed datasets.
