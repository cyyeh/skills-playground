How It Works

Vectorized Execution Engine

Unlike traditional row-at-a-time engines (the Volcano/iterator model), DuckDB processes data in vectors -- arrays of up to 2048 values per column.

A Vector is a typed array for a single column. A DataChunk is a collection of Vectors -- essentially a small columnar table of up to 2048 rows.

Vector Formats

DuckDB supports multiple physical representations for the same logical data, enabling "compressed execution" -- operations performed directly on compressed data.

FormatDescriptionUse Case
FLATStandard contiguous arrayDefault general-purpose format
CONSTANTSingle value for all elementsLiteral values, broadcast operations
DICTIONARYIndex array + dictionaryLow-cardinality string columns
SEQUENCEStart + increment (no array)Row IDs, generated sequences
FSSTFast Static Symbol Table compressedCompressed string data

Morsel-Driven Parallelism

DuckDB parallelizes query execution using morsel-driven parallelism. The physical plan is split into pipelines at pipeline breaker points, and data chunks (morsels) are distributed to worker threads dynamically.

🔭
Why not exchange-based parallelism? Traditional databases use exchange operators that partition data between workers at fixed points. This leads to skewed workloads, adds overhead even for non-parallel plans, and makes it difficult to adapt parallelism dynamically. Morsel-driven parallelism avoids these issues by distributing work like a work-stealing scheduler.

Buffer Manager

The buffer manager maintains a unified pool of memory for both persistent data pages and temporary data. Default memory limit: 80% of physical RAM.

Larger-Than-Memory Processing

DuckDB handles datasets larger than available memory through three mechanisms:

SQL
-- Configure for larger-than-memory workloads
SET memory_limit = '4GB';
SET temp_directory = '/tmp/duckdb_swap';
SET max_temp_directory_size = '100GB';
⚠️
Warning: Spilling to disk is 10-100x slower than in-memory processing. DuckDB's larger-than-memory support targets occasional overflows, not routine processing of datasets far exceeding RAM.

Storage Format

DuckDB uses a custom single-file storage format. Data is organized hierarchically into row groups (default ~122K rows), with each column stored as compressed segments.

Compression Algorithms

Each column segment is compressed independently using the best algorithm for its data distribution. DuckDB achieves typical compression ratios of 3-10x.

AlgorithmBest For
ConstantColumns where all values are the same
RLESorted or clustered data with repeated runs
BitPackingInteger columns with small value ranges
DictionaryLow-cardinality string columns
FSSTHigh-cardinality string columns
ALPDouble-precision floating-point data
💡
Tip: DuckDB automatically selects the best compression algorithm per segment during checkpointing. You do not need to configure compression manually. The database file is typically 30-50% the size of raw data.
