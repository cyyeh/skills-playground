How It Works

Experiment Tracking Internals

When you call mlflow.start_run(), the client creates a Run entity in the backend store via the tracking server's REST API. The run gets a unique UUID and is associated with an experiment. As training progresses, each API call sends data to the server.

mlflow.start_run() creates a Run entity with a UUID in the backend store. The run is linked to the active experiment.

How Autologging Works

When you call mlflow.autolog(), MLflow monkey-patches supported ML libraries to automatically intercept training calls. Here's what happens under the hood:

The patches inject logging calls that capture training parameters (hyperparameters from the model constructor), training metrics (loss and accuracy at each epoch), the trained model as an artifact, and input data signatures. For deep learning frameworks, autologging hooks into model.fit() and Trainer.train() to capture per-epoch metrics automatically.

💡
Key detail: The MLflow fluent tracking API is not currently thread-safe. For concurrent logging from multiple threads, use the lower-level MlflowClient API, or implement mutual exclusion manually. For distributed training, log only from rank 0 to avoid duplicate entries.

How the Model Registry Works

Once a model is logged, registering it creates a governance layer:

mlflow.register_model() creates a RegisteredModel entry (if new) and a ModelVersion linked to the logged artifact. Each version maintains full lineage: the run ID, experiment ID, and artifact path. Model aliases like @champion provide stable references that decouple consumers from version numbers — promoting a new model means reassigning the alias, not changing downstream code.

💡
Alias vs. Stages: Model stages (Staging, Production, Archived) are deprecated in MLflow 3.x. Aliases are more flexible — you can define arbitrary names, reassign them atomically, and a single version can have multiple aliases. Use client.set_registered_model_alias("my-model", "champion", version=3).

How Tracing Works (GenAI)

MLflow Tracing instruments GenAI applications using OpenTelemetry-compatible spans. Auto-tracing patches supported libraries (OpenAI, Anthropic, LangChain, etc.) to wrap every LLM call, tool invocation, and retrieval step in a trace span.

Each span records inputs, outputs, latency, token counts, and cost metadata. Spans are nested to reflect the call hierarchy — an agent trace might contain spans for the orchestrator, a retrieval tool, and multiple LLM calls. Context propagation supports distributed tracing across microservices.

How Model Serving Works

When you run mlflow models serve, MLflow loads the model using its python_function flavor and starts a web server exposing /invocations and /ping endpoints. Input data is validated against the model's recorded signature before prediction.

For production, MLflow integrates with Docker, Kubernetes, AWS SageMaker, Azure ML, and Databricks Model Serving. Model aliases in the registry enable zero-downtime model swaps — updating the @champion alias automatically routes traffic to the new version.

⚠
Production note: The built-in mlflow models serve is suitable for development and low-traffic use. For production-scale serving with autoscaling, health checks, and canary deployments, use a dedicated serving platform (KServe, SageMaker, Databricks) with MLflow models as the artifact source.
