How It Works

Step-by-step walkthroughs of the most common MLflow workflows.

Workflow 1: Logging an Experiment

Manual Logging

The fundamental workflow: set up tracking, create an experiment, start a run, log data, and view results.

python 
import mlflow

# Step 1: Point at a tracking server
mlflow.set_tracking_uri("http://my-tracking-server:5000")

# Step 2: Create or select an experiment
mlflow.set_experiment("fraud-detection-v2")

# Step 3: Start a run and log everything
with mlflow.start_run():
    mlflow.log_param("learning_rate", 0.01)
    mlflow.log_param("n_estimators", 200)

    # ... your training code here ...

    mlflow.log_metric("accuracy", 0.943)
    mlflow.log_metric("f1_score", 0.917)
    mlflow.log_artifact("confusion_matrix.png")
    mlflow.sklearn.log_model(model, "model")
⚙
Under the hood: start_run() sends a POST /api/2.0/mlflow/runs/create to the tracking server. Each log_* call sends its own REST request. Metadata goes to the backend store (database), artifacts go to the artifact store (e.g., S3).

Workflow 2: Autologging

Zero-Code Tracking

Instead of manual log_param and log_metric calls, autologging captures everything automatically with a single line.

python 
import mlflow
mlflow.autolog()  # That's it! Everything below is tracked.

from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split

X_train, X_test, y_train, y_test = train_test_split(X, y)
model = RandomForestClassifier(n_estimators=200, max_depth=10)
model.fit(X_train, y_train)
# All params, metrics, model artifact auto-logged!
💡
How it works: Autologging monkey-patches the library's .fit() method. Before .fit() runs, it logs constructor parameters. After .fit() completes, it logs metrics, the model, and framework-specific artifacts (e.g., feature importance). If logging fails, training continues uninterrupted.

Workflow 3: Registering a Model

From Run to Registry

python 
# Step 1: Register from a run
result = mlflow.register_model(
    model_uri="runs:/abc123def456/model",
    name="fraud-detector"
)

# Step 2: Assign an alias
from mlflow import MlflowClient
client = MlflowClient()
client.set_registered_model_alias(
    "fraud-detector", "champion", version=3
)

# Step 3: Load by alias (downstream services)
model = mlflow.pyfunc.load_model("models:/fraud-detector@champion")
predictions = model.predict(new_data)

Workflow 4: Deploying a Model

Start a Flask-based REST server with a /invocations endpoint:

bash 
mlflow models serve -m "models:/fraud-detector@champion" --port 5001

# Test it:
curl -X POST http://localhost:5001/invocations \
  -H "Content-Type: application/json" \
  -d '{"inputs": [[1.0, 2.0, 3.0]]}'

Build a Docker image and run it anywhere:

bash 
mlflow models build-docker \
  -m "models:/fraud-detector@champion" \
  -n fraud-detector-image

docker run -p 5001:8080 fraud-detector-image

Deploy directly to an AWS SageMaker endpoint:

python 
mlflow.sagemaker.deploy(
    app_name="fraud-detector",
    model_uri="models:/fraud-detector@champion",
    region_name="us-west-2",
    mode="create"
)

Deploy to Kubernetes via Seldon Core or KServe plugins:

bash 
# Build the Docker image first
mlflow models build-docker \
  -m "models:/fraud-detector@champion" \
  -n my-registry/fraud-detector:v3

# Push to container registry and deploy via K8s manifest
docker push my-registry/fraud-detector:v3
kubectl apply -f fraud-detector-deployment.yaml

Workflow 5: GenAI Tracing & Evaluation

Observability for LLM Applications

python 
import mlflow

# Step 1: Enable tracing for your LLM library
mlflow.openai.autolog()
# or: mlflow.langchain.autolog()

# Step 2: Every LLM call is now traced
# View traces in the MLflow UI: inputs, outputs,
# latency, token counts, and full request lifecycle

# Step 3: Evaluate agent quality
results = mlflow.evaluate(
    model=my_agent,
    data=eval_dataset,
    model_type="agent",
    evaluators=["correctness", "safety", "relevance"]
)
# Produces scored report with per-example traces
⚠
Note: MLflow tracing is fully OpenTelemetry-compatible. Since MLflow 3.10, native OTel GenAI Semantic Convention support is available, making it easy to export traces to any OTel-compatible backend.
