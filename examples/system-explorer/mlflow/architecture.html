Architecture

High-Level Design

MLflow follows a client-server architecture with a clean separation between metadata storage and artifact storage. The system has three layers: a client SDK layer (Python, TypeScript, Java, R), a service layer (the Tracking Server with REST endpoints and web UI), and a storage layer split into a Backend Store and an Artifact Store.

This separation is deliberate -- metadata queries (searching runs, comparing metrics) have very different access patterns from artifact retrieval (downloading a 2GB model file), so the two storage systems can be scaled and optimized independently.

Key Design Decisions

🔧

Metadata and artifacts separated by design. Many competing systems store everything together. MLflow's split means you can use a fast, queryable Postgres database for metadata while storing multi-gigabyte model artifacts in cheap S3 storage. The tracking server never proxies large file transfers -- artifact uploads go directly to cloud storage.

🔧

The pyfunc universal interface. Instead of requiring deployment tools to understand every ML framework, MLflow defines a single python_function flavor. A Kubernetes serving system only needs to know how to load and call a pyfunc model, and it automatically works with scikit-learn, PyTorch, TensorFlow, and custom models.

🔧

OpenTelemetry for tracing. Rather than building a proprietary tracing format, MLflow adopted OpenTelemetry as its foundation. This means MLflow traces are compatible with the broader observability ecosystem (Jaeger, Zipkin, Datadog) and distributed tracing works out of the box.

🔧

Pluggable store backends. The AbstractStore and ArtifactRepository interfaces let MLflow support diverse infrastructure. A team can start with SQLite + local files, migrate to Postgres + S3, and move to managed Databricks -- all without modifying training scripts.

Data Flow: Tracking a Training Run

End-to-End Request Flow

1Script calls mlflow.set_experiment() and mlflow.start_run()
↓
2SDK sends CreateRun REST request to Tracking Server with experiment ID and generated run UUID
↓
3Server calls SqlAlchemyStore.create_run() -- inserts SqlRun row with status RUNNING
↓
4During training: log_param() and log_metric() calls send REST requests; server persists to DB
↓
5mlflow.sklearn.log_model() serializes model locally, then uploads directory to Artifact Store
↓
6mlflow.end_run() marks run as FINISHED in the backend store
