Architecture

System Overview

MLflow follows a client-server architecture with three core infrastructure components: the Tracking Server, the Backend Store, and the Artifact Store. These can run together on a single machine for development or be distributed across separate hosts for production scalability.

Core Components

Tracking Server

The MLflow Tracking Server is a FastAPI application that exposes REST APIs for logging and querying experiment data. It serves dual roles:

  • API server: Accepts log calls from client SDKs (Python, R, Java) and queries from the UI. Handles run creation, parameter/metric logging, artifact management, and model registry operations.
  • Web server: Hosts the MLflow UI — a React single-page application for visual exploration of experiments, run comparison, metric charts, and model registry management.
  • Artifact proxy: Optionally relays artifact upload/download requests so clients never need direct access to the underlying object store, eliminating the need to distribute storage credentials.

Backend Store

The Backend Store persists structured metadata: experiments, runs, parameters, metrics, tags, and model registry entries. MLflow supports two categories:

  • File-based (SQLite): Default for local development. Writes to a local mlruns/ directory. Simple but unsuitable for concurrent production use due to locking limitations and lack of proper indexing.
  • Database-based: PostgreSQL (recommended), MySQL, or MSSQL via SQLAlchemy. Provides proper indexing, concurrent access, and transactional guarantees required for team-scale deployments.
💡
Production tip: Always use PostgreSQL for production. The UI performance difference between file-based and database-backed stores is dramatic — queries that take seconds with files complete in milliseconds with a properly indexed database.

Artifact Store

The Artifact Store handles large, unstructured files — model binaries, images, datasets, plots. The backend store only records artifact URIs; the actual data lives in the artifact store.

  • Local filesystem: Default (./mlruns). Fine for development.
  • Cloud object stores: Amazon S3, Azure Blob Storage, Google Cloud Storage. The standard for production.
  • Distributed filesystems: HDFS, NFS, SFTP. For on-premises environments.

AI Gateway

The AI Gateway sits as a reverse-proxy layer in front of LLM providers. It maintains route configurations mapping logical endpoint names to provider-specific credentials and models.

  • Authentication: Centralizes API key management — teams access LLMs through the gateway without seeing provider credentials.
  • Cost tracking: Meters token usage and cost per route, per team, per experiment.
  • Trace integration: Every proxied call automatically becomes a linked trace in the MLflow experiment context.
  • Rate limiting: Prevents runaway costs from unbounded LLM calls.

Deployment Topologies

MLflow supports several deployment patterns, from solo development to enterprise-scale:

1. Local Mode

Client, tracking server, backend store, and artifact store all on one machine. The default when you run mlflow ui with no configuration. Good for solo experimentation and getting started.

2. Remote Tracking Server

A central tracking server with a PostgreSQL backend and remote artifact store (S3, GCS). Clients log over HTTP by setting MLFLOW_TRACKING_URI. The standard team deployment for 5-100 data scientists.

3. Tracking Server with Artifact Proxy

The server proxies all artifact operations, so clients only need HTTP access to the server — no object store credentials. Recommended for security-sensitive environments. Enable with --serve-artifacts.

4. Managed MLflow (Databricks)

Fully managed tracking server, model registry integrated with Unity Catalog, serverless model serving, and built-in access control. Removes all infrastructure management. The most feature-complete option for enterprises.
