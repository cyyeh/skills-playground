Core Concepts

The fundamental building blocks of MLflow — the vocabulary you need before diving deeper.

📁

Experiments

An experiment is a named grouping of runs that correspond to a specific task or project — for example, “fraud-detection-v2” or “image-classifier-resnet.”

💡
Analogy: Think of an experiment as a folder on your desk labeled with a project name. Every time you try a different approach, you drop a new sheet of results into that folder.

Why it matters: Experiments organize your work so teams can compare results within a domain and keep the tracking server navigable as usage grows.

▶

Runs

A run is a single execution of ML training code. Each run captures parameters (inputs), metrics (outputs), tags (metadata), artifacts (files), and the source code or Git commit that produced it.

💡
Analogy: A run is one entry in your lab notebook — the specific recipe you tried, the measurements you took, and the sample you produced.

Why it matters: Runs are the atomic unit of reproducibility. By logging everything about a single execution, you can always go back and understand what happened.

📝

Tracking

MLflow Tracking is the logging API and UI for recording and querying runs. You call functions like mlflow.log_param(), mlflow.log_metric(), and mlflow.log_artifact() from your training code.

💡
Analogy: Tracking is the lab notebook itself — the system that captures every experiment's details as they happen, and the index that lets you find them later.

Why it matters: Without tracking, experiment results live in scattered CSVs and terminal output. With tracking, every result is centralized, searchable, and comparable.

🏦

Model Registry

A centralized store for managing the full lifecycle of MLflow Models. It provides model versioning, aliasing (e.g., “champion”, “challenger”), tagging, annotations, and lineage back to the experiment and run that produced each version.

💡
Analogy: The Model Registry is the trophy case for your models. Every model that makes it past experimentation gets a version number and a label before going to production.

Why it matters: In production ML, you need to know exactly which model is serving traffic, who trained it, and how to roll back. The registry provides this governance layer.

📦

MLflow Projects

A convention for packaging ML code into reusable, reproducible units. An MLproject file defines entry points, parameters, and the environment (Conda, Docker, or system) needed to run the code.

💡
Analogy: A project is a recipe card that lists not just the ingredients and steps, but also the kitchen needed to cook the dish. Anyone can reproduce the meal.

Why it matters: Projects solve the “it worked on my machine” problem. Run mlflow run <project-uri> and get the same environment and execution everywhere.

💾

Artifacts

Output files associated with a run — model weights, plots, serialized data, evaluation reports, or any other file. They are stored in an artifact store (local filesystem, S3, Azure Blob, GCS, etc.).

💡
Analogy: Artifacts are the physical samples and photos from your lab work — not the numbers in the notebook, but the tangible outputs you can pick up and use.

Why it matters: Metrics tell you how good a model is, but artifacts give you the actual model. The artifact store ensures every version is preserved and retrievable.

🔌

Model Flavors

A standard packaging format that describes how to load and serve a model. MLflow defines native flavors for frameworks like sklearn, pytorch, tensorflow, xgboost, and more. The universal pyfunc flavor lets any MLflow model be loaded as a generic Python function.

💡
Analogy: Flavors are like universal power adapters. Your model was trained with PyTorch, but the pyfunc flavor lets it plug into any MLflow-compatible deployment target.

Why it matters: Flavors abstract away framework-specific loading code, enabling a single deployment pipeline for models from any framework.

📋

Recipes (Pipelines)

Opinionated, production-ready templates for common ML workflows like classification and regression. Recipes define a step-by-step pipeline from data ingestion through training, evaluation, and registration.

💡
Analogy: If MLflow Projects are recipe cards, Recipes are the full meal-prep kits — pre-portioned ingredients, step-by-step instructions, and quality-check steps built in.

Why it matters: Recipes reduce boilerplate and enforce best practices for teams that want a structured, repeatable ML workflow without building a pipeline framework from scratch.

How They Fit Together

Experiment → Run → Tracking API → Artifacts + Flavors → Model Registry → Deploy

An experiment groups related runs. Each run uses the tracking API to log parameters, metrics, and artifacts (including models saved in specific flavors). Promising models graduate to the model registry, where they are versioned and governed. Projects ensure the code behind any run can be reproduced, and Recipes provide turnkey templates that orchestrate everything into a production-ready pipeline.

Check Your Understanding

What is the difference between parameters and metrics in an MLflow run?

Parameters are the inputs to your training code (e.g., learning rate, number of trees, batch size) — they describe what you tried. Metrics are the outputs (e.g., accuracy, loss, F1 score) — they describe what happened. Parameters are logged once per run; metrics can be logged at multiple steps (e.g., per epoch) to create time-series visualizations.

Check Your Understanding

Why would you use the pyfunc flavor instead of a framework-specific flavor like sklearn?

The pyfunc flavor is the universal adapter. It lets any MLflow model — regardless of framework — be loaded with mlflow.pyfunc.load_model() and called with a simple .predict() method. This is essential for deployment: your serving infrastructure only needs to understand one interface, not separate loaders for scikit-learn, PyTorch, TensorFlow, etc. Use framework-specific flavors when you need framework-specific features (e.g., GPU inference with PyTorch); use pyfunc for generic, portable deployment.
