Core Concepts

💡

Before diving in: MLflow has nine core concepts. Each builds on the previous. If you understand Experiments, Runs, and Models, you understand 80% of what MLflow does. The rest builds on that foundation.

Experiment

A named container that groups related runs together. When you are working on a specific ML task, you create an experiment to hold all the different attempts you make at solving it.

A folder in a filing cabinet labeled "Customer Churn Models" -- it keeps all your attempts on one task organized in one place.

Run

A single execution of your ML training code. Each run captures everything about that particular attempt: the parameters you chose, the metrics you measured, the output files produced (artifacts), and any labels you attach (tags). Runs are the fundamental unit of tracking in MLflow.

One entry in a lab notebook recording a specific experiment attempt -- what you set up, what you measured, and what you produced.

Parameter

A key-value pair representing an input configuration to your run. Parameters are typically hyperparameters (learning rate, batch size, number of layers) but can be any string value. Once logged, parameters are immutable for that run.

The settings on a camera before taking a photo -- they define how the shot (training run) was configured.

Metric

A numerical measurement recorded during or after a run. Metrics have a key, a numeric value, a timestamp, and an optional step number. Unlike parameters, metrics can be logged multiple times per run at different steps, creating a history you can visualize as a curve.

The score on a test that tells you how well your model performed -- and you can track how the score changes over time.

Artifact

Any output file produced by a run -- model weight files, images, data files, or serialized objects. Artifacts are stored in the artifact store separately from metadata because they are often large (gigabytes for deep learning models).

The physical deliverables from a manufacturing process -- the tangible things your training run produced.

Model (MLflow Model)

A standardized packaging format for ML models. An MLflow Model is a directory containing the serialized model, an MLmodel YAML descriptor, dependency specifications, and optionally an input example and signature. The key innovation is the flavor system.

A shipping container that ensures your cargo (the trained model) can be transported and unloaded anywhere regardless of how it was built.

Flavor

A convention that defines how a model from a specific framework should be saved, loaded, and served. The python_function (pyfunc) flavor acts as a universal interface: any model that supports it can be served through the same REST endpoint, loaded as a Spark UDF, or deployed to any platform.

An adapter that lets different plugs fit the same socket -- sklearn, PyTorch, and TensorFlow models all work through the same pyfunc interface.

Model Registry

A centralized store for managing model versions and their lifecycle. Registered models have a name, auto-incrementing version numbers, mutable aliases (like @champion or @challenger), tags, and markdown annotations. Each version links back to the run that produced it.

A library catalog that tracks every edition of every book, who checked it out, and whether it is recommended for readers.

Trace

Captures the full execution path of a GenAI request through your application. Each trace consists of spans representing individual steps arranged in a tree structure. MLflow Tracing is fully OpenTelemetry-compatible.

An X-ray that shows every bone and organ a signal passes through -- you can see every LLM call, tool use, and retrieval step in a request.

How They Fit Together

MLflow Concept Flow

Params + Metrics
→

Production

You create an Experiment to organize your work. Each attempt is a Run that logs Parameters and Metrics. The most important Artifact is usually a Model with one or more Flavors. Ready models go into the Registry for version tracking. For GenAI apps, Traces provide request-level observability in production.

📌

Key insight: The pyfunc flavor is what makes MLflow models truly portable. A serving system only needs to understand pyfunc, and it automatically works with scikit-learn, PyTorch, TensorFlow, Hugging Face, and custom models. This is the single most important design decision in MLflow.
