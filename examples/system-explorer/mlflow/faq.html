Common Q&A

💬

These are real questions a senior engineer would ask when evaluating or operating MLflow. Click any question to expand the answer.

How does MLflow handle concurrent writes from distributed training?

▼
Each worker creates its own run (unique UUID), so there is no contention at the run level. However, concurrent log_metric calls to the same run can cause issues with SQLite (single-writer lock). For distributed training, use PostgreSQL or MySQL which handle concurrent writes through row-level locking. The recommended pattern is to log from the rank-0 process only in data-parallel setups (PyTorch DDP, Horovod).

What happens to registered model versions if I delete the original run?

▼
The model version becomes broken. The registry stores a source URI pointing to the original run's artifacts but does not copy them. Mitigation: never delete runs with registered versions (use lifecycle management), or implement a workflow that copies artifacts to a dedicated location during registration. Databricks Unity Catalog handles this by managing artifact copies automatically.

What is the actual overhead of MLflow auto-logging?

▼
Generally negligible (< 1% of training time). Three sources: monkey-patching at import (milliseconds, one-time), collecting params/metrics (few ms per epoch), and model serialization at end (varies by size). For sub-second training iterations, enable MLFLOW_ENABLE_ASYNC_LOGGING. System metrics polling adds a background thread if enabled. For extreme performance sensitivity, log manually at reduced frequency.

How should I handle model A/B testing with the registry?

▼
MLflow provides building blocks, not a complete A/B framework. Pattern: (1) Register candidate as new version, assign @challenger alias. (2) Configure serving infra to split traffic between @champion and @challenger. (3) Use MLflow Tracing to capture request-level metrics for both. (4) Analyze comparison. (5) If challenger wins, atomically update @champion alias. For GenAI, built-in evaluation judges can score both versions from production traces.

What are the practical scaling limits of the tracking server?

▼
The server is stateless and horizontally scalable. The bottleneck is the database. PostgreSQL comfortably handles ~10M runs, ~100M metrics, hundreds of concurrent users. Beyond that, query performance degrades. Mitigations: archive old experiments, add indexes on commonly-filtered columns, use read replicas, enable PgBouncer connection pooling. Artifact storage (S3/GCS) scales without limit.

How does MLflow Tracing differ from standard OpenTelemetry?

▼
MLflow Tracing is built on OpenTelemetry and is OTLP-compatible, but adds AI-specific capabilities: structured LLM input/output capture, experiment association, built-in LLM judge evaluation, purpose-built trace visualization UI, and human feedback attachment. You can export traces to any OTLP backend (Jaeger, Grafana Tempo). The dual-mode TracerProviderWrapper lets MLflow coexist with existing OTel instrumentation.

How do I migrate from file-based backend to a database?

▼
No built-in migration tool. Options: (1) Export runs via MlflowClient and replay against new backend (lossy). (2) Write a custom script to iterate all experiments/runs and recreate them. (3) Start fresh with a new database. Best practice: start with a database backend from the beginning. The file-based store is for quick local experimentation, not long-term storage.

Best practice for team experiment organization?

▼
Create one experiment per problem/task (e.g., "customer-churn"), not per person. Use run tags for identity: mlflow.set_tag("developer", "alice"). Use descriptive run names: "rf-baseline-v2". Enable workspaces (3.x) for team isolation. Anti-pattern: per-person experiments for the same task, which makes comparison impossible.
