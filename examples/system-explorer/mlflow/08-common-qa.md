### Q: How does MLflow handle concurrent writes to the same experiment from distributed training jobs?

MLflow's tracking server serializes writes through its REST API and relies on the underlying database's concurrency control. Each training worker creates its own run (unique UUID), so there is no contention at the run level. However, concurrent `log_metric` and `log_param` calls to the same run (e.g., from data-parallel training) can cause issues with SQLite (single-writer lock) and may result in intermittent errors. For distributed training, use PostgreSQL or MySQL as the backend store, which handle concurrent writes properly through row-level locking. With async logging enabled, the client buffers writes and flushes them in batches, reducing contention further. For frameworks like PyTorch DDP or Horovod, the recommended pattern is to log from the rank-0 process only.

### Q: What happens to registered model versions if I delete the original run that produced the model artifacts?

The model version becomes effectively broken. The model registry stores a `source` URI pointing to the original run's artifact location, but it does not copy the artifacts. If you delete the run (and its artifacts are garbage-collected), attempts to load the model version will fail with a "resource not found" error. This is a deliberate design choice -- copying multi-gigabyte model files on every registration would be expensive and slow. The mitigation is to either (a) never delete runs that have registered model versions (use lifecycle management instead of deletion), or (b) implement a workflow that copies model artifacts to a dedicated artifact location during registration. Databricks' Unity Catalog integration handles this differently by managing artifact copies automatically.

### Q: How do I migrate from MLflow's file-based backend (mlruns directory) to a database backend without losing experiment history?

There is no built-in migration tool for this transition. The recommended approaches are: (1) Use `mlflow experiments csv` to export run data and replay it against the new backend (lossy -- artifacts may not transfer cleanly). (2) Write a custom script using `MlflowClient` to iterate over all experiments and runs in the source, then recreate them in the destination using the REST API. (3) For fresh starts, simply point your tracking URI at the new database and start new experiments -- your old `mlruns` directory remains accessible locally. For production use, it is strongly recommended to start with a database backend from the beginning. The file-based store is intended for quick local experimentation, not long-term storage.

### Q: What is the actual overhead of MLflow auto-logging, and can it interfere with training performance?

Auto-logging overhead is generally negligible for most training workloads (< 1% of training time), but it can become noticeable in specific scenarios. The overhead comes from three sources: (1) monkey-patching framework methods at import time (one-time, milliseconds), (2) collecting and logging parameters and metrics during training (per-epoch, a few milliseconds per call), and (3) logging the model artifact at the end of training (one-time, varies by model size). The most expensive operation is model serialization and upload, which happens once. For training loops running at sub-second iteration speed, synchronous metric logging can add visible overhead; enable `MLFLOW_ENABLE_ASYNC_LOGGING` to move logging off the critical path. Auto-logging also captures system metrics (CPU, GPU, memory) if `MLFLOW_ENABLE_SYSTEM_METRICS_LOGGING` is set, which adds a background polling thread. For extremely performance-sensitive code, log metrics manually at reduced frequency rather than using auto-logging.

### Q: How does MLflow's model signature enforcement actually work, and what happens when the input does not match?

When a model is logged with a signature (or one is auto-inferred from the input example), the signature defines the expected input and output schemas using MLflow's type system (column-based for tabular data, tensor-based for array data). At prediction time, `PyFuncModel.predict()` calls `_enforce_schema()` which: (1) checks that all required columns are present, (2) attempts type coercion for compatible types (e.g., int to float), (3) drops extra columns that are not in the schema (with a warning), and (4) raises `MlflowException` for incompatible types or missing required columns. The enforcement is opt-in at the model level -- models without signatures skip validation entirely. Signature enforcement catches common deployment issues (wrong column names, mismatched types) early, before they cause cryptic errors deep in the model code. However, it does not validate data ranges, null handling, or semantic correctness. For GenAI models, signatures can specify chat message schemas and inference parameters.

### Q: How should I handle model A/B testing and gradual rollouts with the model registry?

MLflow's model registry provides the building blocks but not a complete A/B testing framework. The recommended pattern: (1) Register your candidate model as a new version and assign a `@challenger` alias. (2) Configure your serving infrastructure (KServe, Seldon, custom) to split traffic between `models:/MyModel@champion` and `models:/MyModel@challenger`. (3) Use MLflow Tracing to capture request-level metrics (latency, quality scores) for both versions. (4) Analyze the comparison in the MLflow UI or programmatically via the search API. (5) If the challenger wins, atomically update the `@champion` alias to point at the new version. This approach keeps the registry as the source of truth for which version is production, while delegating traffic management to the serving layer. For GenAI applications, MLflow's built-in evaluation judges can score both versions on quality metrics (relevance, safety, groundedness) directly from production traces.

### Q: What are the practical limits of the MLflow tracking server, and when do I need to shard or upgrade?

The tracking server itself is stateless and horizontally scalable -- run multiple instances behind a load balancer. The bottleneck is always the backend database. Practical limits with PostgreSQL: a single database comfortably handles up to ~10 million runs, ~100 million metrics, and hundreds of concurrent users. Beyond that, query performance degrades, particularly for `search_runs` with complex filter expressions across large metric and parameter tables. The `SqlLatestMetric` cache helps, but aggregation queries still scan large tables. Mitigation strategies: (1) archive old experiments to a separate database, (2) add database indexes on commonly-filtered columns, (3) use read replicas for the UI and search queries, (4) enable database connection pooling (PgBouncer). For artifact storage, S3/GCS/Azure Blob scales essentially without limit. MLflow 3.x workspaces provide logical partitioning that helps with organization but do not solve database scaling at the storage level.

### Q: How does MLflow Tracing relate to and differ from standard OpenTelemetry tracing?

MLflow Tracing is built on top of OpenTelemetry and is fully OTLP-compatible, but it adds several ML/AI-specific capabilities. Standard OpenTelemetry traces capture generic request spans (HTTP calls, database queries, function invocations) with arbitrary attributes. MLflow Tracing adds: (1) structured input/output capture for LLM calls (prompts, completions, token counts), (2) experiment association (traces are linked to experiments for organized analysis), (3) built-in LLM judge evaluation (automated quality scoring of traced requests), (4) a purpose-built UI for trace visualization with prompt/completion display, and (5) human feedback attachment for traces. You can export MLflow traces to any OTLP-compatible backend (Jaeger, Grafana Tempo, Datadog) for correlation with infrastructure telemetry. The `TracerProviderWrapper` dual-mode design means MLflow can operate alongside existing OpenTelemetry instrumentation without conflicts -- in unified mode, MLflow traces appear in your existing observability stack; in isolated mode, they stay in MLflow's own storage.

### Q: What is the best practice for tracking experiments across a team where multiple people work on the same problem?

Use a shared tracking server with a naming convention and access discipline. Recommended pattern: (1) Create one experiment per problem/task (e.g., "customer-churn-prediction"), not per person. (2) Use run tags to identify who ran what: `mlflow.set_tag("developer", "alice")`. (3) Use run names that encode the approach: `run_name="rf-baseline-v2"` or `run_name="xgb-tuned-lr0.01"`. (4) Enable MLflow workspaces (3.x) to isolate team projects while sharing the same server infrastructure. (5) Establish a convention for when runs are promoted to the model registry (e.g., only after peer review). (6) Use experiment-level permissions (available in Databricks) or proxy-level access control (Nginx auth) to prevent accidental cross-team interference. The key anti-pattern to avoid is creating per-person experiments for the same task, which makes cross-person comparison impossible without custom scripting.
