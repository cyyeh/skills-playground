Implementation Details

Rust Core

Qdrant is implemented entirely in Rust, leveraging zero-cost abstractions, memory safety, and predictable performance. The codebase is organized into key crates:

  • lib/segment: Core storage and indexing — vector storage, HNSW builder, payload storage, and field indexes.
  • lib/collection: Collection management, shard orchestration, optimizer pipeline, WAL integration.
  • lib/storage: Top-level TableOfContent, consensus integration, dispatcher logic.
  • src/: Main binary, API server setup (actix-web + tonic), configuration parsing.
💡
Why Rust matters: No garbage collector means no GC pauses in the middle of a search. Rust's ownership model eliminates data races in the concurrent optimizer pipeline. The async runtime (tokio) handles thousands of concurrent connections with minimal overhead. SIMD intrinsics are used for vectorized distance computations.

Memory-Mapped Storage

Qdrant's memmap storage uses the mmap system call to create a virtual address space backed by disk files. Rather than loading vectors into RAM directly, the OS page cache decides what stays in memory.

// Creating a collection with memmap storage PUT /collections/my_collection { "vectors": { "size": 1536, "distance": "Cosine", "on_disk": true }, "hnsw_config": { "on_disk": true } }

When a vector is accessed, the OS checks the page cache. Cache hit = RAM speed. Cache miss = disk read + cache. Frequently accessed vectors stay cached; rare ones are evicted. This provides graceful degradation: abundant RAM approaches in-memory performance; limited RAM falls back to disk without crashing.

HNSW Graph Construction

The HNSW index is a multi-layer proximity graph. Construction parameters control the quality/cost trade-off:

ParameterDefaultPurposeImpact
m16Max edges per nodeHigher = better recall, more memory
ef_construct100Build-time candidate listHigher = better graph, slower build
max_indexing_threadsautoParallel build threadsMore threads = faster build

Construction Algorithm

  1. Each new vector enters at the top layer and greedily navigates to the nearest neighbor.
  2. At its assigned insertion layer (random), it connects to its ef_construct nearest neighbors.
  3. The algorithm descends to layer 0, connecting at each layer.
  4. Edges are pruned to maintain max m per node, using heuristic selection that prefers directionally diverse connections.

Optimizations

Delta Encoding: HNSW graph links are compressed using delta encoding — storing differences between sorted neighbor IDs rather than absolute IDs. Minimal CPU overhead for decompression, significant memory savings.

Incremental Building: The incremental_hnsw_building feature flag allows updating the HNSW graph without full rebuilds for small batches of new points.

GPU Acceleration: Since v1.13, GPU-accelerated HNSW indexing is available for faster graph construction on supported hardware.

Scalar Quantization

Each float32 dimension is linearly mapped to int8. Qdrant computes quantile bounds (e.g., 0.99 quantile) to exclude outliers, then maps the float range to [-128, 127]. Distance calculations use SIMD-optimized integer arithmetic.

Binary Quantization

Each float32 dimension becomes a single bit: 1 if positive, 0 otherwise. Distance uses XOR + popcount — hardware-accelerated CPU instructions operating on 64 bits simultaneously. The 2-bit and 1.5-bit variants (v1.15+) explicitly handle near-zero values for better accuracy with lower-dimensional embeddings.

# Enabling scalar quantization on a collection PUT /collections/my_collection { "vectors": { "size": 1536, "distance": "Cosine" }, "quantization_config": { "scalar": { "type": "int8", "quantile": 0.99, "always_ram": true } } }

API Design

The API is organized into four services, each implemented in both REST and gRPC:

  • Collections Service: Lifecycle operations (create, update, delete, list). Defined in collections.proto.
  • Points Service: Vector/payload CRUD and search operations (upsert, delete, get, scroll, search, recommend). Defined in points.proto. Most frequently used.
  • Snapshots Service: Backup creation, listing, download, and recovery.
  • Cluster Service: Cluster status, peer management, shard operations.
ℹ️
Consistency guarantee: Both REST and gRPC are auto-generated from the same protobuf definitions, ensuring identical functionality. The REST API also exposes an OpenAPI spec for client generation in unsupported languages.

Storage Format Evolution

RocksDB era: Early versions used RocksDB for payload storage and structured metadata. Reliable but general-purpose.

GridStore (v1.13+): A custom block-based key-value storage engine optimized for Qdrant's specific access patterns — the mix of random reads (payload lookups) and sequential writes (batch upserts). The payload_index_skip_rocksdb feature flag controls migration.

Vector files: Vector data is stored separately in flat binary files (sequential float32 arrays) or memory-mapped files, independent from the key-value store. This separation enables independent optimization of vector access patterns.
