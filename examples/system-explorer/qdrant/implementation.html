Getting Started

Bash
# Pull and run Qdrant with persistent storage
docker pull qdrant/qdrant
docker run -p 6333:6333 -p 6334:6334 \
    -v "$(pwd)/qdrant_storage:/qdrant/storage:z" \
    qdrant/qdrant
Python
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct

client = QdrantClient(url="http://localhost:6333")

# Create collection
client.create_collection(
    collection_name="my_collection",
    vectors_config=VectorParams(size=768, distance=Distance.COSINE),
)

# Insert points with payloads
client.upsert(
    collection_name="my_collection",
    points=[PointStruct(id=1, vector=[0.05, 0.61, ...], payload={"category": "electronics"})],
)

# Search with filtering
results = client.query_points(
    collection_name="my_collection",
    query=[0.2, 0.1, 0.9, ...],
    limit=10,
).points

HNSW Graph Layers

The GraphLayers struct stores the multi-layered graph. The search method implements two-phase greedy traversal: descend through upper layers, then expand at the base layer.

Rustlib/segment/src/index/hnsw_index/graph_layers.rsView on GitHub ↗
pub struct GraphLayers {
    pub(super) m: usize,
    pub(super) m0: usize,
    pub(super) links: GraphLinks,
    pub(super) entry_points: EntryPoints,
    pub(super) visited_pool: VisitedPool,
}

impl GraphLayers {
    pub fn search(&self, top: usize, ef: usize,
        mut points_scorer: FilteredScorer,
        custom_entry_points: Option<&[PointOffsetType]>,
    ) -> Vec<ScoredPointOffset> {
        let entry = self.search_entry(&mut points_scorer, custom_entry_points);
        self.search_on_level(entry, top, ef, &mut points_scorer, 0)
    }

    fn get_m(&self, level: usize) -> usize {
        if level == 0 { self.m0 } else { self.m }
    }
}

Core Type Definitions

The foundational types: ScoredPoint (search results), VectorDataConfig (vector storage), Filter (boolean payload filtering).

pub struct ScoredPoint {
    pub id: PointIdType,
    pub version: SeqNumberType,
    pub score: ScoreType,
    pub payload: Option<Payload>,
    pub vector: Option<VectorStructInternal>,
}

pub struct Filter {
    pub should: Option<Vec<Condition>>,   // OR
    pub must: Option<Vec<Condition>>,     // AND
    pub must_not: Option<Vec<Condition>>, // NOT
}

pub enum Distance { Cosine, Euclid, Dot, Manhattan }
pub enum QuantizationConfig { Scalar(..), Product(..), Binary(..) }
✅
Design pattern: The Filter struct uses should/must/must_not semantics matching Elasticsearch's boolean query syntax. This makes Qdrant's filtering immediately familiar to engineers coming from document search backgrounds.
