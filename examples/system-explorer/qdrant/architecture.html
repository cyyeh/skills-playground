Architecture

High-Level Design

Qdrant follows a shared-nothing, peer-to-peer distributed architecture. There is no single master or dedicated coordinator node. The system is organized in concentric layers: the API layer accepts client requests, the collection layer manages logical data organization, the shard layer handles distribution across nodes, and the segment layer provides storage and indexing.

Design Decisions

Why Rust?

Memory safety without garbage collection pauses gives predictable latency at the 99th percentile. No GC stop-the-world pauses during search — critical for latency-sensitive workloads. Unsafe SIMD intrinsics for distance calculations are contained without risking memory corruption elsewhere.

Why Segments Over Monolithic Indexes?

Building one giant HNSW graph per collection makes updates extremely expensive (non-linear build time). Splitting data into segments with individual indexes amortizes the cost: new data enters mutable segments (fast writes, scan-based search) and migrates to immutable segments (HNSW-indexed, fast search) via background optimization.

Why Raft for Metadata But Not Points?

Point-level writes bypass Raft consensus and fan out directly to replicas. This gives high write throughput (no consensus round-trip per point) at the cost of eventual consistency. Structural operations (collection creation, shard moves) go through Raft for strong consistency. This split works because most vector workloads are write-heavy during ingestion and read-heavy during serving.

💡
Key design trade-off: Qdrant built a custom storage engine (Gridstore) instead of using RocksDB or LevelDB. Because every vector in a collection has identical byte size, storage locations can be calculated by simple arithmetic rather than B-tree lookups. This provides predictable low-latency access and efficient memory-mapped I/O, but means Qdrant maintains a larger codebase than systems that delegate to off-the-shelf storage.
