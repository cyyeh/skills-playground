Architecture

Seven-Layer Architecture

Qdrant is organized into seven distinct layers, from client-facing APIs down to disk persistence. Click any layer to see its details.

Qdrant Component Stack

1
Client SDKs & Frameworks
Python, JS, Rust, Go, Java, .NET
Official client libraries in six languages connect to Qdrant via REST or gRPC. Framework integrations (LangChain, LlamaIndex, Haystack) build on top of these clients to provide higher-level abstractions for RAG, semantic search, and agent memory.
▼
2
API Layer
REST (actix-web :6333) / gRPC (tonic :6334)
REST serves JSON via actix-web on port 6333. gRPC serves binary Protocol Buffers via tonic on port 6334. Both expose identical functionality. An internal gRPC channel on port 6335 handles peer-to-peer communication in distributed mode.
▼
3
Application Core
Dispatcher + ConsensusManager
The Dispatcher routes requests to the correct collection and shard. The ConsensusManager coordinates cluster-wide state via Raft. Metadata operations flow through consensus; data operations bypass it for performance.
▼
4
Collection Management
TableOfContent + Collection
TableOfContent is the top-level registry tracking all collections. Each Collection manages its sharding config, replication settings, and optimizer parameters. Collections are the primary organizational boundary.
▼
5
Shard Layer
LocalShard / RemoteShard / ReplicaSet
Collections are split into shards. LocalShard owns data with segments and WAL. RemoteShard proxies to peers via internal gRPC. ReplicaSets maintain replicas with states: Active, Dead, Partial, Initializing, Listener.
▼
6
Storage Layer
Segments (vectors + payloads + indexes)
Segments are self-contained stores with vector storage, payload storage, HNSW index, payload field indexes, and ID mapper. They transition from Plain (appendable, no HNSW) to Indexed (immutable, with HNSW) via the optimizer.
▼
7
Persistence
WAL + GridStore/RocksDB + mmap files
WAL ensures durability with sequential versioned writes. GridStore (or RocksDB) handles key-value payload storage. Vector data uses flat binary files or memory-mapped files for efficient random access.

Segment Lifecycle

Segments are the fundamental unit of storage. Understanding their lifecycle is key to understanding Qdrant's write and optimization behavior.

Plain (Appendable) Segments

▼

Every shard maintains at least one plain segment that accepts new writes. Plain segments use simple sequential vector storage without an HNSW index — incoming points are immediately searchable via brute-force but not yet indexed for fast ANN search. This design ensures writes are never blocked by index building.

Indexed (Non-Appendable) Segments

▼

When a plain segment reaches the indexing threshold (default: 10,000 points), the optimizer converts it to an indexed segment. The optimizer builds the HNSW graph, creates payload field indexes, and optionally converts to memmap storage. The segment is sealed — it becomes immutable for inserts but still supports deletes and reads.

Special (Temporary) Segments

▼

During optimization, proxy and copy-on-write segments handle concurrent reads and writes safely. A proxy segment intercepts reads to the old segment while a new indexed segment is being built. Once the new segment is ready, the proxy is replaced atomically.

Optimizer Pipeline

Three asynchronous workers coordinate to keep data fresh, indexed, and durable without blocking client requests.

💡
Key insight: Writes are acknowledged after WAL persistence, not after indexing. This means upserts are fast (WAL append), while expensive HNSW construction happens asynchronously in the background.
  • Update Worker: Applies point operations (upsert, delete, set payload) to segments immediately after WAL write. Triggered on every write.
  • Optimize Worker: Builds HNSW indexes when segments reach thresholds, merges small segments, vacuums deleted points. Converts plain segments to indexed segments.
  • Flush Worker: Persists segment state to disk every 5 seconds (configurable via flush_interval_sec). Ensures non-WAL data like index structures are durably stored.

Distributed Architecture

ℹ️
Tip: Use an odd number of nodes (3, 5, 7) for reliable Raft leader election. Set replication_factor to at least 2 for fault tolerance in production.

Sharding

Collections are split into shards distributed across peers. Automatic sharding (default) uses consistent hashing for even distribution. Custom sharding (v1.7.0+) allows explicit shard assignment per point, useful for multi-tenant architectures where each tenant gets its own shard.

Replication

Each shard can have multiple replicas across different peers, configured via replication_factor. Write operations are forwarded to all replicas with configurable write consistency — how many replicas must acknowledge before the client receives a response.

Consensus

Raft consensus ensures all peers agree on cluster topology and collection structure. A critical design choice: metadata operations (collection management, shard transfers) go through Raft for consistency, while data operations (upserts, searches) bypass Raft for performance, using clock tags for causal ordering instead.

⚠️
Important: Because data operations bypass Raft, Qdrant prioritizes availability and performance over strict consistency for point-level operations. If you need strong consistency for reads, use the read_consistency parameter with a value of all or majority.
