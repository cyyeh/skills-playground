Architecture

High-Level Design

Qdrant is a multi-threaded, network-accessible vector database written in Rust. In single-node mode it runs as a standalone process; in distributed mode it forms a symmetric peer-to-peer cluster where every node runs the full stack. The architecture is organized in four layers.

Qdrant Architecture Layers

🌐 API Layer

REST (port 6333) and gRPC (port 6334) interfaces accepting client requests. Internal gRPC (port 6335) for cluster coordination.

↓
📋 Table of Contents (TOC)

Central storage orchestrator that manages all collections, routes operations to shards, and coordinates distributed operations.

↓
📦 Shard Layer

Horizontal partitioning via shards managed by ReplicaSets. Auto sharding (hash ring) or custom sharding (shard keys) with configurable replication.

↓
💾 Segment Layer

Core storage units containing vector data, payload data, HNSW indexes, and payload indexes. Segments transition from appendable (fast writes) to indexed (fast search).

Asynchronous Workers

Each local shard runs three asynchronous worker pools that separate concerns for optimal performance:

⚙️

Update Worker — Applies point operations (upsert, delete, set_payload) from the WAL to appendable segments. Operations are ACKed immediately after WAL persistence.

🔧

Optimize Worker — Builds HNSW indexes on segments that exceed the indexing threshold, merges small segments into larger ones, and vacuums deleted points. Runs continuously without blocking writes.

💾

Flush Worker — Persists in-memory segment state to disk at configurable intervals (default: 5 seconds). Ensures durability without blocking the hot write path.

Storage Engine: Gridstore

As of v1.17, Qdrant has fully replaced RocksDB with Gridstore, a custom-built key-value store designed specifically for vector database workloads. Gridstore provides lower tail latencies by reducing lock contention, better control over storage layout and I/O patterns, tighter integration with the segment lifecycle, and eliminates RocksDB compaction stalls.

⚠️

Migration note: Direct upgrades from v1.15.x (RocksDB) to v1.17.x (Gridstore-only) are not supported. You must upgrade through v1.16.x first, which handles the RocksDB-to-Gridstore data migration automatically.

Distributed Mode

When cluster.enabled = true, Qdrant forms a symmetric peer-to-peer cluster. The critical design insight is the split between metadata and data operations:

💡

Raft Consensus handles metadata operations: creating/deleting collections, shard transfers, configuration changes. These are rare but need strong consistency.

⚡

Direct Peer-to-Peer gRPC handles data operations: upsert, delete, search. These bypass Raft entirely for low latency, using clock tags for causal ordering.

Clients can choose consistency per operation: 1 (fast, eventual), majority, all (strict), or quorum. This flexibility lets you tune the consistency-latency tradeoff for each use case.

Design Decisions

💡

Rust over C++/Go/Java. Memory safety without GC pauses is critical for predictable search latency. Rust's ownership model prevents data races at compile time, eliminating an entire class of production bugs in a concurrent system.

💡

Symmetric Cluster. Every node runs the full stack — no dedicated coordinator nodes. This simplifies operations: any node can accept any request, and there is no single point of failure beyond Raft leader election.

💡

WAL-First Writes. Durability is guaranteed before acknowledgment. The expensive HNSW index building happens asynchronously, so write throughput is not bottlenecked by index construction.

💡

Segment-Based Storage. Enables zero-copy optimization: new indexes are built on copies while the original segment serves reads. The proxy segment pattern allows atomic segment swaps without read interruption.

💡

Custom Storage (Gridstore). Purpose-built for vector workloads rather than relying on general-purpose key-value stores. Eliminates the impedance mismatch between vector access patterns and generic KV store assumptions.
