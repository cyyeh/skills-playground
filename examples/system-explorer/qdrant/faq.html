Common Q&A

Hard, practical questions a senior engineer would ask when evaluating or operating Qdrant. Click any question to expand the answer.

▼What happens to in-flight writes if a node crashes during replication?
Durability depends on write_consistency_factor. Default (1): write acknowledged after one replica persists to WAL — if that node crashes, data may be lost. With majority, multiple replicas must persist before acknowledgment. On recovery, WAL replays uncommitted ops. If a replica falls behind, wal_delta transfer catches up with minimal data, falling back to stream_records for full resync.
▼How do I choose the right quantization method?
Binary (32x compression, 40x speedup): best for 1024+ dim embeddings (OpenAI, Cohere). Scalar (4x compression, 2x speedup): safe default for 384-768 dim. Product (4-64x): most aggressive, tolerate 5-10% recall loss. Always benchmark with your data: create quantized and unquantized collections, compare recall@k.
▼How does Qdrant handle stale reads in a distributed cluster?
By default, reads go to local replica only (lowest latency, potentially stale). Configure read_consistency per request: majority reads from >50% of replicas, quorum requires agreement, all reads every replica. Use selectively for freshness-critical operations rather than blanket-applying to all reads.
▼What are the practical scaling limits?
Memory: ~1B 768-dim vectors per 256GB node with scalar quantization (quantized in RAM, originals on disk). Shards: 12+ recommended, max practical ~hundreds per collection. Cluster: tested with dozens of nodes. Payload indexes: high-cardinality fields increase memory. For billion-scale, use multi-tenant sharding with shard_key.
▼How should I tune HNSW parameters?
Start with defaults (m=16, ef_construct=100). If recall is low: increase ef_construct first (200, then 400). If still insufficient, increase m (32, then 48). At search time, set ef to at least 2x top_k. Lower full_scan_threshold if filtered queries match small subsets. Always measure recall@k against brute-force ground truth.
▼Separate collections vs. shard keys for multi-tenancy?
Separate collections: strongest isolation, but unmanageable beyond ~100 tenants. Payload filtering: simplest, but every query traverses entire HNSW. Custom shard keys (recommended): assign shard_key per tenant. Queries specify shard key and only search that tenant's shard. For heterogeneous tenants, v1.16's tiered multitenancy promotes large tenants to own shards.
▼How do I diagnose high search latency?
(1) Check segment count — high count means optimizer is behind. (2) Check optimizer status. (3) Enable scalar quantization with always_ram: true. (4) Ensure filtered fields have payload indexes. (5) Lower ef if >512. (6) Limit optimizers_threads to reduce CPU contention. (7) Monitor via Prometheus qdrant_search_latency_seconds.
▼How does Qdrant compare to pgvector?
pgvector: zero extra infrastructure, full SQL, transactional consistency. Works well for <10M vectors where vector search is secondary. Trade-offs at scale: shares resources with transactional workload, no built-in sharding, no quantization. Choose Qdrant when vector search is a primary workload with dedicated performance/scaling needs.
