Common Q&A

Senior-level questions about Qdrant — the kind you'd ask when evaluating it for production.

01 How much memory does Qdrant need for N vectors?

Rule of thumb: memory = num_vectors x dimensions x 4 bytes x 1.5

The 4 accounts for 32-bit floats; the 1.5x covers HNSW graph overhead, metadata, and temporary segments during optimization. Example: 10M vectors at 768 dims = ~46 GB RAM.

💡

To reduce memory: Use mmap storage (on_disk=True), enable scalar quantization (4x savings), or binary quantization (32x savings). With mmap + scalar quantization, 10M vectors can be served with ~3 GB RAM.

02 How do I scale Qdrant for growing data?

Vertical: Add RAM, use faster NVMe SSDs for mmap, increase CPU cores. Horizontal: Start with more shards than nodes (e.g., 12 shards on 3 nodes), add nodes and Qdrant rebalances automatically.

Key tip: Start with shard_number=12. This allows scaling from 1 to 2, 3, 4, 6, or 12 nodes without resharding. Resharding an existing collection requires creating a new collection and re-ingesting data, since shard count is immutable after creation.

03 What consistency guarantees does Qdrant provide?

Single-node: Strong consistency — reads always see the latest writes after WAL flush.

Distributed: Configurable per operation. Write to 1 replica (fastest, eventual), majority, all (strongest), or quorum. Reads similarly configurable.

Important: Qdrant does NOT support multi-document ACID transactions. Each point operation is atomic, but there is no way to atomically update multiple points across collections.

04 How do I handle multi-tenancy?

Three approaches, from simplest to most scalable:

1. Payload filtering: Store tenant_id in payloads and filter. Simplest, but all tenants share the HNSW graph.

2. Collection-per-tenant: Clean isolation, but operational overhead with many collections.

3. Custom sharding (recommended): Use shard keys with tenant IDs. Each tenant's data goes to a dedicated shard. Searches route only to the relevant shard, combining isolation with efficient search. Qdrant v1.16 introduced tiered multitenancy to further optimize this pattern.

05 What happens when a node goes down?

With replication_factor >= 2: reads continue from surviving replicas with no user-visible impact. When the node recovers, Qdrant synchronizes automatically via WAL-based catch-up. If permanently lost, a new replica can be created on a different node.

06 How do I tune search accuracy vs. latency?

Three levers: 1) HNSW ef (query-time) — higher explores more candidates, improving recall but increasing latency. Start at 128. 2) HNSW m (build-time) — more edges per node improves connectivity. Range: 16-64. 3) Quantization + oversampling — retrieve extra candidates with quantized vectors, then rescore with full precision.

07 What operational gotchas should I watch for?

Optimization backlog: After bulk uploads, HNSW building can take significant time and memory. Don't benchmark during active optimization.

Memory spikes: Segment merging temporarily doubles memory. Ensure headroom.

Shard count is immutable: Plan ahead. You cannot change shard_number after creation.

gRPC compatibility: v1.17 changed gRPC response format for vectors. Update client libraries before upgrading.

Payload index memory: Each indexed field consumes memory. Only index fields used in filters.
