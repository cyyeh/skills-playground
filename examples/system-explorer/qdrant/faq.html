Common Q&A

How does Qdrant compare to pgvector?

▼

pgvector adds vector search to PostgreSQL, convenient if you already use Postgres and have modest needs (up to ~100M vectors). Qdrant is purpose-built with a dedicated HNSW implementation, rich payload filtering, multiple quantization options, and native distributed mode.

Choose pgvector when vector search is secondary and you want everything in one database. Choose Qdrant when vector search is core to your application and you need production-grade performance, scaling, or advanced features.

How much memory does Qdrant need for my dataset?

▼

Formula: vectors = num_points x dimensions x 4 bytes (float32) plus HNSW graph overhead (~num_points x m x 2 x 8 bytes).

Example: 1M vectors at 1536 dimensions with m=16: vectors = ~5.7 GB, HNSW = ~256 MB, total ~6 GB. With scalar quantization: ~1.7 GB. With memmap storage, the OS page cache manages memory — you can serve datasets much larger than available RAM.

Should I use cosine, dot product, or Euclidean distance?

▼

Use the metric that matches your embedding model's training. Most text embedding models (OpenAI, Cohere, sentence-transformers) work best with cosine similarity. If your vectors are already normalized (unit length), dot product is equivalent to cosine but slightly faster. Euclidean is for when absolute magnitude differences matter. When in doubt, start with cosine.

When should I use quantization, and which type?

▼

Scalar (int8): Safest default. 4x memory reduction, less than 1% accuracy loss. Works with any model.

Binary (1-bit): 32x compression, up to 40x speedup. Works best with high-dimensional embeddings (>1024d) from models like OpenAI. Test with your specific model first.

Product: Highest compression (up to 64x) but significant accuracy loss (~0.7 recall). Use only when memory is extremely constrained.

How do I handle multi-tenancy?

▼

Payload-based filtering: Store a tenant_id field and include it as a filter in every query. Simple, works well up to moderate tenant counts. Create a keyword index on the tenant field.

Custom sharding (v1.7.0+): Assign each tenant a shard key for physical data isolation. Better performance for large tenant counts since Qdrant searches only the relevant shard.

Can Qdrant handle real-time updates?

▼

Yes. New points are immediately searchable via brute-force in the appendable segment. Once the optimizer builds the HNSW index (triggered at the indexing threshold), those points become available for fast indexed search. Points are never invisible — they are always searchable through one path or the other.

How does Qdrant ensure data durability?

▼

All writes go to the Write-Ahead Log (WAL) before acknowledgment. The WAL is a sequential, append-only file that survives crashes. On recovery, Qdrant replays unprocessed WAL entries. The flush worker persists segment state every 5 seconds. In distributed mode, replication provides additional durability.

What embedding model should I use?

▼

General English text: OpenAI text-embedding-3-small (1536d) or sentence-transformers all-MiniLM-L6-v2 (384d, free, local).

Multilingual: Cohere embed-multilingual-v3.0.

Maximum quality: OpenAI text-embedding-3-large (3072d).

On-device (no API): Qdrant's FastEmbed library included in the Python client.

All vectors in a collection must use the same model and dimensionality.

Does Qdrant support hybrid search?

▼

Yes. Qdrant supports named vectors, allowing multiple vector types per point — for example, a dense embedding from a neural model and a sparse BM25 vector. You can search using either vector independently or combine results using fusion. Hybrid dense+sparse search often outperforms either approach alone.

What is the maximum number of vectors Qdrant can handle?

▼

A single node can handle tens of millions of vectors depending on dimensionality and memory. With distributed mode, Qdrant scales horizontally to billions of vectors by sharding across nodes. The practical limit depends on hardware, dimensionality, quantization, and latency requirements. Qdrant Cloud has been tested with billion-scale deployments.
