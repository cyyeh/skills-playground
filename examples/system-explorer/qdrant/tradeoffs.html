Trade-offs & Limitations

Strengths

  • Best-in-class filtered search — Filterable HNSW integrates filtering into graph traversal, producing both fast and accurate results for filtered queries
  • Rust performance, no GC pauses — Predictable low-latency performance critical for p99-sensitive production workloads
  • Rich vector type support — Dense, sparse, multi-dense, and named vectors in a single collection for native hybrid search
  • Flexible quantization — Scalar (4x), product (up to 64x), binary (32x), and asymmetric quantization for optimal speed-accuracy tradeoff
  • Production distributed mode — Raft consensus, horizontal sharding, configurable replication, per-operation consistency levels
  • Clean API design — Unified Query API for search, recommendation, and hybrid search with six official SDKs
  • Active development — Rapid release cadence with major features: Gridstore, Qdrant Edge, tiered multitenancy, ACORN algorithm

Limitations

  • Smaller community — ~9K GitHub stars vs. Milvus's ~25K. Fewer community tutorials and Stack Overflow answers
  • No built-in embedding generation — Unlike Weaviate's vectorizer modules, you must handle embedding creation separately
  • Immutable shard count — Cannot change shard_number after collection creation. Plan ahead or re-ingest
  • Memory-intensive at full precision — 100M vectors at 768 dims = ~460 GB RAM without quantization or mmap
  • No multi-document transactions — Each point operation is atomic; no cross-collection atomic operations
  • Limited analytics — No aggregations, GROUP BY, or SQL-style queries over payloads
  • Younger sparse vector ecosystem — Sparse/hybrid search support introduced in v1.7; less battle-tested than dedicated full-text engines

Decision Framework

Choose Qdrant when:

  • You need fast filtered vector search in production
  • You want open-source with self-hosting flexibility
  • You need hybrid search (dense + sparse) natively
  • You value predictable latency (Rust, no GC)
  • You need multi-tenancy with custom sharding

Consider alternatives when:

  • You want fully managed with zero ops → Pinecone
  • You need built-in vectorizer modules → Weaviate
  • You operate at true billion-scale with diverse indexes → Milvus
  • You have a PostgreSQL stack with <100M vectors → pgvector
  • You are just prototyping → Chroma
