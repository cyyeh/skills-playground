Trade-offs & Limitations

Strengths

Predictable Rust-native performance. No GC pauses. Sub-20ms p50 at billion-scale with tight p99 variance.

Best-in-class filtered vector search. Filterable HNSW integrates payload conditions into graph traversal with adaptive query planning. Most competitors pre-filter or post-filter; Qdrant avoids both pathological cases.

Full quantization spectrum. Scalar, product, and binary quantization with asymmetric combinations and configurable oversampling. Fine-grained memory/accuracy/speed control.

Flexible deployment model. Single Docker container to multi-node Kubernetes cluster to edge devices. Apache 2.0 license, no vendor lock-in.

Limitations

Eventual consistency for points. Writes bypass Raft. A write-then-immediate-read may miss the just-written point unless read_consistency: "all" is used.

HNSW rebuild cost on bulk updates. Replacing >10% of vectors triggers expensive optimization cycles. Freshly inserted points have higher search latency until optimization completes.

No cross-collection operations. No joins, no cross-collection queries, no multi-collection transactions. Correlation across vector spaces must happen in application code.

Memory pressure without quantization. 1B 768-dim float32 vectors need ~2.9TB RAM uncompressed. Quantization is effectively required at scale.

Strengths

Weaknesses
  • No horizontal scaling
  • No quantization
Best For

PostgreSQL-centric apps where vector search is a secondary feature (<10M vectors).

The Honest Take

💡
Bottom line: Qdrant is the right choice for teams that need a self-hosted, production-grade vector database with strong filtered search, flexible quantization, and Rust's performance guarantees. It hits the sweet spot between Pinecone's zero-ops simplicity and Milvus's enterprise-scale complexity. Choose Pinecone if you want zero operational overhead, Milvus if you need proven billion-scale distributed deployments, or pgvector if vector search is a minor feature alongside SQL workloads.
