Core Concepts

The Building Blocks of Vector Search

Qdrant is built on seven core concepts. Understanding these gives you the vocabulary to work with any vector search system, and the mental model to reason about performance, scaling, and trade-offs.

📁

Collections

Named containers for vectors and metadata. Each defines dimensions, distance metric, and storage config.

Think of a collection as a dedicated photo album. Every photo has the same dimensions and is organized by the same sorting rule. You would not mix passport photos with panoramic landscapes in the same album.

📌

Points

The fundamental record: a unique ID + one or more vectors + an optional payload.

A point is like a pin on a map. The pin's location (vector) tells you where it sits in space, the label (ID) identifies it, and the sticky note (payload) carries extra info like "coffee shop, open until 10pm."

🧬

Vectors

Numerical representations of data produced by embedding models. Dense, sparse, or multi-vector.

A vector is like a DNA fingerprint for your data. Just as DNA encodes biological traits into a sequence of nucleotides, an embedding model encodes meaning into a sequence of numbers. Two similar items have close "DNA."

🏷️

Payloads

Structured JSON metadata attached to points. Indexed for filtering alongside vector search.

If a vector is the DNA fingerprint, the payload is the medical chart. The fingerprint identifies who someone is; the chart carries structured details. Search uses both — similarity of fingerprint AND criteria from the chart.

🕸️

HNSW Index

Multi-layered graph for fast approximate nearest neighbor search in logarithmic time.

HNSW works like navigating a city with progressively detailed maps. Start with a country map (sparse top layer) to find the region, switch to a city map to find the neighborhood, then a street map for the exact address.

📦

Segments

Internal storage units. Mutable segments accept writes; immutable segments have HNSW indexes.

Segments are like chapters in a book being written. New content goes into the draft chapter (mutable). Once it is long enough, it gets published (immutable with a full index), making it fast to search.

🗜️

Quantization

Vector compression (scalar, product, binary) trading precision for memory and speed.

Quantization is like creating thumbnail versions of high-resolution photos. Thumbnails take far less space and can be browsed quickly; when you need details, pull up the full-resolution original.

How They Fit Together

Data Flow Through Qdrant

1
Create a Collection
Define vector dimensions, distance metric (cosine, Euclidean, dot), and storage configuration.
▼
2
Insert Points
Each point carries one or more Vectors (embeddings) and an optional Payload (JSON metadata).
▼
3
Land in Mutable Segment
Points are immediately searchable via scan. The WAL ensures durability.
▼
4
Background Optimization
The optimizer converts mutable segments to immutable with a full HNSW Index. Quantization compresses vectors in RAM.
▼
5
Search Query
The query planner fans out across segments, uses HNSW for fast traversal, applies payload filters, and merges results into a ranked list.
💡
Key insight: Qdrant separates the write path (fast, mutable segments) from the search-optimized path (HNSW-indexed immutable segments). This means freshly inserted data is searchable immediately, but reaches peak search performance after the background optimizer runs. In a distributed setup, this same process happens across shards on multiple nodes.
