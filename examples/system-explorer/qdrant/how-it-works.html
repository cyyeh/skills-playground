How It Works

Indexing: From Upsert to Searchable

When you upsert points into Qdrant, the data goes through a carefully orchestrated pipeline designed to balance write speed with search performance.

Write Path

1
WAL Persistence
The operation is written to the Write-Ahead Log with a sequential version number. The client is not acknowledged until WAL write succeeds, guaranteeing durability even on immediate crash.
durability guarantee
▼
2
Segment Write
The update worker applies the operation to the current appendable (plain) segment. The vector, payload, and ID mapping are stored. The point is immediately searchable via brute-force.
instant availability
▼
3
Optimizer Trigger
When the segment reaches the indexing threshold (default: 10,000 points), a new appendable segment is created for incoming writes. The full segment is converted to an indexed segment with HNSW graph and payload indexes.
async background
▼
4
Periodic Flush
The flush worker persists segment state to disk every 5 seconds, ensuring index structures and in-memory state are durably stored beyond the WAL.
every 5 seconds

HNSW Search: Navigating the Graph

When a search query arrives, Qdrant navigates the multi-layered HNSW graph to find approximate nearest neighbors efficiently.

Search Path

1
Entry Point
Search begins at the top layer of the HNSW graph, which contains very few nodes with long-range connections. An entry point node is selected.
▼
2
Greedy Descent
At each layer, the algorithm greedily moves to the neighbor closest to the query vector. This is like navigating highway-level connections to quickly reach the right "neighborhood."
▼
3
Layer Transition
When no closer neighbor is found at the current layer, the search drops to the next layer with more nodes and shorter-range connections. The search becomes more fine-grained.
▼
4
Bottom Layer Beam Search
At layer 0 (all points), a priority queue of ef best candidates is maintained. Neighbors of candidates are explored until no improvement is found. Higher ef = better recall, higher latency.
▼
5
Result Ranking
The top-k candidates are returned, ranked by distance to the query vector according to the collection's metric (cosine, dot product, Euclidean, or Manhattan).
💡
Performance tip: The ef parameter controls accuracy vs. speed at search time. Typical production values range from 64 to 256. For indexing, ef_construct (default: 100) controls graph quality at build time — higher values yield better graphs but slower indexing.

Filtered Search

Qdrant integrates filtering into the HNSW traversal rather than applying it as post-processing. The strategy adapts based on filter selectivity.

ℹ️
Key design choice: Qdrant automatically selects the optimal strategy based on the estimated number of matching points. Selective filters (few matches) use filtered brute-force scan. Broad filters (many matches) use HNSW traversal with filter checks at each hop. This adaptive approach avoids the worst-case performance of either strategy alone.

Payload Index Types for efficient filtering:

  • Keyword index: Exact string matching and in conditions
  • Integer/Float index: Numeric range queries
  • Geo index: Geographic radius and bounding box
  • Full-text index: Tokenized inverted index for text search
  • Datetime index: Temporal range queries
  • UUID index: UUID field matching

Snapshot & Recovery

Qdrant provides multiple mechanisms for backup and disaster recovery.

Normal startup: Segments are loaded from their last flushed state on disk.

After crash: Qdrant loads the last flushed segment state, then replays WAL entries. Version tracking on each point ensures idempotent replay — operations with version numbers lower than the current point version are safely skipped.

Snapshot recovery: A collection can be fully restored from a snapshot archive file. In distributed mode, the Collection Snapshot Recovery API restores missing shards on new nodes.

Test Your Understanding

Why does Qdrant write to the WAL before writing to segments?

The WAL (Write-Ahead Log) ensures durability. If Qdrant crashes after writing to segments but before flushing to disk, the data could be lost. By writing to the WAL first, the operation is persisted to a sequential, append-only file that survives crashes. On recovery, Qdrant replays WAL entries to reconstruct any in-memory state that was not yet flushed. The client is only acknowledged after the WAL write succeeds.

What happens when a filter matches only a tiny fraction of points?

Qdrant bypasses HNSW entirely and performs a filtered brute-force scan over the matching points. This is more efficient because traversing the HNSW graph would waste time visiting nodes that fail the filter. When the matching set is small, brute-force comparison of just those points is faster than navigating the full graph.

Are newly upserted points immediately searchable? How?

Yes. New points are written to the appendable (plain) segment, which uses brute-force search (no HNSW index). They are searchable immediately, though with slightly higher latency than HNSW-indexed points. Once the optimizer builds the HNSW index for the segment, those points become available for fast indexed search. At no point is a point invisible to queries.
