How It Works

The HNSW Search Algorithm

HNSW (Hierarchical Navigable Small World) is the heart of Qdrant's search engine. It organizes vectors into a multi-layered navigable graph that enables approximate nearest neighbor search in O(log N) time rather than O(N) brute-force scanning.

How HNSW Search Works

1
Enter at the Top Layer

Start at the graph's entry point on the highest layer. This layer has very few nodes with long-range connections — the "express lanes" of the graph.

O(1) — single entry point
↓
2
Greedy Descent Through Layers

At each layer, greedily navigate to the node closest to the query vector by checking neighbors. Drop to the next layer when no closer neighbor exists. Higher layers have fewer nodes, making this extremely fast.

O(log N) — exponentially fewer nodes per layer
↓
3
Beam Search at Layer 0

On the bottom layer (all nodes), perform a beam search with ef candidates. This explores the local neighborhood thoroughly, maintaining a priority queue of the closest nodes found so far.

ef parameter controls accuracy vs. latency
↓
4
Return Top-k Results

Sort the candidate list by distance and return the requested number of nearest neighbors along with their scores and payloads.

Typical: 1-5ms for millions of vectors

Filterable HNSW: The Key Innovation

Standard HNSW has a fundamental problem with filters. If you search the graph first and then filter results, you may discard most matches and return poor-quality results. If you filter the dataset first and then search, the HNSW graph may become disconnected, causing the search to miss entire regions of the vector space.

Qdrant's filterable HNSW solves this with a three-pronged approach:

🔗

Payload-Aware Edges. During index construction, Qdrant builds additional edges between nodes that share common payload values. For each indexed payload field value, a subgraph of matching nodes is constructed, and extra edges ensure these subgraphs remain navigable even when most of the graph is filtered out.

🔍

Filter-During-Search. During HNSW traversal, each candidate is checked against the filter condition. Non-matching candidates are skipped for the result set but still used for navigation — they help traverse the graph toward the right region. Only matching candidates enter the final result set.

⚙️

Adaptive Strategy. Qdrant automatically chooses the best search strategy based on estimated filter selectivity: broad filters use the standard HNSW graph with filter checks, medium selectivity uses filterable HNSW extra edges, and narrow filters fall back to a filtered scan of the payload index with candidate rescoring.

The Indexing Pipeline

1
WAL Persistence

The upsert operation is written to the write-ahead log. The client receives an acknowledgment immediately — no waiting for index building.

↓
2
Appendable Segment

The point is added to a writable segment with no HNSW index (plain storage). This ensures fast write ingestion — new points are immediately searchable via brute-force scan of the small segment.

↓
3
Threshold Detection

The optimize worker monitors segment sizes. When an appendable segment exceeds the configured indexing_threshold, optimization is triggered.

↓
4
Segment Optimization

A new indexed segment is built: HNSW index is constructed with the configured m and ef_construct, payload indexes are created, and the vector storage format is optimized (mmap or in-memory).

↓
5
Atomic Segment Swap

The new indexed segment atomically replaces the old appendable segment. A proxy segment maintains read availability during the transition. The WAL segment is freed once all its operations are confirmed applied.

Hybrid Search Pipeline

Qdrant's Query API enables combining multiple search strategies in a single server-side request, eliminating multiple client-server round-trips:

Hybrid Search: Dense + Sparse + Re-ranking

1
Dense Vector Search

Semantic similarity search using dense embeddings from models like OpenAI or sentence-transformers. Finds conceptually related content.

+
2
Sparse Vector Search

Lexical matching using sparse vectors from BM25 or SPLADE. Captures exact keyword matches that dense search might miss.

↓
3
Score Fusion

Results from dense and sparse searches are combined using Reciprocal Rank Fusion (RRF) or Distribution-Based Score Fusion (DBSF).

↓
4
Optional Re-ranking

Top candidates can be re-scored using multivector (ColBERT) late interaction or a separate re-ranking model for maximum relevance.

Quantized Search

When quantization is enabled, the search pipeline adds a two-phase optimization:

⚡

Coarse phase: HNSW traversal uses quantized vectors (e.g., INT8 or binary) for fast, approximate distance calculations. This quickly narrows candidates from millions to a few hundred.

Rescoring phase: Top candidates are rescored using original full-precision vectors for accurate final ranking. The oversampling parameter (e.g., 2.0) controls how many extra candidates are retrieved in the coarse phase to compensate for quantization error.

This approach provides most of the memory savings of quantization with minimal accuracy loss — typically 95-99% of full-precision recall.
