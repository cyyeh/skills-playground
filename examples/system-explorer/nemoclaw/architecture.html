Architecture

High-Level Design

NemoClaw's architecture is a two-layer system with a clear separation of concerns: a thin TypeScript plugin handles the user-facing CLI and orchestration logic, while a Python blueprint runtime handles the actual provisioning and management of sandboxed environments.

The overall design follows a "wrapping" philosophy rather than a "forking" one. Instead of modifying OpenClaw's source code to add security, NemoClaw places an unmodified OpenClaw installation inside a controlled container and governs it from the outside -- like surrounding an existing building with a security perimeter rather than renovating its interior.

🔭
Architecture insight: The wrapping-over-forking approach trades some depth of integration for maintainability. NemoClaw can absorb upstream OpenClaw updates without merge conflicts, and the security boundary is enforced externally rather than relying on in-application checks that could be bypassed.

Key Design Decisions

🔭
Deny-by-default networking: Every outbound connection is blocked unless explicitly allowed. Autonomous agents executing arbitrary code present a fundamentally different threat model than human developers -- an agent that discovers and starts calling an API is exactly what NemoClaw prevents without human approval.
💡
Host-side credential management: API keys never enter the sandbox. The agent sees only inference.local as its model endpoint. Even a fully compromised sandbox cannot exfiltrate API keys because they do not exist in its environment.
🔭
Two-language split: The CLI and plugin are TypeScript (integrating with Node.js-based OpenClaw), while the blueprint runtime is Python (interacting with OpenShell's Python tooling). This avoids forcing either ecosystem to use a non-native language, at the cost of some cross-language complexity.
