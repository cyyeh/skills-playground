Architecture

High-Level Design

NemoClaw uses a plugin-blueprint architecture with two distinct layers. The Plugin Layer (TypeScript) provides the CLI interface and user interaction, while the Blueprint Layer (Python) contains the orchestration logic and security configuration.

Beneath both layers sits OpenShell, the actual runtime that does the heavy lifting of container isolation, policy enforcement, and inference routing.

Key Design Decisions

The architecture reflects four deliberate choices that prioritize security over convenience. Each decision has a clear rationale rooted in the threat model of autonomous agents.

Out-of-Process Enforcement

Security constraints are enforced at the OS/network level, not by asking the model to behave. A fully compromised agent is still contained by the sandbox. This is the single most important architectural decision.

Deny-by-Default Networking

All outbound connections are blocked unless explicitly allowed in the policy YAML. Safer to start locked down and selectively open than to start open and try to block threats.

Credential Isolation

API keys are stored on the host at ~/.nemoclaw/credentials.json (permissions 0600) and injected by the gateway at request time. The agent never sees raw credentials.

Two-User Container Model

The sandbox runs two distinct Linux users (gateway user and sandbox user). Prevents the agent from escalating privileges to modify the gateway process, even within its own container.

🔭
Architecture insight: The static + dynamic policy split balances security with flexibility. Static policies (filesystem, process) are locked at container creation and cannot be weakened. Dynamic policies (network, inference) can be hot-reloaded for operational flexibility.

Data Flow: External API Call

Here is exactly what happens when an agent inside the sandbox tries to call an external API. Every outbound request follows this path without exception.

  1. Agent initiates request -- The claw makes an HTTP request (e.g., to api.github.com)
  2. DNS interception -- Model calls hit inference.local; other requests hit the sandbox network stack
  3. Gateway intercept -- OpenShell's gateway intercepts the outbound connection
  4. Policy check -- Gateway checks destination host, port, requesting binary, and HTTP method against the YAML policy
  5. Decision -- Allowed (credentials injected, request forwarded), Blocked (logged, operator prompted), or Inference route (Privacy Router classifies and routes)
  6. Response -- Flows back through the gateway to the agent
⚠️
Warning: Credentials are injected by the gateway at request time and never stored inside the sandbox. Even a fully compromised sandbox cannot steal provider API keys. However, NemoClaw does not inspect the content of allowed requests -- an agent allowed to reach an endpoint could theoretically exfiltrate data through it.
