Architecture

Plugin-blueprint design, key components, data flow, and the design decisions behind NemoClaw's out-of-process enforcement model.

High-Level Design

NemoClaw uses a plugin-blueprint architecture with two distinct layers sitting on top of OpenShell, the actual runtime.

NemoClaw Architecture
Plugin Layer (TypeScript)
⚙️
CLI Commands
TypeScript

Registers nemoclaw onboard, nemoclaw list, and other CLI commands. Handles blueprint resolution, version compatibility checks, and user interaction.

💬
Slash Command
/nemoclaw

The /nemoclaw slash command inside OpenClaw sessions. Provides quick health checks and status queries from within the agent environment.

Blueprint Layer (Python)
📐
blueprint.yaml
Config

Defines resource configuration: gateway settings, sandbox image specs, and default inference profiles. Versioned and integrity-checked with SHA256 digests.

🔄
Lifecycle Engine
Python

Implements the five-stage lifecycle: Resolve (validate version), Verify (check SHA256), Plan (determine changes), Apply (execute via OpenShell CLI), Status (report state).

📜
Baseline Policies
YAML

Default policies covering 9 baseline groups: Claude Code, NVIDIA APIs, GitHub, ClawHub, OpenClaw, npm, and Telegram. Community presets available for PyPI, Docker Hub, Slack, and Jira.

OpenShell Runtime (Docker + K3s)
🛡️
Gateway
Control Plane

Central control plane running as a separate Linux user. Intercepts every outbound request, enforces policies, injects credentials, and provides the real-time TUI for operator approvals.

📦
Sandbox
Docker + Landlock

Isolated Docker container with restricted Linux capabilities. Uses Landlock LSM for filesystem restrictions (writes only to /sandbox and /tmp) and seccomp for syscall filtering.

⚖️
Policy Engine
YAML Rules

Declarative rules covering filesystem, network, process, and inference. Filesystem and process policies lock at creation. Network and inference policies are hot-reloadable.

🔏
Privacy Router
Data Classifier

Inspects inference queries for sensitive content (PII, proprietary code, financials). Sensitive queries route to local models; non-sensitive to cloud providers.

☸️
K3s Cluster
Kubernetes

Lightweight Kubernetes inside the Docker container. Manages gateway, sandbox, and policy engine as separate services. Takes 1-3 minutes to apply manifests after onboarding.

Insight — Out-of-process enforcement: The most critical architectural decision is that security constraints live outside the agent process. The Gateway runs as a separate Linux user from the agent, so even a fully compromised agent (via prompt injection or a malicious tool) cannot kill, modify, or bypass the Gateway. This is fundamentally different from prompt-based guardrails, where the agent itself is trusted to enforce its own limits.

Data Flow

Here is what happens when an agent inside the sandbox tries to call an external API (for example, api.github.com):

Request Lifecycle: Agent to External API
1
Agent initiates request

The Claw inside the sandbox makes an HTTP request — for example, calling api.github.com to create a pull request.

2
DNS interception

The request hits the sandbox's network stack. Model inference calls are redirected to inference.local (a local DNS entry pointing to the Gateway). Other requests proceed to the outbound network path.

3
Gateway intercept

OpenShell's Gateway intercepts the outbound connection before it leaves the container. The agent cannot bypass this — it is enforced at the network level.

4
Policy check

The Gateway looks up the destination host, port, requesting binary, and HTTP method against the YAML policy. Each dimension must match an allow rule.

5
Decision

Allowed: Request passes; credentials are injected by the Gateway (never stored in the sandbox). Blocked: Connection is denied and logged; the operator sees a real-time prompt in the TUI. Inference route: Model calls are redirected through the Privacy Router for sensitivity classification.

6
Response

The response flows back through the Gateway to the agent. The entire round trip is logged. Credentials are stripped before the response enters the sandbox.

Design Decisions

Each major architectural choice in NemoClaw was made deliberately. Here are the five most important decisions and why they matter:

Security

Out-of-process enforcement over prompt-based guardrails

The most important architectural decision. Security constraints are enforced at the OS/network level, not by asking the model to behave. This means even a fully compromised agent (via prompt injection or malicious tool) is still contained by the sandbox.

Security

Deny-by-default networking

All outbound connections are blocked unless explicitly allowed in the policy YAML. This is the opposite of most development environments and is deliberately restrictive. The rationale: it is far safer to start locked down and selectively open access than to start open and try to block specific threats.

Security

Credential isolation

API keys are stored on the host in ~/.nemoclaw/credentials.json (permissions 0600) and injected by the Gateway at request time. The agent never sees raw credentials. This prevents credential theft even if the sandbox is fully compromised.

Architecture

Two-user container model

The sandbox runs two distinct Linux users: a gateway user and a sandbox user. This prevents the agent from escalating privileges to modify the gateway process, even within its own container.

Operations

Static + dynamic policy split

Filesystem and process policies are locked at container creation (cannot be changed at runtime), while network and inference policies are hot-reloadable. This balances security (core isolation cannot be weakened) with operational flexibility (network rules can adapt to new requirements without a restart).

💡
Key takeaway: NemoClaw's architecture can be summarized in one sentence: a TypeScript plugin resolves a Python blueprint that configures an OpenShell runtime where the Gateway, Policy Engine, and Privacy Router enforce security outside the agent process. Every layer is independently auditable and none depends on the agent cooperating.
