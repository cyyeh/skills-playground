Architecture

Four-Layer System Design

NemoClaw is a two-tier system layered atop NVIDIA OpenShell. The architecture follows a deliberate separation of concerns: a lightweight TypeScript plugin handles user interaction, while a versioned Python blueprint manages all orchestration logic for sandbox creation, policy enforcement, and inference routing.

⌨️

Layer 1: CLI (TypeScript Plugin)

nemoclaw/src/

The plugin registers commands under openclaw nemoclaw using Commander.js. It contains the entry point (index.ts), CLI wiring (cli.ts), and command modules for launch, connect, status, logs, and slash-command handling. This layer runs in-process with the OpenClaw gateway and is intentionally kept thin.

Its job: accept user input, resolve the correct blueprint version, verify its digest, and delegate all orchestration to the blueprint subprocess.

📐

Layer 2: Orchestration (Python Blueprint)

nemoclaw-blueprint/

The blueprint is a versioned artifact containing the Dockerfile, security policies, network rules, and Python orchestration logic. It follows a five-stage lifecycle: resolve (locate and validate), verify (check digest), plan (determine required resources), apply (execute via OpenShell CLI), and status (report state).

The blueprint is immutable once published — any changes require a new version with a new digest. This guarantees supply-chain integrity.

🛡️

Layer 3: Security (OpenShell Runtime)

NVIDIA Agent Toolkit

NVIDIA OpenShell provides the foundational security primitives: sandbox container management, Landlock filesystem policies, seccomp BPF syscall filters, network namespace configuration, and the inference routing gateway.

OpenShell operates entirely outside the agent's reach — the agent cannot modify, disable, or bypass its controls. The policy engine accepts rules from the blueprint and enforces them at the kernel level.

🤖

Layer 4: Agent (OpenClaw)

Inside Sandbox

The OpenClaw agent runs inside the sandbox container. It sees a standard Linux environment with /sandbox and /tmp as writable directories, a local inference endpoint at inference.local, and network access limited to policy-approved endpoints.

From the agent's perspective, it operates normally — the security controls are transparent and cannot be detected or circumvented.

Component Interaction Chain

Request Flow

1
User → Plugin
Operator runs nemoclaw onboard, connect, or other CLI commands.
▼
2
Plugin → Blueprint
Plugin downloads the blueprint artifact, verifies its digest, and executes it as a subprocess.
▼
3
Blueprint → OpenShell
Blueprint calls openshell sandbox create, openshell provider add, and openshell policy apply.
▼
4
OpenShell → Sandbox
OpenShell creates the isolated container, applies security policies, starts the agent process.
▼
5
Agent → Gateway
All inference requests go through inference.local. OpenShell routes them to the configured provider with host-side credentials.

Repository Structure

  • bin/ — CLI entry point (CommonJS wrapper)
  • nemoclaw/ — TypeScript plugin source (Commander CLI extension)
  • nemoclaw-blueprint/ — Blueprint YAML, policies, Dockerfile, Python orchestration
  • scripts/ — Installation, uninstallation, and automation helpers
  • test/ — Integration and end-to-end tests
  • docs/ — Sphinx/MyST documentation source

Design Principles

💡
Five core tenets: (1) Thin plugin, versioned blueprint — the plugin stays stable; orchestration evolves in the blueprint. (2) CLI boundary respect — no hidden background processes. (3) Supply chain safety — artifacts are immutable and digest-verified. (4) OpenShell-native — direct use of openshell sandbox create. (5) Reproducible setup — re-running recreates identical sandboxes.
