Layer Details

Layer 1: TypeScript Plugin Layer

▼

The plugin layer is a TypeScript Commander CLI extension that integrates with the OpenClaw CLI. It is the operator's primary interface for managing NemoClaw deployments.

  • Command registration: Exposes launch, connect, status, and logs commands for sandbox lifecycle management
  • Blueprint resolution: Locates and downloads blueprint artifacts from OCI registries with local caching to avoid repeated downloads
  • Digest verification: Ensures artifact integrity by validating OCI digests before deployment, aborting if tampering is detected
  • Subprocess execution: Invokes the Python blueprint runner as a child process, maintaining separation between CLI and orchestration layers
  • State management: Persists sandbox configurations in ~/.nemoclaw/sandboxes.json on the host filesystem

The plugin also provides the /nemoclaw slash command within the agent's chat interface, giving operators real-time control over sandbox behavior without leaving the conversation.

Layer 2: Python Blueprint Layer

▼

The blueprint is a versioned artifact that declaratively defines the sandbox environment. It follows infrastructure-as-code principles, making deployments reproducible and auditable.

  • blueprint.yaml: Version constraints and the deployment manifest specifying compatible OpenShell and OpenClaw versions
  • openclaw-sandbox.yaml: Network egress rules, filesystem mount policies, process restrictions, and seccomp profiles
  • Blueprint Runner: A five-stage lifecycle executor:
    • Resolve -- Locate the blueprint artifact and check version compatibility
    • Verify -- Validate the OCI digest for tamper detection
    • Plan -- Calculate the diff between current and desired sandbox state
    • Apply -- Execute OpenShell CLI commands to create or update resources
    • Status -- Report the sandbox's current configuration and health

Layer 3: OpenShell Runtime (The Security Boundary)

▼

This is the most critical layer -- the point where NemoClaw diverges from permissive agent frameworks. OpenShell provides the actual security enforcement.

Out-of-Process Policy Engine: A Rust-based service running in a separate process from the agent. It intercepts system calls and network requests at the kernel boundary. Because the engine runs outside the agent's address space, a compromised agent cannot tamper with its own restrictions.

Kernel Security Primitives:

  • Landlock: Linux security module for fine-grained filesystem access control. The sandbox can only read and write to /sandbox and /tmp; system paths are read-only.
  • seccomp: Syscall filtering that restricts which kernel functions the sandbox process can invoke, blocking operations like raw socket creation or kernel module loading.
  • Network namespaces: Complete network isolation. The sandbox has its own network stack with no default routes. All traffic must go through the OpenShell gateway.
  • Capability dropping: The container runs with minimal Linux capabilities, preventing privilege escalation.

Network Policy Enforcement: Every outbound connection is validated against the YAML policy. Unknown destinations are blocked by default. When an agent attempts to reach an unapproved endpoint, the request is surfaced in the operator's TUI for real-time approval or denial.

Layer 4: Inference Layer

▼

The inference layer manages all LLM communication, providing privacy routing and credential isolation.

Privacy Router: Classifies each inference request by data sensitivity using organization-defined rules. Sensitive requests (PII, proprietary code, financial data) route to local Nemotron models. Non-sensitive requests go to cloud providers for higher capability.

Credential Injection Gateway: The sandbox never holds API credentials. The gateway intercepts outgoing inference requests and injects authentication tokens based on the configured provider.

Supported Providers:

  • NVIDIA Endpoints (default): nvidia/nemotron-3-super-120b-a12b
  • Local Ollama: For air-gapped environments with any Ollama-compatible model
  • Custom providers: Any OpenAI-compatible API endpoint (OpenAI, Anthropic, Google, etc.)

Key Architectural Insight: The defining innovation of NemoClaw is out-of-process policy enforcement. Instead of embedding security rules in the agent's prompts (which can be bypassed) or in the agent's application code (which shares the agent's process), NemoClaw enforces security from a separate Rust process at the Linux kernel level. This creates a trust boundary that holds even if the agent is fully compromised.

Security Architecture Principles

Defense in Depth

▼

Multiple independent security layers ensure that no single failure compromises the system. Even if the agent bypasses one control (e.g., a Guardrails rule), the kernel-level restrictions (Landlock, seccomp, network namespaces) remain active. Each layer operates independently and does not depend on the others functioning correctly.

Least Privilege

▼

The sandbox starts with no capabilities and adds only what is explicitly needed. Network, filesystem, and process access must be granted by policy. This mirrors the security principle used in production infrastructure: deny everything by default, explicitly permit only what is required.

Credential Isolation

▼

API keys, tokens, and authentication secrets live on the host in ~/.nemoclaw/credentials.json, never in the sandbox. The Inference Gateway injects credentials at the boundary when proxying requests. A sandbox compromise cannot leak credentials because they were never present inside the sandbox.

Data Flow Summary: User message arrives via messaging bridge (SSH) → OpenClaw plans response → Tools execute inside sandbox → Network Policy Engine evaluates outbound requests → Privacy Router classifies inference sensitivity → Inference Gateway proxies with injected credentials → Response returns through same path → Every step logged in audit trail.
