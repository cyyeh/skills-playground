Implementation Details

Getting Started

Prerequisites: Linux (Ubuntu 22.04+), Docker 20.x+, Node.js 22.16+, npm 10+, 8 GB RAM minimum (16 GB recommended), 20 GB disk (40 GB recommended).

Bash 
# Install NemoClaw
curl -fsSL https://www.nvidia.com/nemoclaw.sh | bash

# Run the interactive onboarding wizard
nemoclaw onboard

# Connect to your sandbox
nemoclaw my-sandbox connect

# You're now inside the secured environment
# OpenClaw is pre-installed; start your agent
openclaw

The onboard command walks through: creating the OpenShell gateway, registering inference providers, building the sandbox image (~2.4 GB), creating your first sandbox, and applying the baseline network policy.

⚠️
Warning: Do not use piped installs (curl | bash) in automated environments -- they can skip critical prompts. The sandbox image is ~2.4 GB; on machines with less than 8 GB RAM, the build can trigger the OOM killer.

CLI Reference

NemoClaw provides a comprehensive CLI for sandbox management, policy configuration, and auxiliary services.

nemoclaw onboard Interactive setup wizard -- creates gateway, registers providers, builds sandbox nemoclaw list List all sandboxes with model/provider/policy info nemoclaw <name> connect SSH into a sandbox nemoclaw <name> status Show sandbox health and inference config nemoclaw <name> logs [--follow] View or stream sandbox logs nemoclaw <name> destroy Permanently delete a sandbox
nemoclaw <name> policy-add Add a policy preset to a sandbox nemoclaw <name> policy-list List available and applied policy presets openshell policy set <policy-file> Apply a policy to a running sandbox (session-only)
nemoclaw start Start auxiliary services (Telegram bridge, tunnel) nemoclaw stop Stop all auxiliary services nemoclaw status Show sandbox list and service status openshell term Launch TUI for monitoring and egress approvals
nemoclaw deploy <instance-name> Deploy to remote GPU via Brev (experimental) sudo nemoclaw setup-spark DGX Spark-specific setup (cgroup/Docker fixes for Ubuntu 24.04) /nemoclaw status Quick health check inside an OpenClaw session (slash command)

Code Patterns

Custom Network Policy

YAML 
# my-custom-policy.yaml
policies:
  my_internal_api:
    endpoints:
      - host: api.internal.company.com
        port: 443
    allowed_binaries:
      - /usr/local/bin/claude
      - /usr/local/bin/openclaw
    methods:
      - GET
      - POST
Bash 
# Apply statically (persists across restarts)
# Edit nemoclaw-blueprint/policies/openclaw-sandbox.yaml, then:
nemoclaw onboard

# Apply dynamically (session-only)
openshell policy set my-sandbox --policy my-custom-policy.yaml
💡
Tip: Start with cloud inference for evaluation -- local GPU setup adds significant complexity. Monitor with openshell term during initial agent runs to see what network access the agent needs, then build your policy around observed behavior.

Deployment Gotchas

These are the issues that consistently trip up new users. Knowing them in advance will save you hours of debugging.

  • K3s takes 1-3 minutes to apply manifests after onboarding; health checks may time out with misleading namespace errors during this window
  • Podman is not supported -- NemoClaw requires Docker daemon-specific behaviors
  • macOS (Apple Silicon) has partial support via Colima/Docker Desktop with known issues (Ollama routing, /etc/hosts resolution)
  • Intel Macs are unsupported entirely; WSL2 is experimental with significant manual intervention
  • For DGX Spark: Run sudo nemoclaw setup-spark first to fix cgroup v2 and Docker issues on Ubuntu 24.04
ℹ️
Note: NemoClaw is alpha software. Known CVEs include CVE-2026-24763 (Docker sandbox command injection) and CVE-2026-22179 (shell parsing bypass). These are classical container escape vectors, not fundamental design flaws, but they underline why NVIDIA labels this as not production-ready.
