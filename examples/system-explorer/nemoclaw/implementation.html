Implementation Details

Getting started, CLI reference, configuration locations, code patterns, and deployment considerations for NemoClaw.

Getting Started

Prerequisites

  • 🐧 Linux Ubuntu 22.04+
  • 🐋 Docker 20.x+
  • 🟢 Node.js 22.16+
  • 📦 npm 10+
  • 💾 RAM 8 GB min (16 GB rec.)
  • 💿 Disk 20 GB min (40 GB rec.)

Installation

Bash 
# Install NemoClaw
curl -fsSL https://www.nvidia.com/nemoclaw.sh | bash

# Run the interactive onboarding wizard
nemoclaw onboard

The onboard command walks through five steps:

  1. Creating the OpenShell gateway
  2. Registering inference providers (enter API keys)
  3. Building the sandbox image (~2.4 GB, takes a few minutes)
  4. Creating your first sandbox
  5. Applying the baseline network policy

Connect and Run

Bash 
# Connect to your sandbox
nemoclaw my-sandbox connect

# You're now inside the secured environment
# OpenClaw is pre-installed; start your agent
openclaw
💡
Start with cloud inference for evaluation Local GPU setup adds significant complexity. Use cloud inference providers (Anthropic, OpenAI, or NVIDIA endpoints) for your first evaluation, then migrate to local models once you understand the system.

Configuration Essentials

NemoClaw stores configuration across six locations. Most are managed automatically by the blueprint and onboarding wizard, but understanding where each lives is critical for debugging and customization.

Credentials
~/.nemoclaw/credentials.json
API keys for inference providers. Stored with 0600 permissions. Never enters the sandbox.
When adding or rotating provider keys
Sandbox Metadata
~/.nemoclaw/sandboxes.json
Sandbox names, defaults, and model assignments for all managed sandboxes.
When managing multiple sandboxes
Network Policy
nemoclaw-blueprint/policies/openclaw-sandbox.yaml
Allowed endpoints, binaries, and HTTP methods. The primary policy file for network access control.
When the agent needs new network access
Policy Presets
nemoclaw-blueprint/policies/presets/
Pre-built rules for common services: PyPI, Docker Hub, Slack, Jira, and more.
When enabling common integrations
OpenShell State
~/.config/openshell/
Gateway and cluster state. Internal bookkeeping for the runtime.
Rarely -- managed automatically
Blueprint Config
nemoclaw-blueprint/blueprint.yaml
Resource definitions and orchestration logic. Defines the full deployment topology.
When customizing the deployment

CLI Reference

The full NemoClaw CLI organized by functional group. Each command is available after installation. Use nemoclaw --help for inline documentation.

nemoclaw onboard Interactive setup wizard. Creates gateway, registers providers, builds sandbox image, creates sandbox, applies baseline policy.
nemoclaw list List all sandboxes with model, provider, and policy information.
nemoclaw <name> connect SSH into a named sandbox. Opens an interactive shell inside the secured environment.
nemoclaw <name> status Show sandbox health, inference configuration, and gateway status.
nemoclaw <name> logs [--follow] View or stream sandbox logs. Use --follow for real-time monitoring.
nemoclaw <name> destroy Permanently delete a sandbox and all its data. This action is irreversible.
nemoclaw <name> policy-add Add a policy preset to a sandbox. Appends rules from the presets directory to the active policy.
nemoclaw <name> policy-list List all available and currently applied policy presets for a sandbox.
nemoclaw start Start auxiliary services such as the Telegram bridge and network tunnel.
nemoclaw stop Stop all running auxiliary services.
nemoclaw status Show sandbox list and service status overview.
nemoclaw deploy <instance-name> Deploy to a remote GPU via Brev (experimental). Provisions and configures a remote NemoClaw instance.
sudo nemoclaw setup-spark DGX Spark-specific setup. Fixes cgroup v2 and Docker issues on Ubuntu 24.04.
openshell term Launch the terminal UI for monitoring sandbox activity and real-time egress approvals.
openshell policy set <policy-file> Apply a policy to a running sandbox. Changes are session-only and do not persist across restarts.

Code Patterns

Custom Network Policy

Adding a custom network policy for a new internal API. This grants specific binaries access to your endpoint while blocking all other processes.

YAML 
# my-custom-policy.yaml
policies:
  my_internal_api:
    endpoints:
      - host: api.internal.company.com
        port: 443
    allowed_binaries:
      - /usr/local/bin/claude
      - /usr/local/bin/openclaw
    methods:
      - GET
      - POST

Apply the policy statically (persists across restarts) or dynamically (session-only):

Bash 
# Apply statically (persists across restarts)
# Edit nemoclaw-blueprint/policies/openclaw-sandbox.yaml, then:
nemoclaw onboard

# Apply dynamically (session-only)
openshell policy set my-sandbox --policy my-custom-policy.yaml

Slash Command Inside OpenClaw

Once inside the sandbox during an OpenClaw session, you can use the built-in slash command for quick health checks:

Bash 
# Inside the sandbox, within an OpenClaw session:
/nemoclaw status    # Quick health check of sandbox, gateway, policies

Deployment Considerations

Practical guidance for production-path evaluation. NemoClaw is alpha software, so these considerations are especially important for avoiding common pitfalls.

⚠️
OOM risk on machines with less than 8 GB RAM The sandbox image is approximately 2.4 GB compressed. On machines with less than 8 GB RAM, the build process can trigger the Linux OOM killer, terminating the build or other running processes without warning. Use 16 GB RAM or higher for reliable operation.
  • Monitor with openshell term during initial agent runs to see what network access the agent needs, then build your policy around observed behavior.
  • K3s takes 1-3 minutes to apply manifests after onboarding. Health checks may time out with misleading namespace errors during this window.
  • Do not use piped installs (curl | bash) in automated environments -- they can skip critical prompts.
  • Podman is not supported -- NemoClaw requires Docker daemon-specific behaviors.
  • For DGX Spark: Run sudo nemoclaw setup-spark first to fix cgroup v2 and Docker issues on Ubuntu 24.04.
ℹ️
Debugging network issues Use openshell term to see all blocked requests in real time. For deeper debugging, nemoclaw <name> logs --follow streams sandbox logs. The sandbox includes standard networking tools: ping, dig, nslookup, nc, traceroute, netstat. The most common issue is a legitimate request being blocked by the deny-by-default policy.
