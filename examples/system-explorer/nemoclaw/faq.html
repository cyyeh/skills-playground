The sandbox is designed to contain even a worst-case scenario where an attacker achieves arbitrary code execution inside the container. The three-layer isolation (Landlock for filesystem, seccomp for syscalls, network namespaces for networking) is enforced at the Linux kernel level, not by the application. A compromised agent cannot mount new filesystems, trace other processes, or bypass network restrictions. Most importantly: API keys are stored on the host and injected by the OpenShell gateway, so even full sandbox compromise cannot exfiltrate credentials. That said, container escapes via kernel vulnerabilities are not theoretically impossible -- which is why defense in depth matters.

A plain Docker container with --network=none plus a reverse proxy could approximate some functionality, but you would reimplement: inference routing that keeps API keys host-side, the per-domain/per-method network policy engine, the real-time approval workflow for blocked requests, blueprint versioning with digest verification, and the onboarding wizard that validates inference providers. NemoClaw packages these into a coherent stack. The trade-off is complexity: NemoClaw adds OpenShell as a dependency.

Yes. Each sandbox is an independent container instance with its own name (RFC 1123 format), network namespace, and policy configuration. Run nemoclaw onboard multiple times to create different sandboxes with different inference providers or network policies. The nemoclaw list command shows all registered sandboxes. The practical limit is host resources.

The routing adds one extra network hop (sandbox to gateway to provider). For cloud providers, this overhead is negligible compared to LLM inference time (hundreds of milliseconds to seconds). For local inference (Ollama, NIM), the hop is entirely local with sub-millisecond latency. The network policy check is a local table lookup. The most noticeable delay is during initial sandbox setup (~2.4 GB container image download).

Two approaches: (1) For immediate, session-scoped access: wait for the agent to attempt the request, then approve it in the OpenShell TUI (openshell term). (2) For persistent access: use openshell policy set <policy-file> with an updated YAML file. For changes surviving sandbox recreation, edit the policy YAML in the blueprint and re-onboard.

No. NemoClaw is hardware-agnostic at the agent layer. The sandbox runs on any Linux machine with Docker, regardless of GPU vendor. NVIDIA-specific optimizations are in the inference layer: Nemotron models and NIM run best on NVIDIA GPUs. But you can use non-NVIDIA providers (OpenAI, Anthropic, Ollama with CPU-only models) on AMD or Intel hardware. Landlock, seccomp, and namespaces are Linux kernel features, not NVIDIA-specific.

The sandbox container persists across reboots if Docker is configured to restart containers automatically. Agent state inside /sandbox is preserved. However, session-level policy approvals (granted via the TUI) are not persisted -- only static policy changes (in the YAML file) survive restarts. NemoClaw includes backup and restore procedures for the /sandbox directory.

Yes, through the "Custom OpenAI-compatible" or "Custom Anthropic-compatible" options during onboarding. Any endpoint speaking the OpenAI /chat/completions or Anthropic /v1/messages protocol can be configured. NemoClaw validates custom endpoints by sending a real inference request rather than checking a /models endpoint.

As of March 2026, NemoClaw is explicitly alpha software. Released on March 16, 2026, it has not yet had a production-ready release. APIs, configuration schemas, and runtime behavior may change without notice. The project has 17,800+ GitHub stars, but stars measure interest, not stability. For production environments, treat it as an evaluation tool and plan for re-onboarding during upgrades.
