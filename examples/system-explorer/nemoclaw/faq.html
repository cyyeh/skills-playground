Common Q&A

10 frequently asked questions about NemoClaw, from basics to security architecture.

1

Is NemoClaw a model, a framework, or a runtime?

▼

NemoClaw is a runtime and security layer, not a model or a full agent framework. It wraps the existing OpenClaw agent framework with NVIDIA's OpenShell runtime to provide sandboxing, policy enforcement, privacy routing, and managed inference. Think of it as the secure operating environment that an agent runs inside, rather than the agent itself.

2

Do I need NVIDIA GPUs to use NemoClaw?

▼

No. NemoClaw itself requires no GPUs -- its minimum requirement is just 4 vCPU, 8 GB RAM, and 20 GB disk. GPUs are only needed if you want to run Nemotron models locally for privacy routing. Without local GPUs, you can route all inference to cloud providers through the NemoClaw gateway.

The full Nemotron 3 Super 120B works best on DGX-class hardware, while the Nano 4B variant can run on consumer RTX cards.

3

Is NemoClaw production-ready?

▼

Not yet. As of March 2026, NemoClaw is labeled as alpha software in early preview. The official documentation states: "APIs, configuration schemas, and runtime behavior are subject to breaking changes between releases. Do not use this software in production environments."

The architecture and security model are sound, but the project has not undergone independent security audits, and the API surface is still stabilizing.

4

How does NemoClaw differ from running OpenClaw in Docker?

▼

The difference is significant. A standard Docker container provides process isolation but does not enforce fine-grained policies. NemoClaw adds:

  • Landlock filesystem restrictions (more granular than Docker volume mounts)
  • seccomp syscall filtering (blocking dangerous kernel operations)
  • Network namespaces with deny-by-default egress (standard containers have full outbound access)
  • Out-of-process policy enforcement (a Rust engine the agent cannot tamper with)
  • Privacy routing for inference (Docker has no concept of this)
  • Credential isolation (API keys never enter the container)
5

Can I use models other than Nemotron?

▼

Yes. NemoClaw supports any inference provider through three mechanisms:

  • NVIDIA Endpoints for NVIDIA-hosted models (default: Nemotron 3 Super)
  • Local Ollama for any Ollama-compatible model (Llama, Mistral, Qwen, etc.)
  • Custom OpenAI-compatible providers for cloud APIs (OpenAI GPT-4o, Anthropic Claude, Google Gemini)

The Privacy Router decides which provider to use per-request based on data sensitivity.

6

How much latency does NemoClaw add?

▼

The security pipeline adds approximately 300ms to the first request in a session (sandbox initialization, policy engine startup) and under 100ms for subsequent requests (due to policy caching). In practice, LLM inference time (1-10 seconds) dominates total latency, making NemoClaw's overhead negligible.

7

Can the agent bypass NemoClaw's security controls?

▼

By design, no. Unlike prompt-level guardrails (which can be bypassed through injection), NemoClaw's security is enforced at the Linux kernel level by a separate process:

  • Landlock prevents filesystem access regardless of what the agent attempts
  • seccomp blocks system calls before they reach the kernel
  • Network namespaces mean the agent has no network interface to manipulate
  • The Policy Engine runs in its own process -- the agent cannot modify it

However, no security system is perfect -- NemoClaw has not yet been independently audited, and novel kernel exploits could theoretically bypass these controls.

8

How does NemoClaw handle multi-agent scenarios?

▼

OpenClaw supports supervisor-worker patterns where a supervisor agent delegates tasks to specialized worker agents. In NemoClaw, each agent can run in its own sandbox with its own policy. For example: a supervisor agent has access to Slack and Jira, a code-writing worker only has access to GitHub, and a data analysis worker only has access to the internal database.

This provides per-agent blast radius limitation -- a compromised worker can only affect the systems it was explicitly granted access to.

9

What operating systems does NemoClaw support?

▼

NemoClaw primarily targets Ubuntu 22.04 LTS or later on Linux, as it relies on kernel security features (Landlock, seccomp, network namespaces). For macOS, it works via Colima or Docker Desktop, though kernel-level security is provided by the container runtime's Linux VM. For Windows, Docker Desktop with WSL is supported with the same caveat. Full security guarantees require a native Linux kernel.

10

How do I monitor what the agent is doing?

▼

NemoClaw provides three monitoring mechanisms:

  • nemoclaw logs -- Streams structured audit logs including tool calls, network requests, and inference routing decisions
  • nemoclaw status -- Shows sandbox health, active policies, resource usage, and provider connectivity
  • openshell term -- Opens a TUI for real-time interaction: viewing pending approval requests, inspecting connections, and managing state

All audit data is structured JSON, making it straightforward to pipe into enterprise logging systems (ELK, Splunk, Datadog).
