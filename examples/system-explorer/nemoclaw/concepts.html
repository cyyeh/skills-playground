Core Concepts

Key Building Blocks

NemoClaw is built from a small set of interconnected concepts. Understanding these seven building blocks will give you a complete mental model of how the system works and why each piece exists.

🔒

Sandbox

An isolated container environment where the OpenClaw agent runs -- like a sealed clean room in a laboratory.

The agent can work freely inside the room, but every entrance and exit is monitored and controlled. NemoClaw sandboxes enforce filesystem boundaries (the agent can only write to /sandbox and /tmp) and network restrictions using Linux kernel mechanisms including Landlock, seccomp, and network namespace isolation. The sandbox denies everything by default and only permits what is explicitly allowed.

📐

Blueprint

A versioned artifact that defines how a sandbox should be configured -- like an architectural plan for a building.

Blueprints contain a manifest file (blueprint.yaml) and policy definitions (openclaw-sandbox.yaml). The blueprint lifecycle follows five stages: resolve version constraints, verify artifact digest, plan required resource changes, apply those changes, and report system state. Blueprints ensure that sandbox deployments are reproducible and tamper-evident.

🛡️

Network Policy

A declarative YAML file defining exactly which external endpoints the agent may contact -- like a corporate firewall whitelist.

NemoClaw operates on a deny-by-default model: if an endpoint is not listed, the agent cannot reach it. Policies specify allowed domains, ports (all connections enforce TLS on port 443), and HTTP methods. Documentation endpoints may allow only GET requests, for example.

🔀

Inference Routing

Directs the agent's LLM API calls through a controlled path -- like routing phone calls through a corporate switchboard.

The agent communicates with a local endpoint (inference.local) inside the sandbox, while OpenShell on the host manages actual credentials and upstream connections. API keys never enter the sandbox, preventing credential leakage even if the agent is compromised.

⚙️

OpenShell

The NVIDIA runtime that NemoClaw depends on -- like the operating system on which NemoClaw applications run.

OpenShell provides the gateway that intercepts and routes inference requests, the kernel-level isolation mechanisms for sandboxing, and the TUI (terminal user interface) for monitoring and approving network requests in real time. NemoClaw is a purpose-built configuration layer on top of OpenShell.

✅

Approval Workflow

The human-in-the-loop process for reviewing network requests to unlisted endpoints -- like a security guard at the loading dock.

When the agent tries to reach a host not in the network policy, OpenShell blocks the request and surfaces it in the TUI for the operator to approve or deny. Approved requests can be added to the session-level policy to avoid repeat approvals for the same endpoint.

🤖

Inference Provider

A backend that serves LLM responses to the agent -- like choosing which telephone company handles your calls.

NemoClaw supports six categories: NVIDIA Endpoints (including Nemotron), OpenAI, Anthropic, Google Gemini, custom OpenAI-compatible proxies, and local self-hosted runtimes (Ollama, NVIDIA NIM, vLLM). The agent never sees raw API credentials regardless of which provider is selected.

How They Fit Together

NemoClaw's concepts form a layered security model. Here is the flow from deployment specification to running agent:

1
Blueprint

Defines the desired state of the deployment

→
2
Sandbox

Isolated environment created from the blueprint

→
3
Network Policy

Governs outbound traffic from the sandbox

→
4
Inference Routing

Channels model calls through OpenShell to providers

→
5
Approval Workflow

Human review for requests to unlisted endpoints

🔭
Architecture insight: The result is a system where an autonomous agent can operate continuously while every boundary -- filesystem, network, and credentials -- is explicitly controlled. No single mechanism is the sole line of defense; they work together as layers.
