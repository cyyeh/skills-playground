Core Concepts

The seven building blocks of NemoClaw — what each one does, why it exists, and a real-world analogy to make it stick.

The Building Blocks

NemoClaw is built from seven core concepts. Each one has a specific job, and together they form a complete security wrapper around your AI agent. Click "Learn more" on any card to see a real-world analogy.

🦾

Claws (Agents)

The autonomous AI agent running inside NemoClaw. A Claw is an OpenClaw agent that can think, plan, and act on its own. NemoClaw does not change how Claws work — it controls the environment they operate in.

Analogy: A Claw is like a new employee at your company — capable and eager, but needs guardrails on what systems they can access on day one. You trust their skills but limit their keycard access until they prove themselves.

📦

OpenShell

The sandboxed runtime where agents actually execute. Everything the agent needs is inside, but nothing unauthorized gets in or out. Under the hood, it is a K3s Kubernetes cluster in a single Docker container.

Analogy: OpenShell is a secure clean room — everything the worker needs (tools, files, approved network access) is inside. The door locks from the outside, and security cameras watch all traffic in and out.

📜

Policies

Declarative YAML files that define exactly what an agent can and cannot do — which websites it can reach, which files it can write, which system calls it can make. Follows a deny-by-default model.

Analogy: Policies are like firewall rules for a single application, but covering network, filesystem, and process behavior all at once. If it is not on the approved list, it is blocked — no exceptions.

🛡️

Gateway

The control-plane component sitting between the agent and the outside world. Every network request, inference call, and credential use passes through the Gateway. It logs everything and can stop suspicious activity in real time.

Analogy: The Gateway is a building security desk — it checks every visitor (request) against the visitor list (policy), logs every entry and exit, and can deny entry to anyone not on the list, even on a busy day.

🔏

Privacy Router

Inspects each inference query and classifies its sensitivity. Queries containing PII, proprietary code, or financial data get routed to local models and never leave your infrastructure.

Analogy: The Privacy Router is a mail room that sorts all outgoing letters. Confidential mail goes by secure courier (local Nemotron model), while regular mail goes by standard post (cloud API). The sender never has to decide — the mail room knows the rules.

📐

Blueprint

A versioned artifact containing orchestration logic, default policies, and configuration for setting up NemoClaw. It follows a five-stage lifecycle: Resolve, Verify, Plan, Apply, Status.

Analogy: A Blueprint is like an IKEA assembly manual for your security infrastructure — it defines what components are needed, how they connect, and what the default settings should be. You follow the steps, and a working secure environment comes out at the end.

📞

Inference Profile

Defines which AI models handle which queries and where they run. Maps the agent's generic "I need a model" requests to specific providers — cloud APIs, local GPUs, or hybrid configurations.

Analogy: An Inference Profile is a phone switchboard — the caller (agent) dials a single number, and the operator (profile) routes the call to the right destination based on preconfigured rules. The caller never needs to know which line they end up on.

🔭
Insight: Notice that NemoClaw does not modify the agent itself. Every security control is external to the agent process. This is the key architectural principle: even a fully compromised agent (via prompt injection or a malicious tool) is still contained by the sandbox, because the containment lives at the OS and network level, not inside the agent's code.

How They Fit Together

Here is the full flow from setup to a running, secured agent:

1. Install & Blueprint — An operator installs NemoClaw, which uses a Blueprint to set up all the infrastructure. The blueprint resolves versions, verifies integrity, plans resources, and applies the configuration.

2. Sandbox creation — The blueprint creates an OpenShell sandbox — a secure container with a K3s cluster, Gateway, and Policy Engine all running inside.

3. Agent runs inside — Claws (agents) run inside the sandbox. They can write code, browse approved sites, and call APIs — but only within the boundaries set by the sandbox.

4. Gateway enforces — Every action the Claw takes is filtered through the Gateway, which checks the request against the Policies. Allowed requests pass; everything else is blocked and logged.

5. Inference routing — When the Claw needs AI inference, the Privacy Router examines the request for sensitive data. The Inference Profile then routes it to the right model — local for sensitive data, cloud for everything else.

6. Operator monitors — The operator watches everything through the OpenShell terminal UI, approving or denying network requests in real time as the agent works.

💡
Key takeaway: The seven concepts form three logical groups: the agent (Claws), the environment (OpenShell, Blueprint), and the controls (Policies, Gateway, Privacy Router, Inference Profile). Understanding these groups makes the full architecture much easier to reason about.
