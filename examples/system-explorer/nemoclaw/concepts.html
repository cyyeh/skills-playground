Core Concepts

The Building Blocks of Agent Security

NemoClaw is built on six core concepts. Understanding them gives you the vocabulary to reason about how autonomous AI agents can be run more safely, why NemoClaw separates the control plane from the agent process, and how each security layer reinforces the others.

🔒

OpenShell Sandbox

An isolated Linux container environment where the OpenClaw agent runs. It uses kernel-level isolation primitives — Landlock for filesystem access control, seccomp for syscall filtering, and network namespaces for network isolation.

💡
Analogy: Think of a bank vault with a one-way intercom. The agent can work freely inside the vault (run code, process data), but every request to access something outside — a website, a file, a model — must go through the intercom (OpenShell gateway) where a guard (the operator) decides whether to allow it.

Why it matters: Autonomous agents can execute arbitrary code, install packages, and reach external services. Without isolation, a compromised or misbehaving agent has the same access as the user who launched it. The sandbox ensures that even if an agent attempts something unauthorized, the kernel-level controls prevent it from succeeding.

📐

Blueprint

A versioned Python artifact containing the complete specification for creating a NemoClaw sandbox — the Dockerfile, security policies, capability drops, network rules, and inference configuration. It is the reproducible "recipe" for a hardened agent environment.

💡
Analogy: A blueprint is like an architect's building plan. It specifies exactly how the sandbox should be constructed — which walls go where (filesystem boundaries), which doors exist (network egress rules), and what equipment is installed (inference providers). Re-running the blueprint on a different machine produces an identical sandbox.

Why it matters: Reproducibility is essential for security. If every sandbox is built from the same immutable, digest-verified blueprint, you can audit exactly what is running and guarantee that no unauthorized modifications have been introduced.

🔀

Inference Routing

All model API calls are routed through the OpenShell gateway. The agent communicates with inference.local, and OpenShell transparently forwards requests to the configured provider. The agent never sees raw API credentials.

💡
Analogy: Imagine ordering food through a hotel concierge. You tell the concierge what you want (an inference request), and they call the restaurant (the model provider) using the hotel's account. You get your food, but you never see the credit card number. If the concierge decides a restaurant is off-limits, your order simply does not go through.

Why it matters: API keys are high-value secrets. If an agent had direct access to provider credentials, a prompt injection or code execution exploit could exfiltrate them. By keeping credentials on the host and routing through the gateway, NemoClaw ensures that compromising the agent does not compromise your API keys.

🛡️

Network Policy Engine

Controls all outbound connections from the sandbox. A baseline policy file denies all egress by default, then explicitly allows approved endpoints. Unapproved destinations are blocked and surfaced to the operator for approval in real time.

💡
Analogy: Think of a corporate firewall with a "deny all" default rule. Every new destination the agent wants to reach triggers a notification to the security team. They can approve it for the current session or add it permanently. The agent lives in a separate network namespace and physically cannot see the host's interfaces.

Why it matters: Without egress controls, an agent could exfiltrate data to an attacker-controlled server or connect to malicious endpoints. The policy engine provides real-time visibility and control over every outbound connection.

🧩

Plugin / Blueprint Separation

NemoClaw splits into a thin TypeScript plugin (user-facing CLI) and a versioned Python blueprint (orchestration logic). The plugin stays stable across releases while the blueprint evolves independently.

💡
Analogy: The plugin is the car's steering wheel and dashboard — always familiar. The blueprint is the engine — upgradeable without redesigning the controls. You can install a newer engine (blueprint) without changing how you drive (CLI commands).

Why it matters: Independent versioning enables safe upgrades. The plugin validates blueprint versions and verifies digests before execution. The privileged host-side code (plugin) stays minimal while complex orchestration (blueprint) runs in a controlled subprocess.

💾

State Management

NemoClaw manages host-side state for credentials, sandbox metadata, and OpenClaw configuration snapshots. It supports secure migration of agent state across machines, with credential stripping and integrity checks.

💡
Analogy: Think of a diplomat's secure briefcase with a tamper-evident seal. When moving an agent's state between machines, NemoClaw strips out sensitive credentials, seals the remaining configuration with an integrity hash, and re-provisions credentials on the destination.

Why it matters: Production deployments need to survive machine restarts, hardware migrations, and scaling events. State management ensures the agent's learned behaviors and configuration transfer reliably without exposing secrets.

How They Fit Together

When you run nemoclaw onboard, all six concepts work in concert. Here is the journey from a single CLI command to a running, secured agent:

NemoClaw Onboarding Flow

1
CLI Command → Plugin
You run nemoclaw onboard. The TypeScript plugin receives the command and begins the guided setup wizard.
Plugin
▼
2
Blueprint Resolution & Verification
The plugin downloads the blueprint artifact, checks version compatibility, and verifies the cryptographic digest.
Blueprint
▼
3
Sandbox Creation
The blueprint invokes OpenShell to create an isolated container with Landlock, seccomp, and network namespace isolation.
Sandbox
▼
4
Inference Configuration
The gateway is configured to route inference.local calls to your selected provider. Credentials stay on the host.
Inference Routing
▼
5
Policy Application
Network egress rules are loaded. The agent starts in "deny all except explicitly allowed" mode. The operator controls exceptions.
Network Policy
▼
6
Agent Running
The OpenClaw agent runs inside the sandbox with all security layers active. State is persisted on the host for recovery and migration.
State Management
💡
Key insight: These six concepts form an interlocking system. The sandbox isolates the agent. The blueprint ensures the sandbox is reproducible. Inference routing protects credentials. The network policy controls egress. The plugin/blueprint separation enables safe upgrades. State management enables persistence and migration. No single layer provides complete security — they work together as defense-in-depth.
