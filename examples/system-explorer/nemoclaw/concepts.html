Core Concepts

The Building Blocks

NemoClaw is built from seven core concepts that work together to create a secure environment for AI agents. Think of it as a security system for an office building: the agent does the work inside, while everything around it -- the walls, locks, cameras, and security guards -- is controlled by someone else entirely.

🐾
Claws (Agents)
An OpenClaw autonomous AI agent -- like a new employee who is capable but needs guardrails on what systems they can access on day one.

NemoClaw does not change how claws work. It controls the environment they operate in. The agent retains full autonomy over its thinking and planning, but its actions are bounded by the sandbox, policies, and gateway.

📦
OpenShell (Sandbox)
The sandboxed execution environment -- a secure clean room where everything the agent needs is inside, but nothing unauthorized gets in or out.

OpenShell is a standalone NVIDIA project. Under the hood, it runs a K3s Kubernetes cluster inside a single Docker container with multiple isolation layers including Landlock LSM and seccomp.

📜
Policies (Rulebook)
Declarative YAML files defining what an agent can and cannot do -- like firewall rules covering network, filesystem, and process behavior all at once.

Policies follow a deny-by-default model: if access is not explicitly allowed, it is blocked. This is the opposite of most development environments and is deliberately restrictive for safety.

🛡
Gateway (Security Guard)
The control-plane component sitting between the agent and the outside world. Every request passes through the gateway for inspection and logging.

The gateway runs as a separate Linux user from the agent, preventing the agent from killing or modifying gateway processes even if the sandbox is compromised.

🔏
Privacy Router
Inspects inference queries, classifies sensitivity, and routes accordingly -- sensitive data stays local, non-sensitive queries can go to the cloud.

Like a mail room that sorts letters: confidential mail (PII, proprietary code, financial data) goes by secure courier to local models, regular mail goes by standard post to cloud APIs.

📐
Blueprint
A versioned artifact containing orchestration logic, default policies, and configuration -- the installation plan for your security infrastructure.

The blueprint follows a five-stage lifecycle: Resolve, Verify, Plan, Apply, Status. Think of it as an IKEA assembly manual for your security stack.

📞
Inference Profile
Defines which AI models handle which queries and where they run -- like a phone switchboard routing calls to the right destination.

Supports Anthropic, OpenAI, Google Gemini, Ollama, vLLM, NVIDIA NIM, and any OpenAI-compatible endpoint. The agent dials a single number; the profile routes it to the right model.

How They Fit Together

An operator installs NemoClaw, which uses a Blueprint to set up an OpenShell sandbox. The sandbox runs Claws (agents) inside a secure container. Every action the claw takes is filtered through the Gateway, which enforces Policies. When the claw needs AI inference, the Privacy Router examines the request and the Inference Profile routes it to the appropriate model.

🔭
Architecture insight: The key design principle is that security lives outside the agent. Even if the agent is fully compromised via prompt injection or a malicious tool, it physically cannot override its own containment -- because containment is enforced at the OS and network level, not by asking the model to behave.

The Analogy: A Bulletproof Office

NemoClaw is like a bulletproof office where an AI assistant works. The assistant can use the desk, the phone, and the filing cabinet (all pre-approved tools and endpoints), but the walls, locks, and phone line restrictions are controlled by building security, not the assistant.

Even if the assistant is tricked or compromised, it cannot break through the physical containment. The assistant never holds the keys to the building -- those are injected by the security system at the moment they are needed, and taken away immediately after.

💡
Tip: The most important thing to remember is that NemoClaw's security is out-of-process. The agent cannot disable, modify, or bypass its own security controls because those controls run in separate processes under a different Linux user.
