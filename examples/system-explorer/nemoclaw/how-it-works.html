How It Works

Sandbox Isolation: Three-Layer Defense

NemoClaw achieves sandbox isolation through three complementary Linux kernel mechanisms, layered together to create defense in depth -- like a bank vault that uses a combination lock, a key lock, and biometric scanning.

📁

Landlock

Restricts filesystem access at the kernel level. The sandbox permits read-write access only to /sandbox and /tmp, with everything else mounted read-only. Cannot be overridden even with root privileges.

🚫

Seccomp

Filters system calls available to sandbox processes. Blocks calls that could escape the container or modify the host, such as mount, ptrace, and reboot.

🌐

Network Namespaces

Isolates the sandbox's network stack from the host. The agent cannot see host interfaces, bind to host ports, or enumerate connections. All traffic must pass through the OpenShell gateway.

Blueprint Lifecycle

When an operator runs nemoclaw onboard, the system executes a five-stage pipeline that transforms a declarative blueprint into a running sandbox:

→
2

3

5

💡
Tip: The verify stage halts immediately if the digest does not match -- like verifying a wax seal on a letter before reading it. This ensures a blueprint cannot be silently modified after approval.

Inference Routing Pipeline

Inference routing keeps API credentials out of the sandbox while letting the agent call LLM providers. The mechanism works through request transformations -- like a mail forwarding service that rewrites addresses but preserves the letter inside.

1
Agent Call

API call to inference.local

→
2
Gateway Intercept

OpenShell catches at namespace boundary

→
3
Credential Injection

Real API keys added from host

→
4
Provider Forward

Request sent to actual endpoint

→
5
Sanitized Return

Response stripped of host metadata

🔭
Architecture insight: This pipeline is transparent to the agent -- OpenClaw does not need any modifications because the local endpoint is OpenAI-compatible. The wrapping-over-forking philosophy in action.

Network Policy Enforcement

Network policy enforcement operates as a real-time packet filter at the OpenShell gateway -- like a customs checkpoint that inspects every outbound shipment against a manifest of approved destinations.

The policy file (openclaw-sandbox.yaml) defines allowed endpoints as a list of domain-method-port tuples. When the agent attempts an outbound connection:

  1. The gateway resolves the target hostname and checks it against the allowlist
  2. If destination, HTTP method, and port match a policy entry, the connection proceeds
  3. If no match is found, the connection is blocked and logged
  4. The blocked request is surfaced in the OpenShell TUI for operator review
  5. The operator can approve (session-level policy entry) or deny
⚠️
Important: Static policy changes (editing the YAML file) require re-running nemoclaw onboard to take effect. Dynamic updates can be applied to running sandboxes using openshell policy set <policy-file> without a restart.
