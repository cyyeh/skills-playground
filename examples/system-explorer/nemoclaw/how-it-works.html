How It Works

Five-Layer Protection Model

NemoClaw's security is built on five distinct isolation layers, each enforced independently. The key insight is that these layers are independent: even if one is somehow bypassed, the others remain intact.

LayerMechanismTimingAgent Override?
Container Isolation Docker containers with restricted capabilities Locked No
Process Hardening Dropped Linux capabilities, seccomp filtering, 512 process max Locked No
Filesystem Integrity Landlock LSM restricts writes to /sandbox and /tmp only Locked No
Network Control Deny-by-default egress with YAML allowlists per binary/endpoint Hot-reload No
Inference Security All inference routed through gateway; credentials on host only Hot-reload No
🔭
Defense in depth: An agent that tricks its way past network policy still cannot write to system files. An agent that finds a filesystem exploit still cannot reach unapproved network endpoints. Each layer is a complete barrier on its own.

Filesystem Access Model

The Landlock Linux Security Module enforces strict filesystem boundaries. The agent can do useful work in its designated workspace but cannot modify system binaries, tamper with configuration, or access the host filesystem.

Network Policy Enforcement

Policies are defined in YAML with granular control over endpoints, allowed binaries, and HTTP methods. The baseline policy ships with 9 default policy groups covering Claude Code, NVIDIA APIs, GitHub, and more.

YAML 
# Example: Allow GitHub API access only from the gh CLI
policies:
  github_rest_api:
    endpoints:
      - host: api.github.com
        port: 443
    allowed_binaries:
      - /usr/bin/gh
    methods:
      - GET
      - POST
      - PATCH
      - PUT
      - DELETE
💡
Tip: Use openshell term during initial agent runs to see what network access the agent actually needs. Build your policy from observed behavior rather than guessing. Approved endpoints in the TUI persist for the session only -- add them to the YAML policy for permanence.

All model API calls from the sandbox are routed through a local DNS entry (inference.local) to the OpenShell gateway. The gateway then processes and routes the request based on data sensitivity.

Real-Time Network Approval

When an agent tries to reach an unlisted endpoint, NemoClaw creates a "progressive trust" model where the operator starts with minimal access and selectively expands it as the agent demonstrates legitimate needs.

  1. OpenShell intercepts and blocks the connection, logging the attempt
  2. The TUI (openshell term) displays the blocked request with host, port, and requesting binary
  3. The operator approves or denies in real time
  4. Approved endpoints persist for the session only -- they are not saved to the baseline policy
⚠️
Warning: Session-approved endpoints disappear when the sandbox restarts. If an endpoint is consistently needed, add it to your YAML policy file and re-run nemoclaw onboard for permanence. Otherwise the agent will be blocked again next session.
