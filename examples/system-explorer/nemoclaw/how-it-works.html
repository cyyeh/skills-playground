How It Works

Five-layer protection model, filesystem access boundaries, network policy enforcement, inference routing, and real-world performance characteristics.

Five-Layer Protection Model

NemoClaw's security is built on five distinct isolation layers, each enforced independently. The key insight is that these layers are independent: even if one is somehow bypassed, the others remain intact. An agent that tricks its way past network policy still cannot write to system files; an agent that finds a filesystem exploit still cannot reach unapproved network endpoints.

Layer Mechanism When Applied Can Agent Override?
1 Container Isolation Docker containers with restricted capabilities Locked at creation No
2 Process Hardening Dropped Linux capabilities, seccomp syscall filtering, ulimit (512 process max) Locked at creation No
3 Filesystem Integrity Landlock LSM restricts writes to /sandbox and /tmp only; system dirs read-only Locked at creation No
4 Network Control Deny-by-default egress with YAML-defined allowlists per binary/endpoint Hot-reloadable No
5 Inference Security All inference calls routed through gateway; credentials stored on host only Hot-reloadable No

Filesystem Access Model

The Landlock Linux Security Module enforces strict filesystem boundaries. The agent can do useful work -- write code, create files, generate output -- in its designated workspace, but cannot modify system binaries, tamper with configuration, or access the host filesystem.

✍️
Read-Write
  • /sandbox
  • /tmp
  • /dev/null
The agent's workspace. All code, files, and outputs live here. This is the only area the agent can modify.
📖
Read-Only
  • /usr, /lib
  • /proc, /dev/urandom
  • /app, /etc, /var/log
System binaries, libraries, and configuration. The agent can read these but never modify them.
🚫
No Access
  • Everything else
  • Host filesystem
  • Other containers
Completely invisible to the agent. No reading, writing, or directory listing. The host system is unreachable.

Network Policy Enforcement

Policies are defined in YAML with granular control over endpoints, allowed binaries, and HTTP methods. The baseline policy (openclaw-sandbox.yaml) ships with 9 default policy groups covering Claude Code, NVIDIA APIs, GitHub, ClawHub, OpenClaw, npm registry, and Telegram.

YAML 
# Example: Allow GitHub API access only from the gh CLI
policies:
  github_rest_api:
    endpoints:
      - host: api.github.com
        port: 443
    allowed_binaries:
      - /usr/bin/gh
    methods:
      - GET
      - POST
      - PATCH
      - PUT
      - DELETE

This policy allows only the gh CLI binary to communicate with api.github.com on port 443. Even if the agent spawns a curl or python process targeting the same endpoint, the request will be blocked because only /usr/bin/gh is in the allowlist.

Real-Time Network Approval Flow

When an agent tries to reach an unlisted endpoint, NemoClaw uses a "progressive trust" model where the operator starts with minimal access and selectively expands it as the agent demonstrates legitimate needs.

1
Intercept & Block

OpenShell intercepts and blocks the connection, logging the attempt with host, port, and requesting binary.

→
2
TUI Display

The terminal UI (openshell term) shows the blocked request with full details for operator review.

→
3
Operator Decision

The operator reviews the request context and approves or denies it in real time from the TUI.

→
4
Session-Only Persistence

Approved endpoints persist for the current session only -- they are not saved to the baseline policy file.

ℹ️
Progressive trust in practice During initial agent runs, use openshell term to monitor which endpoints the agent legitimately needs. Once you understand the access pattern, encode those endpoints in your YAML policy for persistent access.

Inference Routing

All model API calls from the sandbox are routed through a local DNS entry (inference.local) to the OpenShell gateway. The gateway handles credential injection, data classification, and provider routing so the agent never directly touches API keys or external endpoints.

1
Strip Credentials

The gateway removes any credentials from the sandbox request. The agent never had real API keys.

→
2
Classify Sensitivity

The Privacy Router inspects the query for PII, proprietary code, or financial data.

→
3
Route to Provider

Sensitive data goes to local Nemotron models. Non-sensitive data goes to configured cloud providers (OpenAI, Anthropic, etc.).

→
4
Inject Credentials

The gateway adds host-side API keys for the selected provider from ~/.nemoclaw/credentials.json.

→
5
Forward & Return

The request is forwarded to the model provider. The response flows back through the gateway to the sandbox.

Performance Characteristics

Based on NVIDIA's published benchmarks, NemoClaw-wrapped agents achieve strong inference performance. The sandbox itself adds minimal overhead -- the main performance factor is which inference backend you use (local GPU vs. cloud API).

⚠️
Alpha benchmarks -- not independently verified All performance numbers below are from NVIDIA's own testing. No independent third-party audits have been published as of March 2026. Early community reviews suggest local inference with Nemotron may be slower than these figures indicate. Treat these as reference points, not guarantees.

Performance Profile

PinchBench Score 85.6%
Highest among open models. PinchBench evaluates autonomous agent task completion across coding, browsing, and tool use.
Token Throughput 442 tok/s
2.2x more than GPT-OSS-120B and 7.5x more than Qwen3.5-122B at 8k input / 16k output. Context window up to 1M tokens.
Scalability Single-player
One sandbox per gateway, one trusted operator. No multi-tenant isolation or horizontal scaling. Separate instances needed for multiple agents.
Platform Support Linux-only
Linux-first (Ubuntu 22.04+). macOS has partial support with known issues. WSL2 is experimental. Intel Macs are unsupported entirely.

Additional data points: sandbox image size is approximately 2.4 GB compressed, K3s takes 1-3 minutes to apply manifests after onboarding, and the container startup time is dependent on local Docker performance.
