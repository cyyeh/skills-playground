How It Works

Inference Routing Mechanism

Inference requests follow a controlled path that keeps API credentials secure and off the agent's machine.

Inference Request Flow

1
Agent → inference.local
The agent makes a standard model API call to inference.local, a virtual endpoint visible only within the sandbox's network namespace.
▼
2
Network Namespace Intercept
OpenShell intercepts the request at the network namespace boundary and routes it to the gateway on the host.
▼
3
Gateway Attaches Credentials
The gateway looks up the configured route, retrieves the API key from ~/.nemoclaw/credentials.json, and attaches it to the outbound request.
▼
4
Provider Response
The provider's response flows back through the gateway to the agent. At no point does the agent see the raw API key.
Credentials never enter sandbox

Four Defense Layers

NemoClaw enforces four independent security layers simultaneously. Each layer addresses a different attack surface.

📂 Filesystem Isolation (Landlock)

When the sandbox starts, the openshell-sandbox supervisor fetches a Landlock policy via gRPC and applies it to the agent child process. The agent can read and write /sandbox and /tmp; all system paths are mounted read-only.

Locked at creation The policy cannot be modified by the agent process after application.

🚫 Syscall Filtering (seccomp)

A seccomp BPF filter runs in allowlist mode, permitting only the syscalls the agent needs for normal operation. The filter controls individual syscall arguments at fine-grained level, blocking privilege escalation and dangerous system operations.

Locked at creation Filters are applied when the container is created and cannot be relaxed.

🌐 Network Namespace Isolation

Each sandbox gets its own network namespace, giving the agent a completely isolated view of the network. The agent cannot see the host's interfaces, routing tables, or other containers. Every TCP connection is forced through an HTTP CONNECT proxy operated by OpenShell.

Hot-reloadable Network egress rules can be updated at runtime via operator approval.

🤖 Inference Control

All model API calls are intercepted and routed through the gateway. The agent communicates only with inference.local — there is no direct path to any external model provider. The routing is transparent to the agent.

Hot-reloadable Operators can switch providers or models without restarting the sandbox.

Runtime Policy Approval Workflow

Network Policy Enforcement

1
Agent Attempts Outbound Connection
The agent tries to connect to a host not listed in the baseline policy.
▼
2
OpenShell Blocks & Logs
The connection is blocked at the network namespace boundary. The attempt is logged with host, port, and requesting binary.
▼
3
Operator Review (TUI)
The blocked request appears in the operator's terminal UI (openshell term) with full context for approval or denial.
▼
4
Session-Scoped or Permanent Approval
Approved endpoints persist for the current session. For permanent changes, the operator edits the YAML policy and reapplies.

Messaging Bridges

NemoClaw connects sandboxed agents to Telegram, Discord, and Slack through host-side bridge processes. These bridges run outside the sandbox, maintaining the security boundary — messaging credentials remain on the host and the agent cannot directly access messaging APIs. Communication flows through the sandbox's internal API.

💡
Key insight: By running bridges on the host rather than inside the sandbox, NemoClaw ensures that even if the agent is compromised, it cannot impersonate the bridge, access messaging tokens, or send messages to unauthorized channels.
