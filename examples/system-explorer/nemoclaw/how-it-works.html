How It Works

End-to-end walkthrough: from installation through a complete tool-calling request to ongoing operations.

Installation and First Launch

NemoClaw installs with a single command that handles dependency checks, runtime installation, and interactive configuration.

bash 
# One-command installation
curl -fsSL https://www.nvidia.com/nemoclaw.sh | bash

# The wizard configures:
# 1. Inference provider (NVIDIA Endpoints, Ollama, or custom)
# 2. API credentials (stored in ~/.nemoclaw/credentials.json)
# 3. Network policy (default deny-all with pre-approved domains)
# 4. Sandbox container image download (~2.4 GB)

Blueprint Lifecycle

When the operator runs nemoclaw launch, the blueprint executes five stages to create a reproducible, verified sandbox environment.

Resolve

The CLI locates the blueprint artifact from the OCI registry and checks version compatibility. Local caching avoids repeated downloads on subsequent launches.

Verify

The artifact's OCI digest is validated against the expected hash. If the digest doesn't match (indicating potential tampering), the launch is immediately aborted.

Plan

The blueprint runner compares the desired sandbox state (from blueprint.yaml and openclaw-sandbox.yaml) against the current host state, calculating the minimum set of changes required.

Apply

OpenShell CLI commands execute to: create a network namespace, launch the hardened container, apply seccomp filters, drop capabilities, configure the inference gateway, and start the Policy Engine as a separate process.

Status

The runner reports the sandbox's health including active policies, inference provider connectivity, resource usage, and any configuration warnings.

Tool-Calling Request Walkthrough

Here is a complete trace of what happens when a user asks the agent: "Check the status of our API at api.example.com"

Message Ingestion

The user's message arrives via a messaging bridge (Telegram, Discord, Slack, or terminal). The bridge delivers the message as structured data over SSH into the sandbox. The agent has no direct network socket to the messaging service.

Planning

OpenClaw's planner evaluates the request and determines that it requires an HTTP health check, which needs the http_request tool and network access to api.example.com.

Tool Selection

The planner queries the Tool Registry for available tools. Each tool has a typed schema (name, description, parameters). The agent selects http_request with URL https://api.example.com/health and method GET.

Network Policy Evaluation

Before the HTTP request executes, the Rust Policy Engine intercepts the outbound connection. It extracts the destination (api.example.com:443), performs DNS resolution with SSRF protection, and checks the YAML policy. If allowlisted, the connection proceeds. If not, it is blocked and an alert appears in the operator's TUI.

Operator Approval (if needed)

When the Policy Engine blocks an unknown destination, the operator sees a real-time prompt offering three options: Approve (one-time), Deny, or Approve+Remember (persists the rule).

Tool Execution

With network access approved, the tool's execute function runs inside the sandbox. The HTTP request goes through the OpenShell gateway, which validates the connection, logs it (URL, method, timestamp, response code), and proxies it to the destination.

Inference for Response Synthesis

The agent needs to synthesize a human-readable response from the tool's output. This triggers an inference request. The Privacy Router examines the content, classifies it (API status data is typically non-sensitive), and routes to the appropriate provider. The Inference Gateway injects credentials and proxies the request.

Response Delivery

The synthesized response travels back through the messaging bridge to the user. Every step of this flow has been recorded in the audit log.

Example Response: "The API at api.example.com is healthy. The /health endpoint returned HTTP 200 with response time of 142ms. All services report operational status."

Ongoing Operations

bash 
# Monitor sandbox in real-time
nemoclaw status          # Health, policies, resource usage
nemoclaw logs            # Stream audit logs
openshell term           # TUI for policy management

# Policy hot-reload (no restart needed)
vim openclaw-sandbox.yaml # Edit YAML policy
# Policy Engine detects change and applies immediately

# Workspace persistence
nemoclaw backup           # Backup agent state
nemoclaw restore          # Restore from backup

Check Your Understanding

In the tool-calling walkthrough above, at which step does the agent first interact with an external network endpoint? And who holds the credentials for that interaction?

The agent first interacts with an external endpoint at Step F (Tool Execution), when the HTTP request to api.example.com is proxied through the OpenShell gateway. Critically, the agent never holds the credentials -- the Inference Gateway injects them at the boundary. For the tool's HTTP request, the connection is proxied by the gateway. For inference requests (Step G), the gateway injects the LLM provider's API key. The sandbox itself is credential-free at all times.
