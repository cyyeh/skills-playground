Use Cases & Case Studies

When DuckDB is the right tool, when it is not, and how real companies put it to work.

When to Use DuckDB

DuckDB shines when your data fits on a single machine and your workload is analytical -- aggregations, joins, scans across millions of rows. It requires zero setup: no server, no configuration files, no cloud account. If you can describe what you need in SQL and your data lives in files, DuckDB is likely a strong fit.

🔍
Ad-hoc Data Analysis
SQL on local files with zero setup. Faster than pandas for large datasets -- no imports, no infrastructure, just answers.
🔄
ETL & Data Pipelines
Read from CSV, JSON, and Parquet. Transform with SQL. Write the results back to Parquet or CSV -- all in one script.
🧭
Data Exploration
Query CSVs, Parquet, and JSON files directly. No schema definition, no loading step -- DuckDB auto-detects structure and types.
📊
Embedded Analytics
Add analytical queries to your application without a separate database server. Just a library call from Python, Node.js, or Rust.
⚙️
CI/CD Data Testing
Fast startup, no infrastructure. Spin up DuckDB inside a test suite to validate data pipeline outputs in milliseconds.
🔒
Privacy-Sensitive Analysis
Data never leaves the local machine. No cloud uploads, no network traffic -- ideal for regulated industries and sensitive data.
🐍
Replacing pandas for SQL Work
Better memory efficiency, automatic parallel execution, and native SQL instead of chaining DataFrame methods.
📂
Data Lake Querying
Direct Parquet/Iceberg/S3 queries as a lightweight alternative to Spark -- no cluster required.

Should You Use DuckDB?

Work through the questions below to find out whether DuckDB is the right fit for your use case, or whether a different tool would serve you better.

🌳

Decision Tree

Does your dataset require distributed processing across multiple machines?
DuckDB runs on a single machine and handles datasets up to roughly 2 TB. If your data is measured in petabytes or you need a cluster, a distributed system is the better fit.
Is your workload primarily analytical (reads, aggregations, joins)?
OLAP workloads scan and aggregate large amounts of data. OLTP workloads are transactional -- many small reads and writes, like an e-commerce checkout system.
Do you need many concurrent writers (multiple users or services writing simultaneously)?
DuckDB supports multiple concurrent readers but only a single writer at a time. This is a deliberate design choice for analytical performance, not a limitation to be worked around.
Do you need real-time streaming ingestion with sub-second latency?
DuckDB excels at batch processing and ad-hoc queries. For continuous data streams that must be queryable within milliseconds, dedicated streaming systems are more appropriate.

DuckDB is an excellent fit

Your workload is analytical, fits on a single machine, and does not require many concurrent writers or real-time streaming. DuckDB will give you fast queries with zero infrastructure overhead.

pip install duckdb brew install duckdb

Consider a distributed analytics system

For petabyte-scale data that requires processing across multiple machines, look at distributed analytical databases or processing frameworks.

ClickHouse Snowflake BigQuery Apache Spark

But DuckDB may still play a role

Many teams use DuckDB for local development and prototyping, then deploy to a distributed system for production. DuckDB's SQL is highly compatible with these platforms.

DuckDB is not the right choice for OLTP

DuckDB is optimized for analytical scans and aggregations, not for high-throughput transactional workloads with many small reads and writes.

PostgreSQL MySQL CockroachDB

You can use both

A common pattern: PostgreSQL for your OLTP workload, with DuckDB reading from PostgreSQL (via the postgres_scanner extension) for analytical queries. Best of both worlds.

Consider a database with multi-writer support

DuckDB's single-writer model means only one connection can write at a time. For workloads requiring many concurrent writers, use a database designed for that concurrency pattern.

PostgreSQL ClickHouse

DuckDB still works for the read side

If your writes are batched (e.g., a nightly ETL job writes data, then many users query it), DuckDB's multiple-reader support handles that pattern well.

Use a streaming-native system

For real-time streaming ingestion with sub-second query latency on fresh data, dedicated streaming systems are purpose-built for this pattern.

Kafka + ClickHouse Apache Flink Materialize

DuckDB for the analytical layer

A common architecture: streaming system handles ingestion, periodically writes Parquet files to object storage, and DuckDB queries those files for batch analytics and reporting.

When Not to Use DuckDB

Every tool has boundaries. Knowing where DuckDB's strengths end helps you pick the right tool for each job instead of forcing a square peg into a round hole.

⚡
High-Concurrency OLTP
Many users reading and writing small transactions simultaneously -- e-commerce, banking, SaaS applications.
PostgreSQL, MySQL, CockroachDB
🌐
Petabyte-Scale Analytics
Datasets that span many terabytes and require distributed processing across a cluster of machines.
ClickHouse, Snowflake, BigQuery, Spark
💨
Real-Time Streaming
Continuous data ingestion from event streams with sub-second query freshness requirements.
Kafka + ClickHouse, Apache Flink
✏️
Multi-User Concurrent Writes
Applications where multiple services or users must write data at the same time without blocking each other.
PostgreSQL
⏱️
Sub-Millisecond Point Lookups
Key-value access patterns that need single-digit microsecond response times for individual record retrieval.
Redis, DynamoDB, RocksDB
🔎
Full-Text Search
Searching through unstructured text with ranking, faceting, fuzzy matching, and relevance scoring.
Elasticsearch, Typesense
💡
The Cloud Warehouse Replacement Pattern A growing number of companies are replacing expensive cloud analytics services (BigQuery, Athena, Redshift) with DuckDB running inside scheduled cloud functions -- AWS Lambda or GCP Cloud Run. The pattern: store data as Parquet files on S3, run DuckDB in a serverless function to query them, and pay only for compute seconds instead of per-query or per-TB pricing. Some teams report 10-50x cost reductions for batch analytics workloads that do not require real-time concurrency.

Real-World Case Studies

DuckDB has moved well beyond academic research. These companies and projects have built production systems on top of it, each leveraging different strengths of the engine.

🌍
Watershed
Data Pipelines
Uses DuckDB for data pipeline operations, converting activity data into carbon footprint calculations. They query Parquet files directly as internal tooling, benefiting from DuckDB's zero-setup deployment and fast columnar scans over environmental datasets.
☁️
MotherDuck
Cloud Service
A cloud service built on DuckDB that provides a serverless analytical database. Users can run ATTACH 'md:' to offload heavy queries to cloud compute while keeping the local DuckDB developer experience. Hybrid execution lets queries run locally or in the cloud depending on data location.
📈
Rill Data
BI Engine
Built their BI dashboard engine on DuckDB, powering interactive exploration of large datasets with sub-second query times. DuckDB's vectorized execution and columnar storage enable real-time aggregation and drill-down without pre-computed cubes.
🌐
Evidence
WebAssembly
Built Universal SQL with DuckDB-Wasm, enabling browser-based interactive dashboards that query data locally in WebAssembly. No backend server required -- the entire analytical engine runs in the user's browser tab.
☁️
Cloud Functions Pattern
Cost Optimization
A widespread production pattern: companies replace BigQuery, Athena, or Redshift with DuckDB running in scheduled cloud functions (AWS Lambda, GCP Cloud Run). DuckDB processes log files and Parquet directly from object storage at a fraction of the cost of managed data warehouse services.
🎯
The Bottom Line DuckDB excels when your data fits on one machine (up to roughly 2 TB) and your workload is analytical. It is the strongest choice for local data analysis, ETL pipelines, embedded analytics, and replacing expensive cloud queries. Choose something else when you need distributed scale, high write concurrency, real-time streaming, or sub-millisecond point lookups.
