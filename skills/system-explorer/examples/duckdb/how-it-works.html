How It Works

Trace a query from SQL text to vectorized execution, understand DuckDB's internal vector format, compression algorithms, and how it queries files directly without loading them first.

Query Execution: A Walkthrough

Let's trace exactly what happens inside DuckDB when you run an analytical query. Each step transforms your SQL a little further, from human-readable text into highly optimized machine operations on columnar data.

SQL 
SELECT region, SUM(amount) AS total
FROM sales
WHERE year = 2025
GROUP BY region
ORDER BY total DESC
LIMIT 5;

This query asks: "Show me the top 5 regions by total sales amount for 2025." Simple enough in SQL, but the engine performs a sophisticated sequence of transformations to execute it efficiently. Here is each step:

1
Parse & Bind

SQL parsed into an AST. Binder resolves sales table, confirms columns exist, determines types.

→
2
Logical Plan

Limit(5) → Sort(total DESC) → Aggregate(GROUP BY region, SUM) → Filter(year=2025) → Scan(sales)

→
3
Optimize

Filter pushed into scan. Zone maps skip row groups. Column pruning. Top-5 uses heap.

→
4
Physical Plan

Pipeline 1: Scan → Filter → Aggregate (builds hash table).
Pipeline 2: Finalize → Top-5 Sort → Result.

→
5
Vectorized Execution

Each thread reads morsels, decompresses columns into 2,048-value vectors, applies filters, updates hash table.

Inside Vectorized Execution (Step 5)

Each worker thread in Pipeline 1 repeats this tight loop until all data is consumed:

  1. Read a morsel (batch of row groups)
  2. Decompress only year, region, amount columns into vectors of 2,048 values
  3. Apply filter year = 2025, producing a selection vector (bitset of matching rows)
  4. For matching rows, hash region and update the running SUM in the hash table
  5. Grab the next morsel and repeat

Pipeline 2 merges per-thread hash tables, runs Top-5 selection via a heap (no full sort needed), and returns results. The entire process is automatic -- DuckDB parallelizes across all available CPU cores with zero configuration.

The Vector Format

DuckDB's internal data representation uses several vector types to avoid unnecessary work. Instead of always decompressing data into flat arrays, the engine can pass compressed or constant representations through the pipeline, deferring materialization until it is truly needed.

📊
Flat Vector
Standard contiguous array of values. The baseline format that all operators can consume.
Column of integers from storage
🔃
Constant Vector
A single value repeated for all 2,048 rows. No memory wasted on duplicates.
Literal 2025 in WHERE clause
📖
Dictionary Vector
An index array referencing a smaller unique-value set. Keeps the compressed form in-pipeline.
Dictionary-compressed strings
🔢
Sequence Vector
Represented by an offset and increment. Generates values on-the-fly with zero storage.
Row IDs (0, 1, 2, 3, ...)

Operators output one of these types, and the next operator handles any type through the Unified Vector Format -- a common abstraction that prevents a combinatorial explosion of type-pair handling between operations.

💡
Performance tip: String inlining Strings of 12 bytes or fewer are inlined directly into the vector -- no pointer indirection, no cache miss. Larger strings store a pointer plus a 4-byte prefix, enabling comparisons like WHERE name > 'M' to short-circuit without following the pointer. This optimization is crucial for string-heavy analytical workloads where column scans would otherwise be dominated by pointer-chasing latency.

Compression Algorithms

DuckDB automatically selects the best compression for each column segment during data loading. A compression analyzer evaluates the data distribution and picks the algorithm yielding the best ratio. Each segment within a row group can use a different algorithm, so a single table might combine several strategies.

Algorithm Best For How It Works
Constant Single-value columns Stores one value for the entire segment. Zero per-row overhead.
Dictionary Low-cardinality columns Maps distinct values to small integer codes. Lookups stay in L1 cache.
RLE Sorted / clustered data Stores (value, count) pairs. Long runs of the same value compress to nearly nothing.
Bitpacking Small-range integers Fewer bits per value based on the actual range (e.g., 4-bit for values 0-15).
FSST String columns Builds a symbol table of common substrings, replaces them with short codes.
ALP Float / double columns Adaptive Lossless floating-Point: exploits patterns in IEEE 754 representations.
Chimp Time-series doubles XOR-based compression that exploits similarity between sequential double values.
🔭
Architecture insight DuckDB can operate directly on compressed data during execution. For instance, a dictionary-compressed column stays as a Dictionary Vector through the pipeline. Filters and comparisons work on the small integer codes, and only the final result set gets decompressed. This "partial decompression" strategy means less memory bandwidth and better CPU cache utilization.

Direct File Querying

One of DuckDB's most powerful features is querying files directly without loading or importing them first. Point DuckDB at a file, and it reads only the data it needs using the same optimization pipeline as persistent tables.

📁
Local Parquet
Metadata-driven pushdown, skip irrelevant row groups
📄
CSV
Multi-hypothesis parser auto-infers types and delimiters
☁️
S3 / HTTP
Remote files via httpfs extension, range requests for efficiency
📊
JSON
Newline-delimited or nested JSON with auto schema detection
SQL 
-- Query a local Parquet file
SELECT * FROM 'sales_2025.parquet' WHERE region = 'APAC';

-- Query a CSV with auto-detection
SELECT * FROM read_csv('data.csv');

-- Query Parquet files on S3
SELECT * FROM 's3://my-bucket/data/*.parquet';

-- Query a remote CSV over HTTP
SELECT * FROM 'https://example.com/data.csv';

For Parquet files, DuckDB leverages the Parquet metadata (row group statistics, page indexes) to push down filters and skip irrelevant data before reading. DuckDB's multi-hypothesis CSV parser examines the file to automatically determine the delimiter, quote character, header row, and column types -- handling edge cases that trip up simpler parsers.

ℹ️
Note: When querying remote files on S3 or HTTP, DuckDB uses range requests to read only the byte ranges it needs. Combined with Parquet metadata pushdown, this means a query on a 10 GB remote Parquet file might only transfer a few megabytes of data.

Why Vectorized Execution Matters

Traditional databases process one row at a time through a tree of operators (the Volcano model). Each row incurs function-call overhead, branch mispredictions, and poor cache utilization. DuckDB's vectorized approach processes 2,048 values per operator call, enabling:

  • Tight loops that the compiler can auto-vectorize into SIMD instructions
  • Cache-friendly sequential access patterns -- columnar data in vectors fits L1/L2 cache
  • Amortized overhead -- interpretation cost is paid once per 2,048 rows, not per row
  • Automatic parallelism -- the morsel-driven model distributes vector batches across all CPU cores

The result: DuckDB achieves throughput comparable to hand-optimized C++ code on analytical workloads, while still providing the flexibility of a full SQL engine with dynamic query optimization.
